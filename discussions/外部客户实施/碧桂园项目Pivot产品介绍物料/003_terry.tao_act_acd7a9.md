## 〇、一句话定位

碧桂园在做的"房屋交易生态平台"把交易、履约、服务、AI 四个平台拼起来，覆盖供给侧、需求侧、服务侧、治理侧。但他们的设计里**少了一层**：每件事横跨多个系统、多个角色推进时，谁来记录、判断、调度、收口？

**Pivot 就是这一层——事项级中枢神经 + 调度大脑。**

---

## 一、关键判断：Pivot 不和现有 AI 能力争位置

碧桂园把"AI 平台"和交易/履约/服务并列为四大平台之一。但他们的"AI 平台"实际是一组**嵌入到各交易环节的局部 AI 能力**——智能匹配、估价、3D 建模、知识库、垂直大模型。

Pivot 不取代这一层，也不和它争位置：

- 碧桂园的现有/规划系统是**执行肌肉**——负责房源、决策、履约、服务、信任
- 碧桂园的局部 AI 能力是**专项 AI 技能**——嵌在各肌肉里
- **Pivot 在更高一层**——回答"这件事现在到哪了、谁该做什么、为什么、什么时候"

碧桂园现有设计里，每件事的状态、责任、推进逻辑**散落在各个子系统**。Pivot 把每件事抽象为一个 `matter`，沿时间线沉淀完整数据流，由 AI 做调度判断。

## 三、外部世界：角色与系统在哪

### 3.1 角色定位

| 角色                                            | 在哪     | 与 Pivot 的接触方式                                          |
| ----------------------------------------------- | -------- | ------------------------------------------------------------ |
| **终端用户（C 端，租客/买家）**                 | 最外层   | **不直接接触 Pivot**；通过 C 端 App、IM、电话与平台交互；行为/对话被采集后才进入 Pivot |
| **经纪人/房东（B 端）**                         | 外层     | **双向**——既是 Pivot 调度台的使用者（看待办、领指令），也是事实补录者；同时仍在 B 端工作台做日常作业 |
| **服务商（S 端）**                              | 外层     | 主要通过服务商系统接受派单与回报进度；一般不直接打开 Pivot   |
| **业务人员（碧桂园内部：运营/风控/法务/财务）** | **内层** | Pivot 调度台的重度用户；通过 Pivot 看全局事项状态、处理跨系统判断、介入升级与争议 |

### 3.2 现有/规划系统定位

这些系统是 Pivot 的**输入源 + 执行接口**，不是 Pivot 的内部模块：

- **房源经营系统**——房源生命周期、渠道分发、经营看板
- **决策系统**——AI 对话找房、智能筛选、看房协同
- **交易履约系统**——电签、托管、分账、进度看板、争议
- **服务商生态系统**——入驻、报价、履约、评价
- **平台安全与信任系统**——真实性核验、行为风控、合规、信誉评分
- **AI 能力体系**——智能匹配、估价、3D 建模、知识库、垂直大模型

> Pivot 不重写这些系统，也不重写其中的 AI 能力。Pivot **调用**它们作为工具，**监听**它们作为事实源。

## 四、信息进入 Pivot

进入 Pivot 的素材分四类：

**4.1 系统事件（来自现有业务系统）**

- 房源经营系统：新房源上架、线索进入、房源调价、签约回款节点
- 履约系统：合同生成、押金到账、节点超时、争议触发、工单创建
- 服务商系统：派单、开工确认、进度更新、验收、用户评价
- 信任系统：风控告警、违规识别、投诉立案
- 决策系统：用户表达明确意向、对话停滞、看房放鸽子

**4.2 对话与触达内容**

- 微信/IM 对话节录、电话语音转写、客服工单原文、邮件/短信往来

**4.3 人的主动录入**

- 经纪人/业务人员在调度台**补录**线下发生的事实
- AI 根据补录原文整理候选 think / act / 时间字段，**用户显式确认后**写入

**4.4 行为信号**

- C 端用户在 App / 小程序里的强意向行为（反复看同一房源、AI 对话表达明确预算等）

## 五、Pivot 形成什么判断和指令

**5.1 把每件事抽象为 matter**

任何一件值得跟进的事——一次报修、一笔买卖意向跟进、一次合规争议、一次房源经营动作——都被建模为一个 `matter`，进入 `planning → executing → paused/finished/cancelled → reviewed` 状态机。

**5.2 沿时间线沉淀数据流**

时间线上每一篇文件是 think / act / verify / result / insight 之一。所有事实通过这五种文件落到 Git，成为可追溯证据——这正好对接碧桂园文档反复强调的"全程透明可追溯""争议有据可查"。

**5.3 程序 + AI 双层调度**

- **Scheduler Code**：扫描所有 matter 的 index，按结构化规则发现需关注项（act 逾期、长期无进展、paused 到期复查、failed 未解决、可结束、待复盘、流程风险）
- **Scheduler AI**：把候选包整理为可读提醒，判断优先级，给出建议下一步动作和建议责任人

**5.4 输出三类东西**

1. **判断**——这件事现在卡在哪、有什么风险、是否该结束
2. **指令**——下一步谁应该做什么、何时之前
3. **证据**——所有判断都建立在已沉淀的 matter time line 上，可回看

> **关键边界**：Pivot 永远是 **AI 候选 + 人工确认**。AI 不直接修改 matter 状态，不直接写关键时间字段。

## 六、数据流层：必须流通的信息和接入方式

Pivot 关心的是**数据流**——这条信息必须能流通；至于具体怎么接，是项目变量。

### 6.1 入口数据流（外部 → Pivot）

事实必须能进入 Pivot：

- **业务事件**——房源/履约/服务商/信任/决策系统的关键事件
- **对话内容**——客户/经纪人在 IM、电话、邮件等渠道的关键沟通
- **人工补录**——业务人员/经纪人在调度台手动补录线下事实
- **行为信号**——C 端用户行为里的强意向信号

### 6.2 出口数据流（Pivot → 外部）

指令必须能送达：

- **触达**——微信/短信/邮件/App 推送/站内信
- **系统动作**——派单、调价、电签、托管、催款、上下架
- **文档生成**——合同、报告、知识库引用
- **局部 AI 调用**——把碧桂园的智能匹配/估价/3D/知识库当作子例程调用

### 6.3 接入方式：三条路径

每一条数据流怎么接通，按优先级选择：

| 优先级 | 接入方式       | 说明                                                         |
| ------ | -------------- | ------------------------------------------------------------ |
| ①      | **复用 reuse** | 客户已有系统能直接承担这条数据流，直接对接                   |
| ②      | **适配 adapt** | 客户有但格式/协议/语义不匹配，加中间件做翻译。中间件可由 Pivot 侧交付，也可由客户侧集成基础设施承担 |
| ③      | **自建 build** | 客户没有相应能力，由 Pivot 自带 Agent 补齐                   |

> **不确定性管理**：客户业务系统目前在建设中。接入方式矩阵反映的是**当前快照**——某条数据流今天可能走 build，未来不一定。本文档不展开演进路径，只表达 Pivot 的产品边界与数据流定义**不会因接入方式变化而变化**。

## 七、围绕场景验证

> 以下场景统一按"接入方式均为 Pivot 自建 Agent（build）"描述，便于完整展现 Pivot 的数据流。实际项目中部分步骤会被客户现有系统直接承担。

### 场景 A：租客卫生间漏水报修

| 阶段  | 谁/在哪          | 做了什么                                                     |
| ----- | ---------------- | ------------------------------------------------------------ |
| 起点  | 客户在微信发消息 | "卫生间漏水了，下午能修吗"                                   |
| 入口  | 入口接入         | 抽取关键事实（房源/租约/客户ID），进入 Pivot                 |
| Pivot | 思维大脑         | 创建 matter「卫生间漏水-单元305」；AI 生成 think（问题描述+影响）+ act 候选（owner=维修主管，建议 24h 内完成），用户确认后进入 executing |
| 出口  | 出口接入         | 服务商系统下派单 + 微信回复客户上门时间 + 通知房东           |
| 监听  | Pivot Scheduler  | 12h 内无服务商响应自动升级到客服主管                         |
| 回流  | 入口接入         | 服务商完工事件回流，写入 verify 候选；租客评价 passed        |
| 收口  | Pivot            | result（finished）→ insight 沉淀该类问题 SLA                 |

### 场景 B：房屋买卖意向跟进

| 阶段  | 谁/在哪                         | 做了什么                                                     |
| ----- | ------------------------------- | ------------------------------------------------------------ |
| 起点  | 客户在 C 端 AI 对话明确买房需求 | 决策系统记录意向                                             |
| 入口  | 入口接入                        | 意向事件 + 对话内容进入 Pivot                                |
| Pivot | 思维大脑                        | 创建 matter「客户王某-改善型买房意向」；调用 AI 估价、智能匹配作为子例程整理 think；act 派给经纪人小张，3 天内带看 |
| 出口  | 出口接入                        | 通知经纪人 + 约带看日程                                      |
| 监听  | Pivot Scheduler                 | 7 天无进展自动提醒                                           |
| 回流  | 入口接入                        | 经纪人补录带看反馈 → think                                   |
| 出口  | 出口接入                        | 触发意向合同生成                                             |
| 收口  | Pivot                           | result → insight 客户偏好沉淀                                |

### 场景 C：房源造假风控立案（横跨 3 系统）

> 这是**传统设计最容易掉链子的场景**，也是 Pivot 价值最突出的地方。

| 阶段 | 谁/在哪              | 做了什么                                           |
| ---- | -------------------- | -------------------------------------------------- |
| 起点 | 信任系统识别可疑房源 | 触发风控告警                                       |
| 入口 | 入口接入             | 风控事件进入 Pivot，自动创建 matter                |
| 出口 | 出口接入             | 房源经营系统暂停发布动作；履约系统冻结相关待签合同 |
| 协同 | 出口接入             | 法务被拉进 matter 评论流                           |
| 监督 | Pivot 调度台         | "流程风险"分组让运营负责人一眼看到全部立案         |
| 收口 | Pivot                | result 关闭事项，insight 沉淀风控规则              |

整件事涉及 **4 个系统、3 个角色**，但只有**一个 matter timeline** 把它串起来。这是 Pivot 不可替代的位置。

## 八、三层切分：什么恒定、什么变量

按"产品边界（恒定）+ 数据流定义（恒定）+ 接入方式矩阵（项目变量）"三层切分。

### 8.1 Pivot 产品边界（恒定 / 必交付）

无论客户的现有系统是什么样，下面这些是 Pivot 必须自带的：

| 模块        | 内容                                                         |
| ----------- | ------------------------------------------------------------ |
| 数据模型    | matter 状态机（含 paused → cancelled）、五种文档类型、timeline-first index |
| 调度大脑    | Scheduler Code（七条扫描规则）+ Scheduler AI（自然语言提醒、优先级、建议） |
| 用户面      | 调度台、上下文说明面板、补录引导                             |
| 协作        | creator / owner 责任体系、@mention、评论流                   |
| AI 协同模式 | AI 候选 + 用户显式确认 的写入边界                            |
| 证据层      | Git + Markdown 单一事实源、版本可追溯                        |
| 状态约束    | 文件类型与状态的允许矩阵、状态迁移合法性兜底校验             |

这一层是 Pivot 的产品价值核——和客户是谁无关。

### 8.2 数据流定义（恒定 / 与客户系统现状无关）

| 类型 | 必须流通的信息                         |
| ---- | -------------------------------------- |
| 入口 | 业务事件、对话内容、人工补录、行为信号 |
| 出口 | 触达、系统动作、文档生成、局部 AI 调用 |

每条数据流的语义和最小集是确定的。具体怎么接通可以变，但**数据流本身不变**。

### 8.3 接入方式矩阵（项目变量 / 当前快照）

每条数据流今天在客户环境下怎么接通，需要在项目启动时盘清楚。盘点维度：

- 客户**已有**该信息流的搬运能力 → reuse
- 客户**有但格式不通** → adapt（中间件由 Pivot 侧或客户侧承担）
- 客户**没有** → build（Pivot 自带 Agent）

> 评估某个客户的 Pivot 落地复杂度时：
>
> - 8.1 + 8.2 是固定项
> - 8.3 是项目变量
> - **变量越多 = 项目交付越重，但 Pivot 的产品价值不打折**

## 九、几个原则性边界

**1. 单一事实源不动**

碧桂园已有系统里的房源、合同、订单、用户实体仍是各自的事实源。Pivot 只承担**事项**这一层的事实，引用业务实体 ID，不复制业务实体本身。

**2. AI 永远做候选，不做决定**

所有写入 matter 状态、关键时间字段的动作都需要人显式确认。这是 Pivot 设计哲学的硬约束，对客户来说审计与合规友好。

**3. Pivot 不和现有 AI 能力争位置**

智能匹配、估价、3D、知识库、垂直大模型是被 Pivot 调用的工具，不是被替代的对象。

**4. Pivot 的可见 UI 主要面向 B 端和内部业务人员**

C 端用户不直接面对 Pivot；他们感受到的是"为什么这家平台跟进得很到位"。

**5. 核心 Pivot 先上线，接入方式按需补齐**

落地路径的合理顺序：先把 Pivot 调度台和最小补录闭环跑起来（接最关键的 1-2 个事件源 + 1 个触达通道），再逐步扩 reuse / adapt / build。
