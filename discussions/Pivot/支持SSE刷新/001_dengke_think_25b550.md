# 支持SSE刷新

我们需要给 Pivot Web 客户端增加一个轻量级的实时刷新机制，使得当其他用户新发帖子或回复时，当前在线用户的页面能够较快感知到变化，并正确显示最新内容和未读状态。

方案上不建议使用轮询，也不建议一开始就上 WebSocket。当前系统已经有现成的线程列表、详情页和未读计算逻辑，因此更合适的做法是增加一层基于 `SSE` 的轻量级变更通知机制。也就是说，服务端只负责告诉前端“某个主题发生了变化”，前端再静默调用现有接口刷新数据。这样可以保持现有数据模型不变，降低实现和维护成本。

## 设计方案

- 服务端增加一个 `SSE` 事件流接口，用于向已打开页面的客户端推送“thread 发生变化”的通知。
- 当有新主题创建或某个主题新增回复时，服务端发送一条很薄的事件，只包含主题标识等最小必要信息，不直接推送完整内容。
- 前端收到事件后，不直接重绘页面，而是后台静默刷新现有接口。
- 如果当前用户正在看主题列表，则静默刷新列表数据和未读数。
- 如果当前用户正打开某个被更新的主题详情页，则额外静默刷新该主题详情。
- 如果刷新结果没有实际变化，页面不应闪烁、不应重置滚动位置，也不要出现 loading 骨架。
- 如果页面处于后台、SSE 连接断开或移动端 WebView 暂停连接，则在页面回到前台时补做一次静默刷新，作为可靠兜底。

## 设计边界

- `SSE` 只负责“通知有变化”，不负责承载业务真相。
- 线程列表、帖子详情、未读数仍然完全以现有 HTTP 接口返回为准。
- 不新增复杂的实时同步状态，不引入 WebSocket 房间、在线状态、双向消息等机制。

## 验收标准

- 用户 A 新建主题后，用户 B 已打开的 Web 页面能够在较短时间内看到新主题出现。
- 用户 A 在某个主题下回复后，其他在线用户能够较快看到该主题未读数变化。
- 如果用户当前正停留在该主题详情页，能看到新回复同步出现。
- 整个过程中页面应尽量无闪烁、无整页重载、无明显跳动。
- 在移动端浏览器和飞书内置 WebView 中，即使 SSE 临时断开，页面回到前台后也能自动补刷新，保证结果最终正确。
