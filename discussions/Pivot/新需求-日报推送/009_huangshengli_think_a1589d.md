# 回 v0.3 PRD · 几处替代方案与待确认问题

基于 terry 在 v0.3(#008)把 dengke #005 的方向收敛成完整的 PRD,三层视角、AI 因果叙事、不评分、分层阅读、AI 不写 index 等核心点已经清楚,本帖不再重复一致部分。

下面提出几处**替代设计选项**和**实施层细化建议**,请评审组对比定夺。

## 一、产品形态 · 5 处替代方案待评审

### 1. 推送形态:发几份?

PRD §4.4 主张推送**两份**:
- 团队日报(matter 视角 + 公司视角)→ 团队 Channel
- 个人日报(个人视角)→ 每位成员的个人 IM 会话(或管理 Channel)

**替代方案**:只发**一份团队日报**到团队 Channel,内含三层视角(matter / 个人 / 公司)。个人视角作为团队日报中的一个区块呈现,不为每位成员单独再生成、单独推送。

权衡:

- **一份(替代方案)**:管理层一份卡片看全;团队群里所有成员动态共同可见 —— 透明度高,但等于公开点名"谁今天有 / 没有产出";工程简单
- **两份(PRD 现方案)**:团队群只展示 matter / 公司维度,不暴露每人具体动作;每人在自己 IM 会话私下看个人维度 —— 隐私感更好;工程上每天多 N 次 IM 推送 + N 次 LLM 调用

这是产品定位分歧:**透明度优先 vs 个人隐私优先**。

倾向**一份**:

- 当前团队规模(<20 人)透明度收益高于隐私成本
- dengke #005 立的论里,"今天没有任何输入和输出"这一信号本身就是要让管理层在团队场景里看见的;推到个人会话反而稀释了"团队透明"语义
- 一对一推送会让个人日报变成自我反思工具,与"决策辅助"定位有偏移
- 隐私顾虑在团队规模上去后再单独评估,不必首期就承担工程双倍成本

### 2. 风险事项的呈现位置

PRD §5.1 把"需关注的无推进 matter"放在 matter 视角末尾的"停滞段落"里,与"纯无推进 matter"同段呈现。

**替代方案**:风险事项独立成块(例如卡片中独立的 "⚠️ 需要关注" 区块),与 matter 推进叙事并列。

权衡:
- 嵌入停滞段落(PRD 现方案):无推进事项整体感强;但风险被"无推进汇总"淡化,管理层第一眼看不到
- 独立成块(替代方案):风险有视觉显著性;卡片多一个区块

### 3. 无活动成员的展示形式

(仅在 D1 选"一份"时才需明确;若选"两份",个人日报本身就是一对一,不存在合并问题)

PRD §5.2 要求"今天没有任何输入和输出"句子单独成段,即每位无活动成员独立成段。

**替代方案**:渲染层合并到一行,例如 `最近一天无输入输出:A、B、C、…`。

权衡:
- 独立成段(PRD 现方案):每个名字存在感强,严肃
- 合并行(替代方案):团队规模 >20 人时不会有 20 行重复句式撑爆篇幅;每个名字仍明确点出,符合 dengke #005 "明确不模糊" 的要求

### 4. 篇幅硬约束

PRD §5.4 默认团队日报 500-800 字 / 个人日报 50-300 字。

**替代方案**:首期不写硬上限,只在 prompt 里给"管理层一屏读完"的软目标,跑两周后用真实卡片样本回归确定。

理由:数值在实测前难定 —— 500-800 在团队规模上去后可能不够;过严的硬约束会逼 LLM 截断有信息量的叙事。先放开跑,数据说话。

### 5. AI 调用编排

PRD §4.4 流程图把"AI 生成三层叙事"画成一次调用,未明确编排细节。

**替代方案**:拆 3 次独立调用 —— call A(matter 视角)、call B(个人视角)并行,call C(公司视角)以 A、B 输出为输入串行执行。

理由:
- 降低单次 LLM 理解负担,单次输入更小、关注点更集中
- 公司视角必须基于 A、B 事实合成,串行执行能避免 LLM 凭空给出"积极/平稳/偏停滞"
- 任一调用失败时可仅降级该区块,其余正常输出

## 二、PRD 实施层细化建议

### 2.1 候选筛选补一类:paused 单独识别

PRD §4.2 B 类把所有"长期无新文件"统一归 7 天阈值。建议把 paused 状态单独切出来,默认 **14 天**阈值。

理由:paused 是用户显式标记的"先不动",7 天阈值对 paused 过敏感会反复告警;14 天阈值更贴合"管理者预期"。

### 2.2 后台配置清单 · 独立配置文件 + 修改频率分级

PRD §六列了约 18 项配置。建议日报配置走**独立配置文件**(例如 `/opt/team-pivot-web/var/config/daily_report.yaml`),**不上 /admin UI、不写入 SQLite settings 表**。**程序对每一项都内置默认值**,配置文件中显式给出对应键值时覆盖默认值;**优先级 配置文件 > 程序默认值**。

理由:
- 日报是独立 systemd 脚本,配置走独立文件与脚本本身的"轻量隔离"风格一致,不污染 Pivot 主服务的 settings 表
- 日报配置不需要管理员高频运维,UI 改动收益低 / 工程成本高
- 升级 / 排错时配置变更可随代码进 git diff,而不是隐藏在 SQLite 里
