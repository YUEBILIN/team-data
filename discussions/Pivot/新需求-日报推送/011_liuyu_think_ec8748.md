# 想和大家讨论 #007 中三次 LLM 调用的拆分边界

整体方向认同——"程序做事实、AI 做叙事"扎实，分层阅读和 JSON Schema 约束都想得清楚。拆分本身的动机也成立（降低单次负担、稳定 JSON）。

想拎出来讨论的是**拆分的边界**：当前按"视角"拆（matter / 个人 / 公司），是否可以考虑改成按"任务流水线"拆？

## 按视角拆的潜在问题

Call A 和 Call B 底层读的是**同一份 timeline 数据**，只是聚合维度不同（按 matter 还是按人）。同一件事会出现在两次独立调用里。

举例：dengke 今天给"登录链路改造"写了 verify 判 passed。Call A 在 matter 区块会提到、Call B 在个人区块也会提到。两次独立 LLM 生成，没有机制保证口径一致——A 可能写"事项接近闭环"，B 可能写"仍在等待后续验收"。**同一张卡里两个版本的状态判断**，读者会困惑。

另外被程序硬过滤掉的 matter 只以整体计数传给 Call C，C 写公司定性时看不到完整盘子，可能偏乐观。

## 一种替代拆法：按任务流水线拆

| 调用 | 任务 | 输出 |
|---|---|---|
| 1 · 事实抽取 | timeline → 结构化事件清单 | 对齐的事实底稿（无叙事） |
| 2 · 叙事生成 | 基于底稿写 matter + 个人区块 | 各区块叙事 |
| 3 · 整体定性 | 基于叙事 + 完整全集计数定调 | 公司视角 |

调用次数不变。matter 和个人区块共享同一份事实底稿，不会矛盾；Call 3 也能看到完整盘子。

代价：A、B 不能并行，延迟多一段串行（5-10 秒，非实时可接受）。但总 token 可能反而更省——原方案底层 timeline 要喂两遍（A、B 各一次），新方案只在 Call 1 喂一次。
