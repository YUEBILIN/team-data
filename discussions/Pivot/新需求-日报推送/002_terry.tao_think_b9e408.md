# PRD｜事项日报 · IM 推送

**版本：** v0.1 **状态：** 待评审 **作者：** Terry（产品经理） **日期：** 2026年4月28日

------

## 一、背景与问题

### 需求来源

目标客户的高层管理者需要每天快速了解各事项的推进状态和干系人动态，但他们不会也不应该每天登录 Pivot 逐个翻看 matter。

### 为什么不在 Pivot 内新增日报模块

在早会讨论中，我们评估了在 Pivot 内部新增日报页面的方案，结论是不做，原因有三：

- **信息干扰**：Pivot 内部承载了大量 matter 的细节数据流，日报作为高层视角的聚焦摘要，放进去会被细节淹没，违背"一眼看全貌"的目标。
- **功能重叠**：后续规划的调度台（scheduler）已经承担了"发现待办、推动事项"的职责。如果日报也做成站内模块，两者的定位会模糊，用户不知道该看哪个。
- **使用习惯**：高层的工作习惯是在 IM（飞书 / 钉钉 / Slack / 企业微信）中获取信息，而不是登录一个新系统。把日报推送到他们已有的信息流里，触达率远高于站内页面。

### 设计判断

**日报应当从 Pivot 中解耦出来，以消息通知的方式推送到各 IM 的 Channel。** Pivot 负责生产数据，日报负责消费数据并以摘要形式触达高层。日报不是 Pivot 的一个页面，而是 Pivot 向外输出的一条信息管道。

------

## 二、目标

**一句话定义：** 每天定时向指定 IM Channel 推送一份结构化的事项进展摘要，让高层不登录 Pivot 即可掌握全局。

### 日报回答三个问题

1. **今天发生了什么？** —— 过去 24 小时内哪些 matter 有新的推进
2. **哪些事情需要关注？** —— 逾期、卡住、风险事项
3. **谁在负责什么？** —— 关键干系人与其当前职责

### 不是目标

- 不是完整的工作汇报（不替代团队周报）
- 不是任务分配工具（不在日报里指派工作）
- 不是调度台的替代品（调度台面向一线执行者，日报面向管理层）
- 不承载评论、回复等交互（日报是只读推送）

------

## 三、范围界定

### 本期包含

- 基于 index 的每日数据聚合
- AI 生成结构化日报摘要
- 推送到飞书群 / Channel（首批支持一个 IM 平台）
- 推送时间与频率的基础配置
- 推送范围的基础配置（全部 matter / 按空间筛选）

### 本期不包含

- 多 IM 平台同时支持（首批只做一个，后续扩展）
- 日报的站内存档与历史查看
- 日报内容的交互式操作（评论、回复、一键跳转创建文件）
- 个性化订阅（每人看不同内容）
- 周报 / 月报

------

## 四、核心机制

### 4.1 数据来源

日报的数据完全来自 Pivot 现有的 index 层，不读取 Markdown 正文，不新增独立数据源。

具体读取的字段包括：

- `matter.current_status` —— 判断事项当前阶段
- `matter.updated_at` —— 筛选过去 24 小时内有活动的 matter
- `timeline` 最近文件项 —— 获取最新动态（谁创建了什么类型的文件）
- `timeline[].summary` —— 获取文件摘要
- `timeline[].owner / creator` —— 获取干系人
- `timeline[].status_change` —— 识别状态迁移事件
- `timeline[].verifications` —— 识别验收结论（passed / failed / cancelled）

同时读取 scheduler code 的候选提醒包（如已上线），获取逾期、卡住、风险等异常信号。如调度台尚未上线，日报模块自行执行简化版异常扫描。

### 4.2 生成流程

```
每日定时触发（如每天 09:00）
        │
        ▼
  聚合程序读取所有 matter index
  筛选过去 24h 有活动的 matter
  提取异常信号（逾期 / 无进展 / failed 等）
        │
        ▼
  构造 AI 输入包（结构化数据，非正文）
        │
        ▼
  AI 生成日报摘要
  （分区组织 + 自然语言表达）
        │
        ▼
  格式化为 IM 消息卡片
        │
        ▼
  调用 IM API 推送到指定 Channel
```

### 4.3 日报内容结构

日报分为四个区块，由 AI 基于聚合数据生成：

**区块一：今日概览**

用 2-3 句话概括当天全局状态。例如："今日共 12 个活跃事项，3 个有新推进，2 个存在风险，1 个已正式完成。"

**区块二：事项动态**

列出过去 24 小时内有新文件产生的 matter，每个 matter 一行摘要：

```
● 登录链路改造 [executing]
  liuyu 创建了 verify，判定 003_act 为 passed
  
● 权限体系重构 [planning]
  dengke 创建了 think，分析多租户隔离方案

● 数据迁移方案 [executing → finished]
  dengke 创建了 result，事项正式完成
```

每条动态包含：matter 名称、当前状态（如有迁移则标注箭头）、最近文件的 creator + 类型 + 摘要。

**区块三：需要关注**

列出存在异常信号的 matter，按严重程度排序：

```
⚠️ 支付对账优化 — act 已逾期 3 天，无人验收（owner: wangwu）
⚠️ 消息推送改造 — 14 天无新文件，可能已停滞（owner: liuyu）
⚠️ 用户反馈闭环 — verify 判定 failed，但无后续行动
```

如果没有异常事项，此区块显示"当前无异常事项"。

**区块四：干系人一览**

按人汇总当前负责的活跃 matter 数量和状态分布：

```
liuyu — 4 个活跃事项（2 executing, 1 planning, 1 paused）
dengke — 3 个活跃事项（1 executing, 2 planning）
wangwu — 2 个活跃事项（1 executing, 1 逾期）
```

### 4.4 推送配置

| 配置项       | 说明                       | 默认值                 |
| ------------ | -------------------------- | ---------------------- |
| 推送时间     | 每天几点推送               | 09:00（工作日）        |
| 推送频率     | 每天 / 仅工作日            | 仅工作日               |
| 目标 Channel | 推送到哪个 IM Channel      | 管理员配置             |
| 推送范围     | 全部 matter / 按标签筛选   | 全部                   |
| 静默条件     | 当天无任何活动时是否仍推送 | 不推送，避免无意义消息 |

配置由 Pivot 管理员在系统设置中完成，普通用户不可修改。

## 五、与现有模块的关系

### 与调度台的分工

| 维度 | 调度台                       | 日报               |
| ---- | ---------------------------- | ------------------ |
| 受众 | 一线执行者                   | 管理层             |
| 目标 | 驱动个人处理待办             | 让高层掌握全局     |
| 载体 | Pivot 站内页面               | IM Channel 消息    |
| 交互 | 可操作（创建文件、延后提醒） | 只读（无交互）     |
| 粒度 | 单个提醒卡片级别             | 每日汇总级别       |
| 触发 | 持续扫描，实时更新           | 每日定时，一次推送 |

两者数据来源一致（都基于 index），但服务不同用户、不同场景、不同载体，不存在替代关系。

### 与 index 的关系

日报只读取 index，不写入、不修改。日报是 Pivot 数据流的消费者，不是生产者。

------

## 六、关键用户链路

### 场景：CEO 早上打开飞书看日报

1. 每天 09:00，系统自动聚合前一天的 matter 数据。
2. AI 生成日报摘要，格式化为飞书消息卡片。
3. 推送到"管理层日报"飞书群。
4. CEO 打开飞书，在群里看到日报卡片，30 秒内掌握全局。
5. 发现"支付对账优化"逾期 3 天，在群里 @ 相关负责人询问进展。
6. 后续跟进仍在 IM 中完成，或由负责人回到 Pivot 创建文件推进。

日报的价值在到达 IM 的那一刻已经完成，不需要用户回到 Pivot 才能发挥作用。

------

## 七、验收标准

| 验收项       | 标准                                                  |
| ------------ | ----------------------------------------------------- |
| 定时推送     | 配置的时间点准时推送，误差 < 5 分钟                   |
| 内容准确性   | 日报中的事项状态、干系人、摘要与 index 数据一致       |
| 异常信号覆盖 | 逾期 act、长期无进展、failed 未跟进三类异常均能被识别 |
| 静默规则     | 当天无活动时不推送                                    |
| 消息可读性   | 推送内容在 IM 中排版清晰，不出现乱码和格式错乱        |
| 端到端耗时   | 从触发到推送完成 < 2 分钟                             |

## 八、开放问题

| 编号 | 问题                               | 建议                                                         |
| ---- | ---------------------------------- | ------------------------------------------------------------ |
| Q-01 | 首批支持哪个 IM 平台？             | 根据目标客户和内部实际使用情况，飞书优先级最高               |
| Q-02 | 日报是否需要站内存档？             | 首版不做。如后续有需求，可将生成的日报内容存为独立文件       |
| Q-03 | 调度台未上线时，异常信号由谁扫描？ | 日报模块自行实现简化版扫描（复用调度台的规则逻辑），调度台上线后切换为读取其候选包 |
| Q-04 | 多团队 / 多 Channel 的场景？       | 首版只支持单 Channel 推送。后续可按空间或其他新定义的类型分组，推送到不同 Channel |
| Q-05 | 推送内容的语言？                   | 首版跟随系统语言设置。如客户有中英文混合需求，二期考虑       |

------

## 九、二期方向

- **多 IM 平台支持**：飞书 + 钉钉 + Slack + 企业微信
- **周报 / 月报**：在日报基础上聚合更长周期的趋势
- **个性化订阅**：不同角色订阅不同范围的 matter
- **交互式卡片**：在 IM 卡片内直接跳转到 Pivot 对应 matter
- **日报历史存档**：在 Pivot 内可查看历史日报，形成管理层视角的事项演进记录
