# PRD｜事项日报 · IM 推送

**版本：** v0.3
**状态：** 待评审
**作者：** Terry（产品经理）
**日期：** 2026年4月29日
**变更说明：** 基于 v0.2 反馈，将日报核心从"结构化区块统计"重新定位为"基于 matter index 的因果叙事分析"。日报内容由"四区块团队日报 + 两区块个人日报"重构为 **matter 视角 / 个人视角 / 公司视角** 三层叙事；AI 的核心动作由"过滤 index 字段、按规则归类"调整为"理解事项推进、生成因果线叙事"；明确"程序筛选候选 matter，AI 判断是否读原文并生成叙事"的分层阅读策略；删除任何指向产出排名 / 绩效评价的措辞。

---

## 一、背景与问题

### 需求来源

目标客户的高层管理者需要每天快速了解各事项的推进状态和干系人动态，但他们不会也不应该每天登录 Pivot 逐个翻看 matter。

### 为什么不在 Pivot 内新增日报模块

在早会讨论中，我们评估了在 Pivot 内部新增日报页面的方案，结论是不做，原因有三：

- **信息干扰**：Pivot 内部承载了大量 matter 的细节数据流，日报作为高层视角的聚焦摘要，放进去会被细节淹没，违背"一眼看全貌"的目标。
- **功能重叠**：后续规划的调度台（scheduler）已经承担了"发现待办、推动事项"的职责。如果日报也做成站内模块，两者的定位会模糊，用户不知道该看哪个。
- **使用习惯**：高层的工作习惯是在 IM（飞书 / 钉钉 / Slack / 企业微信）中获取信息，而不是登录一个新系统。把日报推送到他们已有的信息流里，触达率远高于站内页面。

### 设计判断

**日报应当从 Pivot 中解耦出来，以消息通知的方式推送到各 IM 的 Channel。** Pivot 负责生产数据，日报负责消费数据并以叙事形式触达高层。日报不是 Pivot 的一个页面，而是 Pivot 向外输出的一条信息管道。

### 相对 v0.2 的关键修正

v0.2 将日报定位为"决策辅助"，方向正确，但格式仍带有较强的**结构化区块统计**色彩——把活跃事项分桶展示、按风险类型列条、按数字罗列干系人。这种做法实际上还是在让 AI 做"过滤 index 字段、统计数量、按规则归类"的工作。

经过进一步讨论和理解，我们认定日报的真正价值不是把 index 的字段过滤出来重排，也不是统计今天有几个 think、几个 act、谁写了多少条评论，而是 **基于 matter index 做因果叙事分析**：讲清楚每个 matter 是干什么的、到目前为止做了哪些工作、谁参与了、今天发生了什么变化、推进情况怎么样、现在处于什么状态、接下来应该关注什么。

因此 v0.3 在两个维度上做了根本调整：

- **内容生产的核心动作**：从"分类罗列事项与状态"调整为"生成事项的因果线叙事"
- **内容呈现的组织方式**：从"按区块切分（概览 / 进展 / 风险 / 建议）"调整为"按视角切分（matter 视角 / 个人视角 / 公司视角）"

## 二、目标

### 核心定位：因果叙事分析，不是结构化统计

日报的本质是让管理者通过一份摘要，理解今天每个推进中的事项发生了什么、为什么发生、对事项整体意味着什么——也就是一段段紧贴事实的因果线叙事，而不是一张张归类表。

因此日报对 AI 的要求不是"过滤 index 字段、统计数量、按状态归类"，而是：

- 理解每个 matter 在做什么
- 还原事项截至今天的演进路径
- 解释今天的输入和行动让事项发生了什么变化
- 判断事项现在的真实位置
- 指出下一步值得关注的方向

### 日报回答三类问题

1. **每个推进中的事项今天怎么样了？** —— matter 视角的因果线叙事，含背景、前情、今日变化、当前状态、下一步关注
2. **每个团队成员今天做了什么？** —— 个人视角的输入与输出明细，无活动也明确表达
3. **整个公司今天在往哪些方向发力？** —— 公司视角的整体推进总结，含状态判断（积极 / 平稳 / 偏停滞）

### 不是目标

- 不是逐条任务清单（日报面向理解，不是 todo list）
- 不是调度台的替代品（调度台面向一线执行者，日报面向管理层）
- 不承载评论、回复等交互（日报是只读推送）
- **不是绩效考核工具**：个人日报只用于信息透明，不参与产出排名、打分、对比
- **不是数字统计报表**：不以"今天 X 个 think / Y 个 act / Z 条评论"为输出形态

## 三、范围界定

### 本期包含

- 程序：基于 index 的每日候选筛选（有推进 / 需关注的无推进 / 今日有活动的人）
- AI：matter 视角的因果叙事生成
- AI：个人视角的输入输出叙事生成
- AI：公司视角的整体推进总结
- AI：按需读取文件正文的弹性机制（可配置开关）
- 推送到 IM Channel（首批支持一个平台）
- 后台管理配置（推送策略、内容边界、AI 读取权限）

### 本期不包含

- 多 IM 平台同时支持（首批只做一个，后续扩展）
- 日报的站内存档与历史查看
- 日报内容的交互式操作
- 周报 / 月报
- 个人产出的排名、评分、绩效相关呈现

## 四、核心机制

### 4.1 分工原则

| 角色 | 职责                                                         |
| ---- | ------------------------------------------------------------ |
| 程序 | 扫描所有 matter 的 index，按确定性规则筛选三类候选，构造 AI 输入包 |
| AI   | 基于候选 matter 的 index 判断信息密度，必要时按需读取原文，生成三层叙事 |
| 后台 | 提供配置入口（推送策略、阈值、AI 读取权限）                  |

核心边界：

- **程序负责确定性筛选**：哪些 matter 进入今日候选、哪些算无推进、哪些算风险——这些都基于 index 字段做客观判断，不做主观叙事
- **AI 负责理解、判断、叙述**：事项推进了什么、为什么、下一步看哪里——AI 不直接写 index、不修改 matter、不创建文件
- 日报始终是 Pivot 数据流的消费者，永远不是生产者

### 4.2 候选筛选规则

每日定时触发后，程序扫描所有 matter 的 index，形成三类候选：

**A. 有推进的 matter**（matter 视角逐个展开因果叙事）

满足任一即纳入：

- 过去 24 小时内 timeline 出现过新文件
- 过去 24 小时内出现过 status_change

**B. 无推进但需重点关注的 matter**（matter 视角的"停滞段落"中点名）

满足任一即纳入：

- 长期无新文件（默认 ≥ 7 天）
- act 已逾期（默认 ≥ 1 天）且无 verify 覆盖
- act 被 verify 判为 failed，超过阈值（默认 ≥ 2 天）仍无后续 think 或 act

**C. 纯无推进、无风险的 matter**（仅在 matter 视角的"停滞段落"以总数形式提及）

既不属于 A 也不属于 B 的 matter。

**D. 今日有活动的人**（个人视角逐人生成）

- timeline 中过去 24 小时内 creator / owner / mention 命中该成员

**E. 今日无活动的人**（个人视角仍生成，且必须出现）

- 在 Pivot 注册成员名单中、但未出现在 D 的人
- 必须输出"今天没有任何输入和输出"的明确句子，不允许静默跳过

如果调度台已在实现中，A、B 两类候选可直接复用调度台输出的候选包，而非由日报模块独立扫描。

### 4.3 AI 分层阅读策略

AI 拿到候选包后，先基于 index 中的 `summary`、`timeline`、`status_change`、`verifications`、`comments`、`creator / owner` 判断能否生成可靠叙事。

判断逻辑：

- **summary 已足够清楚** → 直接基于 index 生成叙事
- **AI 判断 index 信息不足以生成可靠叙事** → AI 自主决定读取该文件正文，作为叙事生成的补充依据

不足的判断标准（由 AI 自行评估，不硬编码）：

- 因果链条不清：今天的变化无法从 summary 推出
- 状态变化原因不明：status_change 发生了，但不知道为什么
- summary 过短：单条 summary 信息密度过低，叙事会变成空话
- 可能产生误判：仅凭 index 容易写出与事实冲突的叙事

规则与约束：

- AI **只在判断信息不足时才读取正文**，不是默认读取所有文件
- 每次日报生成时，后端记录 AI 读取正文的次数、涉及文件数和额外 Token 消耗
- 管理员可在后台关闭正文读取权限（关闭后 AI 仅使用 index 数据）
- 首期以实验方式运行，通过数据对比评估正文读取对叙事质量的提升幅度与 Token 成本增量，作为后续策略调整依据

### 4.4 生成流程

```
每日定时触发
      │
      ▼
程序：扫描所有 matter index
程序：筛选 A/B/C/D/E 五类候选
      │
      ▼
程序：构造 AI 输入包
      （A 类与 B 类的完整 index 切片 + C 类的总数 + D 类的活动条目 + E 类成员列表）
      │
      ▼
AI：基于 index 判断每个候选的信息密度
    └─ 不足时按需读取部分原文
      │
      ▼
AI：生成三层叙事
    ├─ matter 视角：A 类逐个因果叙事 + B 类点名 + C 类汇总句
    ├─ 个人视角：D 类逐人输入输出叙事 + E 类显式空句
    └─ 公司视角：整体方向与状态判断
      │
      ▼
程序：组装 IM 卡片
      │
      ├──► 团队公共部分（matter 视角 + 公司视角）→ 团队 Channel
      └──► 个人部分（个人视角）→ 个人会话或指定管理 Channel
```

### 4.5 无进展事项的处理策略

无进展事项不在 matter 视角中逐个展开，但也不静默忽略。处理方式：

- **常规无进展事项（C 类）** → 在 matter 视角末尾的"停滞段落"做一句话宏观汇总（如："另有 12 个事项昨日无新输入，整体处于平稳期。"）
- **存在风险的无进展事项（B 类）** → 在"停滞段落"中单独点名并附简要建议（如："消息推送改造已 14 天无新文件，owner: liuyu，建议确认是否转 paused 或补录线下进展。"）

判断由 AI 自主完成，不依赖硬编码规则。管理员可在后台调整"无进展静默阈值"（默认 7 天），超过该阈值的事项 AI 会重点关注并纳入 B 类候选。

## 五、日报内容结构

日报由三层视角组成。三层独立但同源——同一份候选包同时驱动三层叙事。

### 5.1 第一层：Matter 视角 — 事项推进的因果叙事

#### 内容

针对当天有推进的 matter（A 类候选），逐个生成一段因果线叙事。每段叙事至少回答以下五个维度：

| 维度             | 含义                                                         |
| ---------------- | ------------------------------------------------------------ |
| 这件事是干什么的 | 事项的最小背景说明，让读者无需再翻 matter 也能理解           |
| 之前进展到哪里   | 上一阶段的关键状态或里程碑                                   |
| 今天谁做了什么   | 今日新增的输入（think / 评论 / 讨论意见）或行动（act / verify / result / insight / status_change） |
| 现在处于什么状态 | 事项当前位置（含 matter status 与具体进展）                  |
| 下一步关注什么   | 后续值得跟进的方向                                           |

针对当天无推进的 matter，不逐个展开，统一在该层尾部用"停滞段落"概括：

- B 类（需关注的无推进）：单独点名 + 简要建议
- C 类（纯无推进）：汇总成一句话

#### 排版规则

- 每个有推进的 matter 单独成段，开头以 matter 名称引出
- 单段叙事控制在 3-5 句话以内，必须覆盖五个维度，但不强制每段都按"是什么 / 前情 / 今天 / 现在 / 下一步"机械分句
- 段落数量上限默认 8 个；超出时 AI 选取最重要的 8 个，其余事项以"另有 N 个事项有动作"的形式折入停滞段落
- 不使用表格、不堆砌统计数字、不使用计数型措辞
- 末尾"停滞段落"为一段连续叙述，控制在两句话以内（除非 B 类点名条目较多）

#### 示例

> **登录链路改造**
>
> 这是一项把现有登录链路替换为统一身份方案的事项。前期已经完成方案分析与主链路实现，并于上周进入 executing 阶段。今天 liuyu 创建了 verify，对 003_act 给出 passed 判断，主链路验收通过；dengke 在评论中确认边界情况已记录。事项目前位于 executing 阶段尾段，主链路已闭环。下一步关注的是边界情况的二轮验收，以及是否择期生成 result 正式收口。
>
> **数据迁移方案**
>
> 这是把旧库存量数据按新 schema 迁移落地的事项。前期已就方案完成两轮 think，并在昨天进入 executing。今天 dengke 创建了 result，宣布事项正式完成，状态从 executing 迁移到 finished。下一步关注的是是否需要补一篇 insight 沉淀本次迁移的经验。
>
> **停滞段落**
>
> 另有 12 个事项昨日无新输入。其中"消息推送改造"已 14 天无动作，owner 为 liuyu，建议确认是否转 paused 或补录线下进展；其余 11 个事项整体处于平稳期，无明显风险。

### 5.2 第二层：个人视角 — 团队成员的输入与输出

#### 内容

按团队成员逐人生成个人日报，从 Pivot 时间线看每个人今天的输入和输出：

- **输入**：think、评论、讨论意见、验证意见、被 mention 后的回应
- **输出**：act、verify、result、insight、由其触发的 status_change、对其他 matter 的关键贡献

#### 关键规则

- 如果某个成员今天在 Pivot 时间线上**没有任何活动**，必须明确写"今天没有任何输入和输出"。**不允许使用"较为安静""暂无明显动作""活动较少"等模糊话术**，也不允许静默跳过该成员
- 不引入排名、打分、产出量级对比、贡献度评分等绩效化表达
- 个人日报只反映已存在于 Pivot 时间线的事实，不做"是否努力""是否高效"等主观判断
- 重点动作（如触发 status_change、产生关键 verify 结论、推动他人 matter 闭环）应在叙事中点出，但不与其他成员的产出做比较

#### 排版规则

- 每个成员单独成段，开头以成员名引出
- 单段控制在 5 句话以内
- 输入与输出可以混合叙述，不强制分两栏
- 单人当日动作过多时（默认 > 8 条），AI 取最重要的 8 条入叙事，其余以一句话汇总（如："此外在另外 4 个 matter 的评论中给出了反馈意见。"）
- "今天没有任何输入和输出"的句子单独成段，不与其他内容混排

#### 示例

> **liuyu**
>
> 今天主要在登录链路改造与权限体系重构两个方向有动作。在登录链路改造中作为 verify owner 完成了对 003_act 的验收并判定为 passed，是主链路闭环的关键动作；在权限体系重构中创建了一篇 think，分析多租户隔离方案并建议采用基于角色的权限模型。此外在数据迁移方案的 result 评论中给出了对边界处理的反馈。
>
> **dengke**
>
> 今天在数据迁移方案中创建了 result，正式将事项推进到 finished 状态。此外在登录链路改造的 verify 评论中确认边界情况已记录。
>
> **wangwu**
>
> 今天没有任何输入和输出。

### 5.3 第三层：公司视角 — 整体推进总结

#### 内容

站在整个公司或团队管理者视角，用一段简单文字总结今天的整体推进。必须覆盖：

- 主要推进了哪些方向
- 哪些事项有实质进展（产生了关键文件或状态迁移）
- 哪些事项还在讨论（仍在 think 阶段，未出 act / verify）
- 哪些事项卡住了（B 类点名）
- 整体推进状态判断：**积极 / 平稳 / 偏停滞**（必填）

#### 排版规则

- 一段连续叙述，不再切分子区块
- 控制在 3-5 句话
- **必须明确给出整体状态判断**（积极 / 平稳 / 偏停滞），并附一句话依据
- 不重复 matter 视角中已有的细节，只做高一层的方向汇总

#### 示例

> 今天团队整体处于**积极推进**状态。登录链路改造与数据迁移方案两个核心方向均产生了实质性闭环动作（一项主链路验收通过、一项正式完成）；权限体系重构仍在方案分析阶段，今天进入了新一轮 think。整体看推进节奏是健康的，但消息推送改造已停滞 14 天，是当前最值得管理层关注的点。

### 5.4 三层视角的整体边界

| 维度                                         | 默认值                                                       |
| -------------------------------------------- | ------------------------------------------------------------ |
| 整份团队日报（matter 视角 + 公司视角）总长度 | IM 一屏至一屏半（约 500-800 字）                             |
| matter 视角因果叙事段落数量                  | ≤ 8 段 + 1 段停滞汇总                                        |
| 单段 matter 叙事长度                         | ≤ 5 句话                                                     |
| 个人视角条目数                               | 等于今日有 Pivot 身份的所有成员数（含输出"今天没有任何输入和输出"的人） |
| 公司视角段落                                 | 1 段，3-5 句话，必含状态判断                                 |
| 异常高亮阈值                                 | 逾期 > 1 天、无进展 > 7 天、failed 未跟进 > 2 天             |
| 事项引用                                     | 只引用 matter 名称和叙事必要的关键文件，不展示文件正文       |

以上规则均为默认值，管理员可在后台按团队偏好调整。

## 六、后台管理配置

日报的推送、内容边界和 AI 行为均通过后台配置，由管理员统一控制。所有配置项提供默认值。

### 6.1 推送配置

| 配置项               | 说明                                       | 默认值                   |
| -------------------- | ------------------------------------------ | ------------------------ |
| 团队日报开关         | 是否启用团队日报（matter 视角 + 公司视角） | 开启                     |
| 个人日报开关         | 是否启用个人日报（个人视角）               | 关闭（需管理员主动开启） |
| 推送时间             | 每天几点推送                               | 团队 09:00 / 个人 09:30  |
| 推送频率             | 每天 / 仅工作日                            | 仅工作日                 |
| 团队日报目标 Channel | 推送到哪个 IM Channel                      | 管理员配置               |
| 个人日报推送方式     | 个人会话 / 指定管理 Channel                | 个人会话                 |
| 推送范围             | 全部 matter / 按标签筛选                   | 全部                     |
| 团队侧静默条件       | 当天无任何 matter 活动时是否仍推送         | 不推送                   |
| 个人侧静默条件       | 当天无活动的成员是否仍推送                 | **仍推送**（输出空句）   |

注意：个人侧默认仍推送，是为了让"今天没有任何输入和输出"这个事实本身可见。如果默认静默，会让读者无法区分"系统漏推"和"该成员真的无活动"。

### 6.2 内容配置

| 配置项              | 说明                             | 默认值 |
| ------------------- | -------------------------------- | ------ |
| matter 视角段落上限 | 当日最多展开几个 matter          | 8 段   |
| 单 matter 叙事长度  | 单段叙事最大句子数               | 5 句   |
| 个人视角动作上限    | 单人最多入叙事的动作数           | 8 条   |
| 公司视角长度        | 公司视角段落句子数               | 3-5 句 |
| 逾期高亮阈值        | 超过几天算逾期                   | 1 天   |
| 无进展高亮阈值      | 超过几天算需在停滞段落点名       | 7 天   |
| failed 未跟进阈值   | verify failed 后多久未响应算风险 | 2 天   |

### 6.3 AI 行为配置

| 配置项           | 说明                                        | 默认值 |
| ---------------- | ------------------------------------------- | ------ |
| AI 正文读取权限  | 是否允许 AI 在 index 信息不足时读取文件正文 | 开启   |
| 单次正文读取上限 | 单次日报生成中 AI 最多读取几篇正文          | 10 篇  |

### 6.4 可观测性

后端在每次日报生成时记录以下数据，供管理员查看：

- 候选筛选结果（A / B / C / D / E 各类的命中数量）
- AI 读取正文的次数和涉及文件数
- 额外 Token 消耗量
- 日报生成总耗时
- 三层叙事的生成耗时分布（用于诊断瓶颈）

首期以实验方式运行，通过数据对比评估正文读取对生成质量的实际提升，作为后续策略调整依据。

## 七、与现有模块的关系

### 与调度台的分工

| 维度     | 调度台                       | 日报                             |
| -------- | ---------------------------- | -------------------------------- |
| 受众     | 一线执行者                   | 管理层                           |
| 目标     | 驱动个人处理待办             | 帮助管理者理解事项推进           |
| 载体     | Pivot 站内页面               | IM Channel 消息                  |
| 交互     | 可操作（创建文件、延后提醒） | 只读（无交互）                   |
| 粒度     | 单个提醒卡片级别             | 三层叙事（matter / 个人 / 公司） |
| 触发     | 持续扫描，实时更新           | 每日定时，一次推送               |
| 输出形态 | 结构化卡片 + 操作按钮        | 自然语言因果叙事                 |

两者数据来源一致（都基于 index），但服务不同用户、不同场景、不同载体，不存在替代关系。日报模块在调度台上线后，可直接复用调度台的候选包作为 A 类与 B 类的输入。

### 与 index 的关系

日报以 index 为基础数据源。在 AI 正文读取权限开启的情况下，AI 可按需读取部分 Markdown 正文作为补充判断依据。但日报**不写入、不修改 index 或任何 matter 数据，也不创建任何文件**。日报始终是 Pivot 数据流的消费者，不是生产者。

### 与产品哲学的对应

本期设计延续 Pivot 的两条核心原则：

- **程序负责收敛问题空间，AI 负责高阶判断**：候选筛选由程序基于 index 字段执行确定性逻辑，叙事生成由 AI 在结构化候选包之上做因果理解
- **index 是事项解释层，原始 MD 是第一手事实**：日报默认只读 index；只有 AI 判定 index 解释信息不足时，才下钻到原始 MD

## 八、关键用户链路

### 场景一：CEO 早上看团队日报理解全局

1. 每天 09:00，系统自动聚合前一天的 matter 数据。
2. 程序筛选出 A 类（有推进）、B 类（需关注的无推进）、C 类（纯无推进）三类候选。
3. AI 基于 index 判断信息密度，对 2 个 matter 读取了正文（因 summary 过短）。
4. AI 生成 matter 视角的 6 段因果叙事 + 1 段停滞段落 + 公司视角的整体总结。
5. 推送到"管理层日报"飞书群。
6. CEO 打开飞书，先读公司视角（一句话掌握当日整体推进节奏为"积极"），再读 matter 视角（理解每件事今天发生了什么、下一步关注什么）。
7. 看到"消息推送改造已停滞 14 天"，决定 @ 该 matter 的 owner 在群里同步现状。

### 场景二：员工收到个人日报回顾产出

1. 每天 09:30，系统按人聚合前一天的 matter 活动。
2. 程序识别 D 类（今日有活动的人）和 E 类（今日无活动的人）。
3. AI 为 liuyu 生成个人视角叙事，发送到 liuyu 的飞书个人会话。
4. liuyu 打开飞书，看到自己昨天在哪些 matter 中有输入和输出。
5. 发现"权限体系重构"的 think 还没有后续 act 跟进，主动回到 Pivot 继续推进。

### 场景三：成员当日无活动也收到日报

1. wangwu 昨天在 Pivot 上没有任何动作。
2. 系统不静默跳过，仍生成一段一句话的个人日报："今天没有任何输入和输出。"
3. wangwu 收到后明确知道系统对他昨天的 Pivot 活动是空的——如果实际有线下推进，可主动补录到 Pivot；如果确实没有动作，至少日报本身没有产生信息错位。

### 场景四：AI 在 index 不足时下钻读取原文

1. 程序候选包中含一条 status_change：某 matter 从 executing 迁到 paused。
2. 该 matter 触发 status_change 的 think 文件 summary 仅写了"决定暂停"。
3. AI 判定因果链条不清——为什么暂停？什么条件下恢复？仅凭 summary 无法生成可靠叙事。
4. AI 自主读取该 think 的正文，发现暂停原因是"等待外部 API 稳定"。
5. AI 在 matter 视角中输出："今天 dengke 创建了一篇 think，决定将事项暂停，原因是依赖的外部 API 当前不稳定，预计待外部接口稳定后恢复。"
6. 后端记录本次日报生成中 AI 读取了 1 篇正文。

## 九、验收标准

| 验收项                | 标准                                                         |
| --------------------- | ------------------------------------------------------------ |
| 定时推送              | 配置的时间点准时推送，误差 < 5 分钟                          |
| 内容准确性            | 三层叙事中提及的事项状态、干系人、动作摘要均与 index 数据一致 |
| 因果叙事完整性        | matter 视角的每段叙事覆盖"是什么 / 前情 / 今日变化 / 当前状态 / 下一步"五个维度 |
| 公司视角必含状态判断  | 公司视角段落必须明确给出"积极 / 平稳 / 偏停滞"判断及一句话依据 |
| 个人视角无活动表达    | 当日无活动的成员必须输出"今天没有任何输入和输出"，禁止使用模糊话术或静默跳过 |
| 个人视角无绩效化      | 个人视角不出现排名、评分、产出量对比等表达                   |
| 无进展事项处理        | 常规无进展事项以一句话汇总；高风险无进展事项在停滞段落中点名 |
| 候选筛选正确性        | 程序对 A/B/C/D/E 五类候选的筛选与定义一致，可通过构造测试数据验证 |
| AI 读取原文行为可观测 | 后端记录读取次数、涉及文件、Token 消耗                       |
| AI 不写 index         | 全链路日报生成过程中无任何 index 写入或 matter 状态修改动作  |
| 篇幅控制              | 团队日报 500-800 字；单个个人日报 50-300 字（无活动时为单句） |
| 静默规则              | 团队当天无任何 matter 活动时不推送；个人当天无活动时仍生成显式空句并推送 |
| 端到端耗时            | 从触发到全部推送完成 < 3 分钟                                |
| 后台配置生效          | 管理员修改配置后，下次推送即按新配置执行                     |

## 十、开放问题

| 编号 | 问题                                      | 建议                                                         |
| ---- | ----------------------------------------- | ------------------------------------------------------------ |
| Q-01 | 首批支持哪个 IM 平台？                    | 根据内部使用情况，先接入飞书                                 |
| Q-02 | AI 正文读取的实验周期多长？               | 建议 2-4 周，对比开启 / 关闭正文读取的叙事质量差异           |
| Q-03 | 调度台未上线时 A/B 类候选由谁扫描？       | 日报模块自行实现简化版扫描，调度台上线后切换为读取其候选包   |
| Q-04 | 个人日报推送到管理 Channel 时的隐私顾虑？ | 默认推送到个人会话；推送到管理 Channel 需管理员显式配置并告知团队 |
| Q-05 | 后台配置界面是否需要独立设计？            | 放入 Pivot 现有系统设置中，作为"日报管理"子模块，不独立入口  |
| Q-06 | 因果叙事的事实准确性如何防虚构？          | AI 必须严格基于 index 与原文事实生成叙事，不允许虚构未发生的动作。后端可对叙事中提及的文件、动作做事后抽查（采样比对叙事与 index），发现虚构则标记并触发提示词调优 |
| Q-07 | "今日无活动"成员的范围如何界定？          | 默认覆盖所有在 Pivot 中至少创建过一次文件的成员；管理员可配置 opt-out 名单（如离职、长期休假、临时账号） |
| Q-08 | 三层叙事是否必须同时生成？                | 是。但如果某天团队没有任何 matter 有动作（A 类为空），matter 视角仅输出停滞段落 + 公司视角输出"偏停滞"判断；个人视角仍按 D/E 类生成 |

## 十一、二期方向

- **多 IM 平台支持**：飞书 + 钉钉 + Slack + 企业微信
- **周报 / 月报**：在日报基础上聚合更长周期的因果叙事，呈现事项级和方向级的演化曲线
- **交互式卡片**：在 IM 卡片内直接跳转到 Pivot 对应 matter 的时间线位置
- **日报历史存档**：在 Pivot 内可查看历史日报，按 matter 倒查其因果叙事的连续记录
- **个人视角的"长期叙事"模式**：从单日动作聚合为该成员近期参与事项的因果线
- **公司视角的"方向级叙事"模式**：从单日推进聚合为公司近期发力方向的演化叙事
- **跨 matter 关联叙事**：当 AI 识别到多个 matter 之间存在技术或业务关联时，在公司视角中点出该关联
