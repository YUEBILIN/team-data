# Pivot 最小可用化（MVP）产品需求文档


**版本：** v0.1 **状态：** 待讨论 **作者：** 产品经理Terry **日期：** 2026年4月27日

------

## 一、背景与问题

预期当前 Pivot 已上线基于 Git 的话题讨论功能，用户可以在 matter 下创建 think、act、verify、result、insight 等文件，形成时间线数据流。这套基础设施是可用的。

但随着 matter 数量增加，出现了一个新问题：**用户需要自己每天翻看所有 matter，才能知道哪些事情需要推进。** 系统没有主动感知能力。

这不是信息展示的问题，而是系统主动性缺失的问题。Pivot 的核心定位是"AI 调度者"，但目前 AI 是被动响应的。调度台是让这个定位第一次真正落地的功能。

### 用户真实困境

用户每天面对多个并行 matter，无法快速判断：哪个 act 超期了没人验收？哪个 matter 已经两周没有动静？哪个暂停事项到了该重新检查的时间？

这些判断目前靠人的记忆和手动翻阅，效率低，容易遗漏，也是 Pivot 作为"操作系统"最明显的能力空白。

------

## 二、目标

**MVP 目标：** 让用户每天打开 Pivot 后，不需要手动翻 matter，系统主动告知今天需要处理什么。

**不是目标：** 新的任务管理系统、新的事实来源。调度台只负责发现和提醒，改变 matter 的仍然是创建新文件。

### MVP 回答三个问题

1. 我今天需要处理什么？
2. 哪些 matter 的推进被卡住了？
3. 下一步我应该创建什么文件？

------

## 三、范围界定

### MVP 包含

- 调度台入口页面（个人视图）
- 基于 index 的结构化扫描（scheduler code）
- AI 对候选提醒的整理与表达（scheduler AI）
- 提醒卡片的四种状态：new / read / snoozed / resolved
- 按分组展示的提醒列表
- 进入 matter 后的上下文说明面板
- 补录引导流程

### MVP 不包含

- 团队视图（二期）
- 复杂权限控制
- 多级任务树
- AI 自动修改 matter 状态
- AI 全量阅读所有文件正文
- 提醒之间的依赖关系
- dismissed / archived 状态

------

## 四、核心机制设计

### 4.1 两层架构

调度台由两层组成，职责严格分开。

**第一层：Scheduler Code（结构化扫描）**

读取所有 matter 的 index，基于结构化字段进行确定性规则扫描，输出候选提醒包。不读正文，不做判断，只做发现。

**第二层：Scheduler AI（智能整理）**

接收候选提醒包，整理成用户可理解的语言，判断优先级，给出下一步建议。不直接改变 matter，只输出建议。

这个分层的核心价值是：**程序负责发现确定性问题，AI 负责理解和表达。** 避免 AI 全量扫描所有正文，保持响应效率。

### 4.2 扫描规则（MVP 覆盖）

MVP 阶段覆盖以下七类触发规则：

| 规则编号 | 触发条件                                               | 提醒类型          |
| -------- | ------------------------------------------------------ | ----------------- |
| R-01     | act 超过 due_at，但没有 verify 覆盖                    | needs_verify      |
| R-02     | matter 超过 N 天没有任何新文件                         | no_progress       |
| R-03     | paused matter 到达 next_check_at                       | paused_check      |
| R-04     | act 被 verify 判为 failed，但之后没有新的 think 或 act | failed_unresolved |
| R-05     | matter 已有充分 verify 支撑，可能可以创建 result       | ready_to_close    |
| R-06     | finished / cancelled matter 超过 N 天没有 insight      | needs_insight     |
| R-07     | result 缺少 verify 支撑                                | process_risk      |

> **关于 N 天的默认值：** MVP 阶段建议 no_progress 默认 7 天，needs_insight 默认 14 天，具体数值后续根据用户反馈调整，不在 MVP 阶段硬编码太多配置项。

### 4.3 与我相关的判定规则

调度台默认展示"与我相关"的提醒。MVP 阶段判定规则：

- 我是相关文件的 owner
- 我是相关文件的 creator
- 我在相关文件的 comments 中被 mention
- 我是某个 act 的 owner，而该 act 需要 verify
- 我创建或负责的 matter 长时间没有进展
- 我负责的 matter 到了 due_at 或 next_check_at

------

## 五、功能设计

### 5.1 调度台入口

调度台作为独立入口存在，与 matter 列表并列，不嵌套在某个 matter 内部。

入口位置建议放在主导航顶部或左侧导航突出位置，带未读数字角标，用于提示有新提醒。

### 5.2 提醒列表

#### 分组展示

MVP 阶段按以下分组展示，组内按优先级排序：

- **今天必须处理**（高优先级 + due_at 今日）
- **已经逾期**（超过 due_at）
- **等待验收**（needs_verify 类）
- **长时间无进展**（no_progress 类）
- **暂停事项到期复查**（paused_check 类）
- **可以考虑结束**（ready_to_close 类）
- **流程风险**（process_risk 类）
- **已处理**（resolved 状态，折叠展示）

#### 筛选能力

MVP 阶段支持：

- 只看我的 / 看全部
- 只看未处理 / 只看已逾期 / 只看等待验收 / 只看流程风险 / 只看已处理

筛选只影响调度台展示，不改变 matter。

### 5.3 提醒卡片

每张提醒卡片包含以下信息：

```
matter 名称
提醒类型（needs_verify / no_progress / 等）
优先级（high / medium / low）
责任人
触发原因（一句话说明）
相关证据文件（可点击跳转）
建议下一步动作
可执行操作按钮
```

**卡片示例：**

```
[high] 登录链路改造
等待验收

003_liuyu_act_xxx.md 已超过 due_at，当前没有 verify 覆盖。

责任人：liuyu
建议：请 liuyu 或对应验收人创建 verify

[创建 verify]  [创建 think]  [提醒 owner]  [延后提醒]
```

### 5.4 提醒卡片状态

提醒卡片有独立的生命周期，与 matter 状态解耦。

| 状态     | 含义                 | 触发方式                                    |
| -------- | -------------------- | ------------------------------------------- |
| new      | 新提醒，未看过       | 系统扫描生成                                |
| read     | 已看过，问题未解决   | 用户进入对应 matter，但未创建文件           |
| snoozed  | 用户延后提醒         | 用户点击"延后提醒"                          |
| resolved | 问题已被后续文件解决 | scheduler code 下次扫描确认触发条件不再存在 |

**关键原则：**

- 卡片被看过不代表问题解决，read 状态下卡片继续保留。
- 只有 scheduler code 扫描确认触发条件消失，卡片才变为 resolved。
- MVP 不加入 dismissed 状态，避免用户用"忽略"代替"解决问题"。

> **延后提醒的时间选项：** 明天、3 天后、一周后。选择后卡片进入 snoozed，到期自动恢复为 new。

### 5.5 进入 matter 后的上下文说明

用户点击提醒卡片后，打开对应 matter 详情页，并自动跳转到相关文件位置。

在 matter 详情页顶部或右侧，展示一个上下文说明面板：

```
为什么提醒你：
003_liuyu_act_xxx.md 已超过 due_at，且当前没有任何 verify 文件覆盖它。

建议下一步：
请 owner 或对应验收人创建 verify，判断该 act 是 passed、failed 还是 cancelled。

[创建 verify]  [创建 think]  [创建 result]  [提醒 owner]  [延后提醒]
```

操作说明：

- **创建文件**：进入正常的 matter 文件创建流程，系统自动填入 quote 指向相关文件。
- **提醒 owner**：仅发送通知，不改变 matter。
- **延后提醒**：仅改变调度台卡片状态，不改变 matter。
- **如果用户认为 matter 应该暂停或结束**：通过创建 think / result 文件来表达，不提供直接修改状态的按钮。

### 5.6 补录引导

调度台发现缺口时（如 act 实际已完成但没有 verify），引导用户补录，而不是直接修改状态。

MVP 补录流程：

1. 用户点击提醒卡片，打开 matter 和相关文件。
2. 上下文面板提示"这件事可能已经在线下发生了，建议补录"。
3. 用户说明实际发生了什么（文字输入框）。
4. AI 整理出建议创建的文件内容，供用户确认。
5. 用户确认后，写入 MD 文件并更新 index。

补录仍然走正常文件流，所有事实最终落在 matter timeline 里。AI 只负责帮用户整理内容，不自动写入。

## 六、与现有功能的关系

### 基于现有 Git 话题讨论能力延展

调度台不新建一套数据层，而是在现有 Git + index 基础上延展：

- **数据来源**：读取现有 matter index，不新增独立数据库。
- **状态变化**：仍然通过创建文件触发，调度台自身不修改 matter。
- **文件创建流程**：复用现有的 think / act / verify / result / insight 创建交互。
- **调度台自身的状态**（卡片 new/read/snoozed/resolved）单独维护，与 matter 状态解耦。

### 对 index 的新增字段需求

为支持部分扫描规则，index 需要补充以下字段（如尚未存在）：

- act 文件的 `due_at`
- matter 的 `due_at`
- paused matter 的 `next_check_at`
- 文件的 `updated_at`（最后活跃时间）

这些字段的写入仍然由程序控制，AI 只读取，不直接写入 index。

------

## 七、用户体验原则

**主动，但不打扰。** 调度台主动发现问题，但不强制用户立刻处理。snoozed 机制给用户足够的自主权。

**引导行动，不替代判断。** 提醒卡片给出建议下一步，但最终创建哪种文件、如何判断，由用户决定。AI 不自动修改任何 matter。

**最短路径到现场。** 点击提醒直接跳转到 matter 相关文件位置，上下文说明面板同步展示原因和建议，操作按钮就在旁边，减少用户在工具内的跳转成本。

**消失需要理由。** 提醒卡片不因为"被看过"而消失，只有触发条件真正被解决，卡片才进入 resolved。避免假完成。

------

## 九、二期方向（不在 MVP 内）

以下能力在验证 MVP 核心价值后推进：

- **团队视图**：回答"哪些人负责的 matter 卡住了，哪些事情需要我推动"
- **可配置的扫描规则**：用户或管理员调整 N 天阈值、触发规则开关
- **跨 matter 关系感知**：识别 matter 之间的依赖和阻塞关系
- **主动推送通知**：飞书 / 邮件 / App 通知，不只在产品内展示
- **AI 生成周报**：基于调度台数据，自动生成事项进展摘要

------

## 十、开放问题（开发前对齐）

在进入设计和开发之前，以下问题需要团队对齐：

1. **index 现有字段覆盖情况**：`due_at`、`next_check_at`、`updated_at` 是否已经在现有 index schema 中？
2. **扫描频率**：scheduler code 以什么频率扫描？实时触发、每小时、每天定时？频率影响提醒的及时性，也影响系统负载。
3. **R-05 "充分 verify 支撑"的判断标准**：如何定义"充分"？是所有 act 都有 verify，还是最近一次 verify 没有 failed 结果？这个规则需要在开发前明确，否则误报率会很高。
4. **调度台卡片状态的存储位置**：卡片状态（new/read/snoozed/resolved）独立于 matter，应该存在哪里？建议解耦开，不放入 Git，而是用轻量数据库存储，待确认。
