# Pivot 5月 MVP 产品功能全景文档（大纲示例）

## 一、 MVP 交付边界与核心目标
- **一句话定义成功标准**：5月底上线时，用户能完成什么？（如：通过调度台主动发现并处理待办，形成完整的 create-verify-close 闭环）
- **本期包含 / 明确不包含**：明确范围，砍掉二期干扰项。
- **核心用户画像与场景**：谁在用？高频场景是什么？

## 二、 产品架构与数据流总览
- **现有能力盘点**（简写）：Git 存储、Index 结构、基础文件流等。明确哪些不动。
- **新增模块位置**：调度台在架构中的定位，依赖哪些前置条件。
- **数据流转图**：用户操作 → Index 更新 → Scheduler 扫描 → 提醒生成 → 用户处理闭环。

## 三、 功能模块清单（按优先级排列）
### 3.1 基础数据与索引层（已有补齐）
- *编写指引*：列表形式+字段说明。重点写缺失字段（`due_at`, `updated_at` 等）的补齐方案，已实现的不要占篇幅。

### 3.2 核心交互层（增强）
- *编写指引*：流程图+关键交互。重点写“补录引导”、“上下文面板”、“快捷操作按钮”等增量点。

### 3.3 智能调度层（核心增量）
- *编写指引*：详细规则+状态机。可将你原有的调度台需求文档作为本节的子文档挂载。明确扫描规则、卡片状态流转、筛选分组。

### 3.4 辅助与触达层
- *编写指引*：轻量定义。数字角标、基础身份识别（owner/creator）、站内通知策略。

## 四、 关键用户链路 (User Journey)
- **典型场景串讲**：用 2-3 个场景（如：新建 matter → 推进 → 触发提醒 → 补录/验收 → 闭环）串起全貌，标注各环节涉及的前端/后端/AI 边界。

## 五、 验收标准与上线节奏
- **核心指标**：提醒覆盖率、误报率阈值、操作路径步数。
- **试跑计划**：灰度策略。
