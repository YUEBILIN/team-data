  Pivot 通知改版需求（v2 · 范围收缩版）
                                                                                                                                                                                                                                    
  阶段说明（本版新增）
  因底层 API 即将进行重大重构，新设计规范要求 summary 不再写入 MD 文件、仅保留在 index 中、由新 API 统一要求双写。本期需求范围相应收缩：原版中涉及 summary AI 提示词调整、MD/index
  双写、服务端格式校验、客户端强制升级拦截等条目，统一延期至新 API 上线后迭代，本期不投入相关开发。本期聚焦通知样式迁移（仅裁剪原文与版本号）与 mention 通知结构化。

  1. 目标

  将老版 2discuss（teamDocs/appv2）中的 notify 样式与版式移植到新版 Pivot，使新版通知在信息组织、可读性、可扫描性上达到老版水平。

  本次不是重新设计通知体系，而是以老版实现为基准做迁移，并规范化 mention 通知。

  summary 相关补齐能力本期不做，延期至新 API 上线后按新规范统一实现。

  2. 总体要求

  2.1 老版对标

  先阅读老版 2discuss（teamDocs/appv2）下 notify 的实现。
  重点是迁移其通知的样式、版式、信息层次和阅读体验，不只是表面 UI。

  2.2 本次允许保留的差异

  相对老版通知，本次只保留以下两点差异：

  不再附带原始文件全文
  原因：现在通知里的链接可以直接跳转查看原帖，没必要重复带全文。

  不再展示版本号
  原因：现在发版由服务器端统一完成，客户端用户无需关心当前服务器版本。

  除这两点外，通知的整体组织方式应尽量向老版对齐。

  2.3 summary 能力延期说明

  summary 的客户端生成、格式校验、MD 与 index 双写、服务端发布校验、客户端因 summary 触发的强制升级拦截等能力，本期统一不做。新 API 上线后，按新规范（summary 仅写 index、由新 API 统一双写）再行实现。

  本期通知生成在 summary 缺失时降级为不展示摘要字段，不应因此拒绝发布。

  3. [本期已删除 · 原 Summary 补齐要求章节]

  本章节原规定 summary 的生成方式、写入位置（MD + index 双写）、格式约束、客户端升级要求、服务端发布校验等。因底层 API 即将重构（见阶段说明），此整章延期至下一阶段，新 API 就绪后按新规范重新定义并实现。

  4. 通知内容调整要求

  4.1 常规通知

  迁移老版通知样式与版式时：

  保留老版通知的信息组织优势
  删除原始全文
  删除版本号

  除此之外，不要随意删减老版通知中有价值的信息层次。

  4.2 [本期已删除 · 原 Summary 在通知中的作用 节]

  因 summary 能力延期（见 §2.3 阶段说明），通知侧本期不作 summary 相关消费要求。待新 API 上线后，通知侧按新规范消费 index 级 summary。

  5. Mention 通知要求

  mention 是新版新增能力，不能简单沿用老版普通通知结构，必须单独按新版语义设计清楚。

  5.1 必须展示的信息

  无论是群发消息还是单人消息，mention 通知都必须明确包含：

  这次 mention 圈了哪些人
  这是针对哪一个主题（thread）
  这是针对该主题下哪一条帖子（post）
  mention 的内容是什么
  原始跳转链接

  5.2 跳转要求

  通知中必须带可点击链接
  点击后必须直接跳转到被 mention 的那一条帖子，而不是只跳到 thread 页面

  5.3 展示原则

  mention 通知要让接收者一眼看清：

  谁提到了谁
  在哪个主题下
  针对的是哪条帖子
  提到的具体内容是什么
  点哪里可以直接过去看

  6. 研发拆分

  6.1 老版实现阅读

  先阅读并拆解老版 appv2/notify，至少输出以下结论：

  哪些样式和版式需要迁移
  哪些字段要保留
  哪些字段在新版要删除
  mention 在新版应新增哪些信息结构

  （原"summary 的老版约束是什么"条目随 §3 一同延期）

  6.2 客户端侧

  支持发布失败错误的正确展示（通用错误路径）
  VSCode 客户端的版本自升级机制（UpdateManager）继续保留并运行，本期不因 summary 能力新增拦截用例
  Web 客户端的错误 toast 能展示服务端返回的任意 detail 字符串

  （原"调整 AI 生成最终发布 MD 的提示词"、"Web 客户端正确展示'缺少 summary'/'summary 非法'等发布错误"随 §3 一同延期）

  6.3 服务端侧

  调整通知生成逻辑：
  去掉原始全文
  去掉版本号
  新增符合要求的 mention 通知内容结构

  （原"发布 API 增加 MD 格式校验"、"校验 summary 是否存在且合法"、"校验 summary 是否同步写入 index"、"缺失或非法时拒绝发布"、"返回标准化错误信息"随 §3 一同延期）

  7. 验收标准

  满足以下条件即可视为完成：

  新版通知整体样式和版式明显对齐老版 2discuss/appv2/notify
  通知中不再附带原始全文
  通知中不再展示版本号
  mention 通知能明确展示：
  圈了谁
  对应哪个主题
  对应哪条帖子
  mention 内容
  可直达该帖子的链接

  （原"客户端 AI 生成最终发布内容时会带上合法 summary"、"发布成功后对应 index 中能看到同步写入的 summary"、"summary 的格式与限制与老版 2discuss 保持一致"、"缺少 summary、summary 非法、或未正确写入 index
  时，服务端都会拒绝发布"、"拒绝发布的错误信息能在 VSCode 客户端和 Web 客户端正确展示"、"VSCode 客户端升级后可强制要求旧终端升级" 等条目随 §3 一同延期至新 API 阶段）
