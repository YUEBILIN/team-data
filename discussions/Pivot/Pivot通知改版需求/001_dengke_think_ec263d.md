# Pivot 通知改版需求

## 1. 目标

将老版 `2discuss`（`teamDocs/appv2`）中的 `notify` 样式与版式移植到新版 Pivot，使新版通知在信息组织、可读性、可扫描性上达到老版水平。

本次不是重新设计通知体系，而是以老版实现为基准做迁移，并补齐新版当前缺失的 `summary` 能力，以及规范化 `mention` 通知。

## 2. 总体要求

### 2.1 老版对标
- 先阅读老版 `2discuss`（`teamDocs/appv2`）下 `notify` 的实现。
- 重点是迁移其通知的样式、版式、信息层次和阅读体验，不只是表面 UI。

### 2.2 本次允许保留的差异
相对老版通知，本次只保留以下两点差异：

- 不再附带原始文件全文  
  原因：现在通知里的链接可以直接跳转查看原帖，没必要重复带全文。
- 不再展示版本号  
  原因：现在发版由服务器端统一完成，客户端用户无需关心当前服务器版本。

除这两点外，通知的整体组织方式应尽量向老版对齐。

## 3. Summary 补齐要求

### 3.1 业务要求
新版目前无论 Web、VSCode 客户端还是移动端，都没有真正实现 `summary`。本次必须补齐，并将其变成发布时的硬性要求。

### 3.2 生成方式
- `summary` 仍由客户端侧 AI 或移动端 AI 主动生成。
- 需要调整 AI 在“生成最终发布 md 文件”时使用的提示词。
- 目标是让 AI 生成的最终发布内容天然带有符合规范的 `summary`，而不是发布后补写。

### 3.3 写入位置
`summary` 必须同时写入以下两个位置，缺一不可：

- 发布的 MD 文档结构字段
- 对应 `index` 中的 `summary` 字段

不能只写 post，不写 index。  
`index` 中的 `summary` 才是后续列表展示、通知生成、AI 消费等场景可依赖的标准摘要来源。

### 3.4 约束规则
- `summary` 的格式、长度、内容限制，统一参照老版 `2discuss` 的实现。
- 不要在新版里自行重新定义一套近似规则。
- 研发在开始实现前，应先把老版 `2discuss` 对 `summary` 的约束整理清楚，并以此作为新版兼容基线。

### 3.5 客户端升级要求
- VSCode 客户端升级后，必须要求所有终端强制升级。
- 未升级的终端如果继续使用旧格式发帖，应直接报错，不允许静默兼容。
- VSCode 客户端已经实现了相关机制，本次应阅读其说明文档并复用现有方案，不要重新设计一套升级拦截逻辑。

### 3.6 服务端发布校验
服务端发布 API 必须增加程序化校验：

- 校验最终发布的 MD 文档格式是否符合规范
- 校验是否包含合法的 `summary`
- 校验 `summary` 是否已按规范同步写入 `index`
- 如果缺少 `summary`，或 `summary` 不符合老版约束，则直接拒绝发布
- 返回明确、可展示、可理解的错误信息

该错误信息必须能在以下端正确显示：

- VSCode 客户端
- Web 客户端

原则如下：

- `summary` 不是建议项，而是发布门槛
- `summary` 不只是 post 级字段，也是 index 级字段
- 约束标准以老版 `2discuss` 为准
- 校验由服务端兜底，不依赖客户端自觉

## 4. 通知内容调整要求

### 4.1 常规通知
迁移老版通知样式与版式时：

- 保留老版通知的信息组织优势
- 删除原始全文
- 删除版本号

除此之外，不要随意删减老版通知中有价值的信息层次。

### 4.2 Summary 在通知中的作用
通知侧应消费已经标准化写入的 `summary`，不应继续依赖正文截取、全文附带或临时拼接来代替摘要。

## 5. Mention 通知要求

`mention` 是新版新增能力，不能简单沿用老版普通通知结构，必须单独按新版语义设计清楚。

### 5.1 必须展示的信息
无论是群发消息还是单人消息，`mention` 通知都必须明确包含：

- 这次 mention 圈了哪些人
- 这是针对哪一个主题（thread）
- 这是针对该主题下哪一条帖子（post）
- mention 的内容是什么
- 原始跳转链接

### 5.2 跳转要求
- 通知中必须带可点击链接
- 点击后必须直接跳转到被 mention 的那一条帖子，而不是只跳到 thread 页面

### 5.3 展示原则
`mention` 通知要让接收者一眼看清：

- 谁提到了谁
- 在哪个主题下
- 针对的是哪条帖子
- 提到的具体内容是什么
- 点哪里可以直接过去看

## 6. 研发拆分

### 6.1 老版实现阅读
先阅读并拆解老版 `appv2/notify`，至少输出以下结论：

- 哪些样式和版式需要迁移
- 哪些字段要保留
- 哪些字段在新版要删除
- `summary` 的老版约束是什么
- `mention` 在新版应新增哪些信息结构

### 6.2 客户端侧
- 调整 AI 生成最终发布 MD 的提示词，确保生成合法 `summary`
- 支持发布失败错误的正确展示
- VSCode 客户端接入强制升级机制
- Web 客户端正确展示“缺少 summary”“summary 非法”等发布错误

### 6.3 服务端侧
- 发布 API 增加 MD 格式校验
- 校验 `summary` 是否存在且合法
- 校验 `summary` 是否同步写入 `index`
- 缺失或非法时拒绝发布
- 返回标准化错误信息
- 调整通知生成逻辑：
  - 去掉原始全文
  - 去掉版本号
  - 新增符合要求的 `mention` 通知内容结构

## 7. 验收标准

满足以下条件即可视为完成：

- 新版通知整体样式和版式明显对齐老版 `2discuss/appv2/notify`
- 通知中不再附带原始全文
- 通知中不再展示版本号
- 客户端 AI 生成最终发布内容时会带上合法 `summary`
- 发布成功后，对应 `index` 中能看到同步写入的 `summary`
- `summary` 的格式与限制与老版 `2discuss` 保持一致
- 缺少 `summary`、`summary` 非法、或未正确写入 `index` 时，服务端都会拒绝发布
- 拒绝发布的错误信息能在 VSCode 客户端和 Web 客户端正确展示
- VSCode 客户端升级后可强制要求旧终端升级
- `mention` 通知能明确展示：
  - 圈了谁
  - 对应哪个主题
  - 对应哪条帖子
  - mention 内容
  - 可直达该帖子的链接
