# 新需求-pivot已读状态支持

## 背景

目前 Pivot 中一篇文档发布后，系统无法清楚展示哪些员工已经阅读过这篇文档。对于团队协作来说，这会带来一个问题：作者或管理者不知道某篇重要文档是否已经被相关成员看到。

可以参考飞书文档中的已读状态展示方式，让 Pivot 能够记录并展示每个员工对某篇文档的阅读状态。

## 目标

第一期先支持“已读状态”。

系统只记录并展示：

> 某个员工是否已经读过某一篇文档。

暂时不做完整的“未读状态”判断。

## 需求描述

当用户明确打开或展开某篇文档并完成阅读行为后，系统应记录该用户已经阅读过这篇文档。

在文档界面中，需要提供已读状态展示，让用户可以看到哪些员工已经读过这篇文档。

第一期的重点是记录真实发生过的阅读行为，而不是判断谁应该阅读。

## 已读状态的判定规则

目前 Pivot 中的文章默认是收缩展示状态，只显示前面一小部分内容。如果一篇文章处于收缩状态，用户只是看到了列表或预览内容，但没有主动展开阅读，则不应标记为已读。

已读状态必须基于用户明确打开或展开文档后的阅读行为产生。

第一期可以采用以下规则：

- 用户进入 matter 页面，只看到收缩状态文章，不记为已读。
- 用户点击展开某篇文章，记为该用户已读该文档。
- 如果文章本身足够短小，系统默认完整展开展示，则可以在用户看到该文章时记为已读。
- 如果后续需要更精细的阅读判断，例如停留时间、滚动到底部等，可以再扩展。

## 已读状态记录内容

系统需要记录：

- 文档 ID。
- 阅读用户。
- 首次阅读时间。
- 最近阅读时间。
- 阅读来源，例如展开阅读、完整展示、直接打开等。

如果同一个用户多次阅读同一篇文档，系统可以保留首次阅读时间和最近阅读时间。

## 数据存储要求

已读状态不应写入 Markdown 文档正文、frontmatter 或 matter index。

原因是：已读状态是“某个用户和某篇文档之间发生过一次阅读行为”的事实，而不是文档自身的属性。

一篇文档可能被很多员工阅读，每个人都有自己的首次阅读时间和最近阅读时间。如果把已读状态写入 Markdown frontmatter 或 index，会导致：

- 每个人读一次都要修改文档或 index。
- 文档时间线会被阅读行为污染。
- 数据仓库里充满和文档内容无关的已读状态变更。
- Pivot 的只读时间线模型被破坏。
- 后续查询、统计和权限控制也不方便。

更合理的做法是单独保存阅读记录，例如建立独立的阅读记录表，用于记录用户和文档之间的阅读关系。

示例字段包括：

- 文档 ID。
- 用户 ID。
- 首次阅读时间。
- 最近阅读时间。
- 阅读来源。

前端展示某篇文档的已读员工列表时，应由 API 聚合返回，而不是把这些状态写回 Markdown 或 index。

## 未读状态暂不实现

第一期不做未读状态。

因为“未读”不只是“没有读过”，还隐含一个前提：

> 这个用户本来应该阅读这篇文档。

目前 Pivot 中还没有完整的提醒机制、阅读分配机制、owner 关联机制或个人相关未读机制。因此现在直接显示未读，容易产生错误理解和不必要压力。

## 后续扩展方向

等后续具备以下能力后，可以再扩展未读状态：

- 提醒某个员工阅读某篇文档。
- 某篇文档被分配 owner。
- 某个 matter 或文件被判定与某个员工相关。
- 管理员或作者指定某些员工应阅读某篇文档。

到那时，系统可以标记：

> 某个员工应该读这篇文档，但尚未阅读。

这才是真正有意义的未读状态。

## 期望结果

第一期上线后，Pivot 可以展示每篇文档的已读情况，让团队知道哪些人已经实际看过文档。

同时，系统不会过早引入“未读”概念，避免把所有没有阅读过的人都错误地标记为未读。

第一期只解决阅读事实记录，后续再基于提醒、owner 和相关性机制扩展阅读义务与未读提醒。
