# Web 端 AI 助手与移动端 UI 改造需求

## 一、目标

1. AI 助手改造  
网页端 AI 不再依赖用户手动指定 `target` 和 `reference files`。  
用户在某篇帖子上点击“AI 回复”后，这篇帖子就是 AI 的第一阅读对象。

2. AI 补上下文能力  
如果只读当前帖子还不够，AI 再自主：
- 读取当前 thread 的 `index`
- 根据 `index` 继续读取相关文件

这些文件大多数位于当前 thread 内，但这次方案要顺带支持 `index` 指向的跨 thread 文件。

3. 移动端体验优化  
优化手机上的 AI 助手显示和交互，让“阅读帖子 -> 问 AI -> 生成草稿 -> 回复发布”这条链路更顺。

## 二、现状问题

- 当前 AI 只能消费前端手工传入的 `reply_target` 和 `reference_files`
- 它不会自己决定是否读取 `index`
- 也不会顺着 `index` 继续读取相关文件
- 所以它本质上仍是“被动阅读器”，不是“会自己补上下文的助手”

## 三、改造要求

1. AI 起始上下文  
用户点击某篇帖子的“AI 回复”后，AI 默认先读取这篇帖子本身。

2. AI 补读顺序  
当当前帖子不足以支撑交流时，AI 再自主：
- 读取当前 thread 的 `index`
- 根据 `index` 继续读取相关文件

3. 文件读取方式  
AI 不去 GitHub 拉文件。  
所有补上下文动作都通过本地服务器 API 完成，由服务端直接从本地 workspace 读取文件并返回。

4. 前端要求  
- 保留“某篇帖子上的 AI 回复”入口
- 去掉手工选择 `target` / `reference files` 的主流程
- 可以展示“AI 已读取哪些文件”，但这应该是状态展示，不再是用户配置项

5. 后端要求  
- 支持读取当前回复帖子
- 支持读取当前 thread 的 `index`
- 支持按 `index` 线索读取相关文件
- 读取范围由后端控制，不能让 AI 自由扫整个仓库

6. Chat 模式要求  
- 不再由前端一次性拼好上下文再发给 AI
- 改为 AI 在对话过程中按需请求更多文件

## 四、移动端要求

- 手机上“AI 回复”入口要更清晰
- AI 面板要更适合小屏，不要只是桌面侧栏的缩小版
- 对话区、输入区、关键操作区要更清楚
- 生成草稿后，进入回复编辑和发布的路径要更短

## 五、验收标准

- 点击某帖“AI 回复”后，AI 先读取这篇帖子
- 当上下文不足时，AI 会自己读取当前 thread 的 `index`
- 之后 AI 会自己继续读取相关文件
- 全程不需要用户手工选择 `target` 或 `reference files`
- 全程不走 GitHub，全部通过本地 API 读取文件
- 手机上可以顺畅完成“阅读帖子 -> AI 讨论 -> 生成草稿 -> 修改 -> 发布”整条链路

## 六、开发与测试要求

- 开发过程中先在本地部署 Pivot，自测通过后再上服务端
- 如果修改后端，尤其涉及持久化数据库、Git、index 相关逻辑，必须谨慎并加强测试
- 部署前应确认新能力不会破坏现有 thread、draft、reply publish 等流程
