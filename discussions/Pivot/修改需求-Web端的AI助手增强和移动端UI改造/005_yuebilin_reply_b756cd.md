yuebilin，分析得很透彻，现状盘点也很准确。针对你提出的三个核心问题，结合我们的产品设计方向，回复如下：

### 1. Tool use 升级是否算在本次需求内
**结论：算，必须包含在内。**

这不仅仅是技术架构的升级，而是决定 AI 能否真正成为"事项调度者"的关键。
正如我们在设计哲学里定义的，Pivot 的 AI 不能只是一个被动接收上下文的"问答机"（那是传统 AI 客服的路子），而必须是能理解工作流状态、自主判断需要哪些信息、按需获取并推动事项的"调度者"。
如果不做 tool-calling loop，AI 就失去了自主性，只能退化成服务端无脑拼上下文的"人工喂料"模式，这与我们要做的产品定位不符。因此，这块工作量必须纳入本次需求。

### 2. 跨 Thread 引用边界
**结论：采纳方案 B（保留逃生舱）。**

同意你的判断。我们要解决的是"减少用户手工组装上下文"，而不是"禁止用户指定上下文"。
在实际协作中，index 的 refs 登记可能会有滞后。如果严格按白名单（方案 A），会导致用户明明想引用某个文件但 AI 读不到，严重阻碍工作流。
保留一个次要入口（逃生舱），允许用户为本次会话临时追加 1-2 个任意文件，既满足了核心场景的自动化，又保留了灵活性，风险更小。

### 3. 读取预算 / 频控
**结论：必须定死。请在正式设计文档中给出具体参数和截断策略。**

初步建议如下，供参考：
* **单轮 Tool Call 上限**：建议 ≤ 5 次。AI 应具备"信息够用即停"的判断力，避免陷入循环。
* **文件体积上限**：沿用 `MAX_CONTEXT_CHARS` 或类似逻辑。如果 index 的 summary 质量高，优先让 AI 读 summary，而非全文。
* **上下文截断**：当超出 Token 限制时，策略上应优先保留：`当前 Thread Index 摘要` + `当前帖子内容` + `最近几次 Tool Result 结果`。丢弃早期的 Tool 调用细节，但要保留结果的摘要。

### 4. 整体设计与下一步
初步设计的骨架方向是对的。关于移动端，既然我们的 AI 定位是"调度助手"，面板设计可以重点考虑：在小屏上如何最直观地展示"AI 查到了什么关键信息"以及"基于此建议用户下一步做什么"。
