# Context

本 act 承接当前需求，进入 Web 端 AI 助手增强与移动端 UI 改造的实现阶段。目标是把 AI 助手从“依赖用户手工指定 target / reference files”的模式，改造为“基于当前帖子起步、按需补充上下文”的助手；同时优化手机端从阅读帖子到 AI 回复、生成草稿、编辑发布的完整链路。

# What To Do

- 实现 AI 回复入口的新主流程：用户在某篇帖子上点击“AI 回复”后，系统自动把该帖子作为 AI 的第一阅读对象。
- 去掉主流程中的手工 `target` / `reference files` 配置，把“AI 已读取哪些文件”作为状态展示，而不是用户配置项。
- 后端提供受控的本地文件读取能力：支持读取当前回复帖子、当前 帖子 的 index，以及 index 指向的相关文件。
- 支持 index 指向的跨 帖子 文件读取，但读取范围必须由后端控制，不能让 AI 自由扫描整个仓库。
- Chat 模式改为 AI 在对话过程中按需请求更多文件，不再由前端一次性拼好完整上下文。
- 改造 Web 端 AI 面板状态展示，让用户能看到 AI 已读取的关键文件。
- 优化移动端 AI 回复体验：让“阅读帖子 -> 问 AI -> 生成草稿 -> 回复编辑 -> 发布”的路径更清楚、更短。
- 在手机端重新组织 AI 面板的对话区、输入区、草稿区和关键操作区，避免直接缩小桌面侧栏。
- 本地部署 Pivot 并完成自测后，再上服务端。

# Verification

- 点击某帖“AI 回复”后，AI 首先读取这篇帖子。
- 当上下文不足时，AI 能主动读取当前 帖子 的 index。
- AI 能根据 index 继续读取相关文件，包括受控的跨 项目 文件。
- 全程不需要用户手工选择 `target` 或 `reference files`。
- 全程不走 GitHub，所有上下文读取都通过本地服务器 API 完成。
- 手机上可以顺畅完成“阅读帖子 -> AI 讨论 -> 生成草稿 -> 修改 -> 发布”。
