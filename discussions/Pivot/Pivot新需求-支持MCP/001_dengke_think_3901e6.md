# Pivot新需求-支持MCP

我们需要基于新的 Matter API，同步启动一条外部 AI 接入线，让 Pivot 不只支持内置 AI，也支持用户使用自己偏好的 AI 终端参与讨论与写作，例如 Codex、Claude Code、ChatGPT 类产品，以及后续可能接入的其他 AI 客户端。

这项工作的背景是：Pivot 正在从旧的 thread/reply 模型切换到新的 matter/timeline 模型，老 API 以及建立在其上的 VS Code 客户端和其他第三方客户端，后续都可能失效或需要重构。如果我们只推进新 Web 和内置 AI，而不同时定义统一的外部 AI 接入方式，后面不同客户端很可能各自接一套，形成新的碎片化。

这项需求要解决的核心场景是：用户在 Pivot Web 中浏览某个 matter 的 timeline，看到某篇文件后，希望借助自己熟悉的外部 AI 一起阅读、分析、讨论或起草下一篇文件。用户应当能够从 Web 页面上轻量复制一个上下文标识，把它交给外部 AI，外部 AI 再通过统一接入方式定位到对应的 matter 和文件，并继续协作。

在协议选择上，本项目应采用：

**MCP 为主，CLI 为辅。**

原因很简单：

1. 我们要服务的是 AI 终端，而不是单纯服务人工命令行用户。
2. 新 Matter API 本身就是结构化的，适合被建模为 AI 可调用的 tools。
3. MCP 更适合作为多种 AI 客户端共享的正式接入面。
4. CLI 仍然有价值，但更适合承担调试、脚本化、fallback 和 MCP 宿主等辅助角色，而不应成为主要产品接口。

因此，这项工作的正确定位不是“做一个命令行工具”，而是为 Pivot 建立一套正式的外部 AI 协作接入面。CLI 可以做，但应建立在同一套能力之上，而不是单独再定义一套业务模型。

这项工作的目标是：

- 基于新的 Matter API，定义统一的外部 AI 能力面
- 支持外部 AI 读取 matter 列表、matter 详情、timeline 和具体文件内容
- 支持外部 AI 在用户确认后提交新的 `think / act / verify / result / insight` 文件
- 支持从 Web 页面复制某种稳定的上下文标识，并让外部 AI 能据此恢复正确上下文
- 让外部 AI 的接入不绑定某一个具体产品，而是建立在统一协议之上

后续研发设计时应注意几个原则：

1. 新 Matter API 是唯一业务底座，不要让 MCP 或 CLI 直接绕过它写底层文件。
2. MCP 是正式协议，CLI 是辅助壳层。
3. 对外能力面应围绕 `matter` 和 `file` 组织，不应继续停留在旧 thread 语义上。
4. 外部 AI 的写入必须以结构化字段显式提交为前提，不能依赖正文反向解析。
5. 第一版优先保证“读取上下文 -> 讨论 -> 起草 -> 提交”这条闭环，不必一开始追求大而全。

这项需求完成后，至少应达到以下效果：

- 用户可以在 Web 中复制某个 matter 或具体文件的上下文标识
- 外部 AI 可以基于该标识恢复正确上下文并继续讨论
- 外部 AI 可以帮助用户整理正文和结构化字段
- 用户确认后，可以通过正式接入能力把内容写回 Pivot
- 这套流程不依赖某一个 AI 产品，也不依赖旧 API

一句话总结：这项工作的目标，是基于新 Matter API，为 Pivot 建立一套 **MCP 为主、CLI 为辅** 的外部 AI 协作接入能力。
