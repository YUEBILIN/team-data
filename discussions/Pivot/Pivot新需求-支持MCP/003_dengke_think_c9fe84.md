我这里补充一个关于外部 AI 接入方式的判断，核心结论是：

**Pivot 未来如果要支持 Codex、Claude Code、ChatGPT 类产品以及其他 AI 终端，应该默认只靠 MCP 就能工作，而不应把 skill 作为前置依赖。**

原因很简单。

MCP 本身就是标准的工具接入层。只要 MCP server 把 tool 的名称、说明、参数结构、返回结构以及必要的使用约束定义清楚，LLM 在运行时就能理解这些能力，并决定如何调用。它不是先“学会”再使用，而是在使用时直接读取 MCP 暴露出来的工具信息。

因此，对 Pivot 来说，更合理的设计原则应该是：

**把关键协作流程尽量固化在 MCP 自身，而不是依赖某个 AI 终端私有的 skill。**

这件事尤其重要，因为我们的目标不是只适配某一个客户端，而是希望不同 AI 终端都能接入同一套能力面。如果流程主要写在某个客户端的 skill 里，那这套逻辑就很难迁移，也不利于后续统一维护。

所以第一原则应该是：

- MCP 自己就把阅读、定位、起草、确认、提交这些流程表达清楚
- AI 终端拿到 MCP 后，就应该能够直接工作
- 不要求用户额外安装一个 skill，系统才能正常使用

当然，这不等于 skill 永远没有价值。  
如果后续某个具体客户端在使用 MCP 时仍然经常出现误用，或者我们希望在某个平台上进一步强化某种协作习惯，那么可以再补一个该平台专用的 skill 作为增强层。

但这个顺序不能反过来。  
正确顺序应该是：

1. 先把 MCP 设计正确  
2. 让系统在没有 skill 的情况下也能完整工作  
3. 如果确实有必要，再补 skill 做局部增强

另外，单纯在 MCP 描述里写很多操作说明，也不是最稳的做法。  
如果某个流程非常关键，最好不要只靠说明文字让模型自己猜，而是直接通过 tool 的拆分和约束把流程做进去。

例如：

- 先有读取上下文的 tool
- 再有准备草稿但不提交的 tool
- 再有校验结构化字段的 tool
- 最后才有真正提交写入的 tool

这样模型不是“读说明理解流程”，而是被工具本身的结构自然约束。

一句话总结：

**对于 Pivot 这种面向多种外部 AI 终端的接入方案，应该默认“只靠 MCP 就能用”，而不是默认“还要再装一个 skill 来教它怎么用”。skill 可以有，但只能是增强层，不能是主方案。**
