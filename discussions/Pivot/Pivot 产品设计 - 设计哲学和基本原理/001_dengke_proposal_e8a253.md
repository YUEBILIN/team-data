# Pivot 产品设计 - 设计哲学和基本原理



## 一、设计哲学

### 1. 企业所有工作都可以抽象为在时间线上持续演进的事项

Pivot 对企业工作的基本理解是：企业不是在处理一堆彼此孤立的模块，而是在持续推进大量事项。每个事项从被提出开始，都会沿着自己的时间线不断演进，直到完成计划、执行、验收与回顾的完整闭环。

事项是 Pivot 的顶层对象，英文名为 `matter`。事项不是静态对象，而是一条持续演进的工作过程。Pivot 的视角，不是围绕模块组织信息，而是围绕事项的演进组织信息。

### 2. 企业工作的本体是工作数据流

事项沿时间线演进的过程，不应只存在于人的记忆、口头沟通和即时聊天里，而应持续沉淀为可追溯、可分析、可计算的工作数据流。Pivot 的基础，不是单纯的“文档化”，而是把企业工作过程沉淀为一条条结构化、连续化的数据流。

在这个定义下，Markdown 文档不是系统的顶层对象，而是事项在时间线上留下的可追溯、可分析的足迹。文档记录了事项如何被提出、如何被讨论、如何被拍板、如何被执行、如何被验收和回顾。没有这些足迹，事项就无法被真正理解，也无法被持续推进。

### 3. AI 是事项调度者

Pivot 中的 AI 不是建议者，而是调度者。它的职责不是局部回答问题，而是基于事项的数据流理解该事项当前所处位置、判断下一步应当发生什么、推动事项进入下一阶段，并在整个过程中持续协调、提醒、收口和沉淀知识。

因此，Pivot 的一句话定义可以表述为：

**Pivot 是一个以工作数据流为基础、以 AI 调度为核心的企业工作操作系统。**

## 二、基本原理

### 1. 事项有主阶段，阶段描述的是事项状态

每个事项在任一时刻都处于一个主阶段。当前，Pivot 对事项的标准阶段定义为四个：

1. `plan`
2. `execution`
3. `acceptance`
4. `review`

这四个阶段构成事项的完整闭环。阶段描述的是事项整体当前处于什么状态，而不是在描述某一篇文档的类型。

### 2. 文档属于数据流，文档类型不等于事项阶段

事项的阶段和文档的语义不能混为一谈。事项进入 `execution` 后，仍然可能继续产生 `plan` 阶段相关文档；事项进入 `acceptance` 后，也仍然可能产生新的 `execution` 文档。这种情况是正常的，因为事项的主状态和文档在时间线中的位置本来就不是同一个维度。

因此，Pivot 的系统设计必须满足两个条件：一方面，系统要能明确表达事项当前所处的主阶段；另一方面，系统又不能用事项阶段去强行替代所有文档语义。

### 3. 数据流应保持开放，系统解释应收敛到上层

工作过程中的文档语义天然具有开放性和模糊性。不同事项、不同团队、不同场景下形成的文档，可能承担完全不同的局部作用。因此，Pivot 不应试图构建一个无限膨胀的细粒度文档角色体系。文档层应尽量保持开放，只保留少量必要的结构化字段。

系统级解释能力不应压在文档本体上，而应收敛到更高一层。也就是说，文档负责保留工作足迹，系统负责在更高层对这些足迹进行理解、提炼和组织。

### 4. `index` 是事项解释层

`index` 的作用，是围绕事项的数据流形成系统级解释。它不是原始事实层，也不是原始内容本身，而是系统对事项时间线的结构化理解。原始 Markdown 文档仍然是第一手事实材料；`index` 则承载系统围绕这些材料得出的解释。

因此，`index` 的定位应当明确为：**程序与 AI 共享的事项解释层。**

`index` 应当服务两类主体：

- 程序：获得稳定的结构抓手
- AI：获得高密度的上下文入口

### 5. `index` 记录解释，不记录原始事实本身

`index` 中适合记录的是围绕事项数据流形成的系统解释信息，例如：

- timeline
- 当前阶段
- 阶段跃迁点
- 关键文档引用
- 文档摘要
- 文档重要性
- AI 对关键节点的说明

这里要明确两个边界。第一，文档重要性不等于阶段跃迁锚点；第二，阶段跃迁最好在发生时就被显式记录，而不是事后主要依靠推断。

### 6. `index` 的结构控制权必须属于程序

既然 `index` 同时服务程序和 AI，那么它必须保持结构稳定和语义一致。因此，`index` 的 schema 与写入控制权必须属于程序。AI 可以提供候选解释内容，但不直接拥有自由写 `index` 的权限。

更合理的分工是：

- 程序定义 `index` 结构
- AI 提供摘要、重要性、锚点判断、跃迁说明等候选输入
- 程序负责校验、约束、写入和维护

### 7. 程序负责收敛问题空间，AI 负责高阶判断

Pivot 的 AI 驱动流程不应走向两个极端。一方面，不应让 AI 每次自由读取所有内容再自行判断一切；另一方面，也不应让程序把所有逻辑都算死，AI 只负责输出文本。更合理的方式是：程序先基于 `index` 做确定性筛选，构造最小上下文包；AI 再在这个上下文上做高阶理解、判断、生成和调度；必要时再下钻到原始 Markdown。

归纳起来就是一句话：

**程序负责收敛问题空间，AI 负责高阶判断和事项推进。**

## 三、当前产品落地

### 1. 当前 Pivot 已经具备 `plan` 阶段的大部分基础能力

从产品现状看，Pivot 并不是从零开始。现有系统已经具备了围绕事项展开讨论、持续沉淀 Markdown 文档、通过状态流转表达推进、借助 AI 理解上下文、通过 Git 保存数据流等基础能力。也就是说，当前系统已经承担了事项 `plan` 阶段的大部分基础设施能力。

因此，当前产品的主问题不是“如何从零实现 plan”，而是“如何在已有基础上完成从 `plan` 到 `execution` 的闭环”。

### 2. `concluded` 表示 `plan` 阶段完成

当前系统中的 `concluded` 不应被理解为“讨论结束”。它的准确含义应当是：**该事项的 `plan` 阶段已经完成。**

这意味着：

- 围绕该事项已经形成了足够充分的数据流
- 某位同事基于整条数据流，借助 AI 做了整体回顾、收敛和整理
- 事项已经形成了正式拍板结论
- 后续进入 `execution` 的前提已经具备

对于小团队，这个拍板动作不需要依赖强制化角色绑定。任何真正理解上下文并愿意承担判断责任的同事，都可以完成这一步。

### 3. `goal` 是 `plan` 的产物，也是 `execution` 的起点

事项在 `plan` 阶段完成后，应产出一份正式拍板文档。当前，这份文档统一命名为 `goal`。

`goal` 不是泛泛意义上的目标描述，而是事项在完整计划数据流基础上形成的正式拍板文档。它明确该事项接下来要完成什么，并成为进入 `execution` 的直接依据。

因此，`goal` 具有双重地位：

- 它是 `plan` 的正式产物
- 它是 `execution` 的起点文档

事项从 `plan` 到 `execution` 的闭环，可以清楚地表述为：

**事项在 `plan` 阶段持续积累数据流，由某位同事基于 AI 做整体回顾和拍板，形成 `goal`，事项因此进入 `execution`。**

### 4. `execution` 围绕 `goal` 展开

事项进入 `execution` 后，系统就不再停留在计划层，而是围绕 `goal` 展开实际执行。`goal` 会被拆解为多个 task，每个 task 作为 execution 阶段中的执行单元展开推进。

在这一层，逻辑已经足够清晰：

- `matter` 是顶层事项
- `goal` 是事项从 `plan` 进入 `execution` 的正式起点
- `task` 是围绕 `goal` 展开的执行单元

execution 阶段产生的文档，统一属于该事项执行过程中的工作足迹。至于这些文档是否统一使用 `exe` 标识，属于后续实现层的设计问题，不影响当前主逻辑。

## 四、最终判断

Pivot 真正要做的，不是把企业工作的每一个局部动作都工具化，而是把企业工作还原为一条条事项时间线，并让 AI 在这些时间线上持续推动事项向前演进，直到完成完整的工作闭环。

如果这件事成立，Pivot 就不会只是一个讨论系统、任务系统或知识系统，而会成为一个真正围绕企业工作本身运转的系统。
