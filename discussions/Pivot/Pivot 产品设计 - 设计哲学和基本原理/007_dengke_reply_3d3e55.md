我认同把 `matter` 的状态和 MD 文档的类型彻底拆开，这样语义会清楚很多。`matter` 只回答“这件事现在推进到哪里了”，MD 只回答“这份文档承担什么作用”，两者不再互相替代。

但在这个基础上，我倾向于再收紧一步：不要再为 `matter` 额外设计一个独立实体对象。`matter` 在概念上仍然是顶层事项，但在物理落地上，直接由 Git 里的 `index` 文件承载就够了。这里说整个系统的 source of truth 更单一，意思其实很具体：系统里的权威数据最好只在一个地方保存。这样程序运行时，谁是准的、该读哪里、该写哪里，都是清楚的。

如果我们一边在 Git 的文档和 `index` 里保存这套权威数据，一边又在 SQL 里再保存一份 `matter` 主记录，迟早会遇到两边不一致的问题。到那时，系统就会开始背上额外复杂性：同步谁先谁后、冲突时以谁为准、审计时看哪一边、回滚时怎么恢复、异常状态怎么修正。这些问题一旦出现，后面的实现和维护成本都会迅速上升。我觉得在现在这个阶段，没有必要主动把这层复杂性引进来。

以目前的产品阶段，使用 MD 文档和 Git 文件来承载这套数据，已经足以满足需求。如果后面真的发展到完全不同的规模，比如要在成千上万篇 MD 文档上直接运行，那个时候更合理的方向也不是继续维持 Git 和 SQL 双轨并存，而是直接把这套数据整体迁移到数据库里，让数据库成为新的单一主记录。那会是未来某个阶段可能需要做的事，但不是现在。

对应地，我也倾向于不再单独设计 `goal` 文档。原因不是目标不重要，而是它作为独立对象的价值已经不高了。事项从 `plan` 进入 `execution` 之后，执行方向本来就可能持续调整；如果这里再固化一份 `goal` 文档，后面很容易变成语义复杂、维护成本高的中间层。对我来说，更自然的做法是：当事项从 `plan` 进入 `execution` 时，由 `index` 明确标记一篇 `think` 文档作为这次阶段切换的依据文档。它不需要再叫 `goal`，但它天然承担了当前执行基线的作用。

这样设计还有一个好处，就是 AI 更容易理解。AI 不需要再额外学习一套 `goal` 对象，只要把规则讲清楚：`matter` 的状态看 `index`，文档用途看 `think / act / verify / insight`，而 `plan -> execution` 的切换依据也是由 `index` 显式引用的文档来表达。这样语义是直接的，也更贴近真实协作过程。

这里我唯一想补的一点是，不建议把规则写成“最后一个 `think` 文档就是目标文档”，因为进入 `execution` 后也可能继续产生新的 `think`。更稳的定义应该是：在阶段切换发生时，由 `index` 显式指定哪一篇 `think` 文档是当前执行依据；如果后续执行过程中方向调整，再由新的 `think` 文档和 `index` 更新这条执行基线。

如果按这个口径收敛，我理解整体模型可以变得很简单：

- `matter` 是顶层事项，但主记录就是 `index`
- `plan / execution / acceptance / review` 是 `matter` 的状态
- `think / act / verify / insight` 是 MD 的用途类型
- 程序负责维护 `index` 结构和阶段锚点，AI 基于 `index` 和原始文档做理解、判断和推进

这样一来，模型会比“额外建 matter 实体 + 再建 goal 文档”更轻，也能避免过早进入 Git 与 SQL 双主存并存的复杂局面。在当前阶段，先把单一主记录守住，比提前设计一套更重的存储架构更重要。
