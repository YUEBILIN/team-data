设计整体方向我是赞同的，特别是 §13 那一节为未来权限模型做的架构预留，思路很清晰。这里集中提几个**落地前需要明确的契约缺口**，避免实施时方向跑偏。

## 1. §7.1 不只是措辞问题，是解析路径设计本身的问题

现在的描述把"先用外部身份反查 pivot_user"当作**所有**展示路径的统一起点。这不是文字模糊，而是会带来两个具体后果：

**性能层面**：新内容 @-mention 本应直接 `SELECT pivot_user WHERE id = ?` 就能拿到所有展示信息，按 §7.1 的写法却要先经过 `external_binding` 反查这一跳——多一次 join、多一次缓存击穿。线上每一次 post 渲染、每一个 thread 列表、每一个 mention chip 都会白白付这个成本，且**完全没有任何收益**（因为新内容的 ULID 直查更快、更准）。

**架构层面**（更严重）：这种描述会让后端实施同学把所有展示入口都串到 `external_binding` 反查；同时会让前端开发者形成"反正展示层会反查"的心智，新内容就更倾向继续存 `open_id`——因为存什么都能展示出来。结果就是 §13 那一整块"为脱耦做的预留"在引用层全部失效，`pivot_user.id` 沦为内部主键、外部引用永远绕回飞书。等到真要接钉钉/企微的时候，发现历史内容、mention、缓存、索引里全是 open_id，已经没法解耦了。

所以 §7.1 必须重写为**直查降级 resolver**——入参统一是字符串引用，不在解析层做格式判别，处理顺序：

1. 直接 `SELECT pivot_user WHERE id = ref`——命中即返回，**这是新内容的稳态路径**
2. 未命中 → `external_binding WHERE external_id = ref` 反查 → 拿到 pivot_user——**这是历史 open_id 的兼容路径**
3. 仍未命中 → contacts 兜底（仅飞书）→ frontmatter 原值兜底，标 unknown

整条链全程过 LRU 缓存，历史路径的 miss 一次后即缓存结果。这样实施同学一看就知道：**新内容的稳态没有 binding 那一跳**，老内容多一次 SELECT 被缓存兜走。

这条改完，第 2 条的"@-mention 存 ULID"契约才能真正落地——否则解析器入口的形状本身就在反向暗示"还是存 open_id 吧"。

## 2. 新内容 @-mention 的存储契约请明文写出

建议在 §7 或 §11 加一段约定：

> 自迁移上线起，所有新发布内容里的 @-mention、作者引用、以及联系人选择器返回值，统一持久化为 `pivot_user.id`（ULID）。历史内容里的 `open_id` 字符串不做回填，由 §7.1 直查降级 resolver 兼容。

不写明，实施同学就只能各自猜。

## 3. §14 验收加一条断言

> 迁移上线后，**新发布**内容里所有 @-mention 实际持久化的是 ULID，不是 open_id。

可以用 `git grep` 抽查迁移上线后新增的 post，或在 mention 写入路径加单测。

## 4. §9.1 步骤 3 的 6 张外键表是穷举吗？

`session / 草稿 / 阅读状态 / 收藏 / AI 对话 / PAT`——这是当前**所有**依赖用户身份的表清单吗？另外还想确认两点：

- 是否有**非 FK** 但用 `open_id` 当查询 key / 缓存 key / 派生索引字段的数据（比如 read_state 索引、mention_index、未来的全文索引等）也需要一起改？
- 迁移前打算如何确保没有遗漏？是否走"`git grep open_id` + 表结构反查"双重过一遍并把结果列在迁移 runbook 里？

漏一张表就是一类数据孤儿，且单事务回滚的兜底只能保 FK 一致性、保不了非 FK 的字符串引用。

## 5. 飞书原生用户没有 fallback 登录入口

§4 四种入口里，邀请码用户有「邮箱密码」作为常态登录方式，而**飞书原生用户没有任何备份登录路径**——一旦飞书 OAuth 故障、token 撤销、或公司未来迁出飞书，这批用户只能等管理员手工逐个补发邀请。

不一定本期做，但建议在 §12「本期不做」里明示一行，比如：

> 飞书原生用户自助设置邮箱密码作为备份登录方式——本期不做，原因：当前飞书 OAuth 稳定且团队规模可控；未来若 IM 迁移或对外协作扩大时单独立项。

免得日后这个缺位被当 bug 提报。
