## 0. 需求概览

### 0.1 这次要做什么

Pivot 当前账号体系强绑定飞书扫码——任何被允许扫码的飞书用户都直接成为 Pivot 用户，没有审批、没有用户状态管理、管理员资格靠一个写死在代码里的密码识别。这种模式适合早期内部使用，但已经无法支撑产品化阶段的治理诉求。

本次升级把 Pivot 改造成一个**有自己的用户主数据、有审批治理、有可扩展登录方式**的产品级用户管理体系：

- **用户身份不再等同于飞书账号**——飞书、未来的钉钉/企业微信、邀请码等都只是登录来源，归并到 Pivot 自己的用户实体上
- **新用户加入需经管理员审批**——不再是"扫码即加入"
- **用户状态可治理**——管理员能暂停 / 恢复 / 标 deleted 已加入的用户，状态变化不破坏历史数据
- **管理员可治理**——废除硬编码密码，改为可初始化、可互相升降级的角色机制
- **多一条不依赖飞书的登录路径**——邀请码 + 邮箱密码，未来可服务外部协作者或迁移其他 IM 时复用

### 0.2 怎么做（解决方案主线）

按四条主线推进，分别落到对应章节：

1. **身份解耦**：把"Pivot 用户"和"外部 IM 身份"建模为两个独立实体，通过绑定表多对一关联；现有飞书用户一次性迁移到新模型
2. **治理流程**：新陌生人扫码 → 待审批 → 管理员同意（新建 / 合并到已有）/ 拒绝；管理员侧提供后台审批面板与同人匹配建议
3. **状态生命周期**：用户三态（active / suspended / deleted）+ 状态机；所有治理动作走"至少一个在线管理员"的硬护栏；deleted 仅是 DB 标记，数据完全保留，历史展示用灰色样式区分
4. **鉴权重构**：移除硬编码 `X-Admin-Password`，改为基于 `pivot_user.role` 的角色鉴权；首次部署走 `/init` 初始化或迁移脚本指定首位管理员

附加一条横切关注：**为未来权限模型做架构预留**——本期暂不做菜单/数据权限，但 schema 与历史数据要避免堵死扩展路径。

## 1. 术语与命名约定

| 术语 | 含义 |
|---|---|
| `pivot_user` | Pivot 自己的用户主数据；用户身份的权威源 |
| `external_binding` | 某个外部 provider（飞书 / 邀请码 / 未来钉钉 / 企微）下的身份与 `pivot_user` 的多对一绑定 |
| `provider` | 外部身份源代号；本期实际接 `'feishu'` 和 `'invite'`，schema 预留 `'dingtalk'` / `'wecom'` |
| `external_id` | 该 provider 内的稳定标识（飞书=`open_id`、钉钉=`unionid`、邀请码=邮箱地址） |
| `join_application` | 待审批的加入申请；申请期还不是 `pivot_user` |
| `invite` | 管理员发出的一次性加入链接；被邀人接受后创建 `pivot_user` + 邮箱密码 binding |
| `deleted` 状态 | 仅指 `pivot_user.status='deleted'` 这个标记位 |

### 关于"永久删除"措辞

需求原文使用「永久删除」一词。本设计避免使用此词，统一用 **`deleted` 状态** / **标记为 deleted** 表述。
**`deleted` 完全不是物理删除**——所有相关表的数据保留，仅 `pivot_user.status` 改为 `'deleted'`，登录被拒、列表默认隐藏、历史展示加灰色标记。撤销 deleted 即可恢复使用。

## 2. 关键决策一览

仅列结论。详细展开见后文对应章节。

| 项 | 决策 |
|---|---|
| 身份源范围 | 本期落地飞书 + 邀请码兜底；schema 已为钉钉、企业微信等后续 provider 做架构预留 |
| 兜底登录形态 | 邀请码（管理员发链接、被邀人设密码）+ 邮箱密码登录 |
| 期次 | 单一 PR 一次性交付 |
| 管理员密码层 | 完全废除 `X-Admin-Password`，鉴权基于 `pivot_user.role` |
| 菜单/数据权限 | 本期不做；schema 与历史数据已为未来权限模型做架构预留 |
| 审批同人匹配 | 多候选实时算 + 标签 reason（邮箱/姓名/全拼/拼音首字母） |
| `pivot_user` 主键 | 新生成 ULID，与外部身份完全解耦 |
| 旧 `users` 表 | 迁移后 drop |
| `deleted` 状态 | 仅 DB 标记，数据完全保留；可恢复，恢复需强二次确认（输入显示名） |
| 在线管理员护栏 | 服务端硬约束：`(role='admin' AND status='active')` 数 ≥ 1 |
| 操作日志 | 本期不做单独审计表，仅在 `pivot_user` 行内记 `status_changed_at/by` |
| 迁移：初始管理员 | 迁移脚本 `--initial-admin <pinyin>` 必传 |
| 迁移：sessions 处理 | 无感保留（rewrite FK，不清空） |

## 3. 数据模型

### 3.1 新增 4 张表

#### 3.1.1 `pivot_user`：用户主数据

Pivot 自己的用户实体，与外部身份完全解耦。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | 字符串（ULID） | 是 | 主键，业务侧使用 |
| display_name | 字符串 | 是 | 显示名 |
| pinyin | 字符串 | 是 | 拼音，用作 git commit author + 站内排序 |
| email | 字符串 | 否 | 邮箱，唯一 |
| avatar_url | 字符串 | 否 | 头像 |
| github_username | 字符串 | 否 | 沿用现有字段 |
| role | 枚举 | 是 | `admin` / `member`，默认 `member` |
| status | 枚举 | 是 | `active` / `suspended` / `deleted`，默认 `active` |
| status_note | 字符串 | 否 | 管理员可见的状态变更备注，普通用户接口不返回 |
| created_at | 时间戳 | 是 | 创建时间 |
| updated_at | 时间戳 | 是 | 任意字段变更时刷新 |
| last_login_at | 时间戳 | 否 | 每次登录成功刷新 |
| status_changed_at | 时间戳 | 否 | 状态变更时刷新 |
| status_changed_by | 字符串 | 否 | 操作的管理员 user id |

#### 3.1.2 `external_binding`：外部身份绑定

某个 provider（飞书 / 邀请码 / 未来钉钉、企微）下的身份与 `pivot_user` 的多对一绑定。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | 字符串 | 是 | 主键 |
| pivot_user_id | 字符串 | 是 | 关联到 `pivot_user.id` |
| provider | 字符串 | 是 | `feishu` / `invite` / 未来 `dingtalk` / `wecom` |
| external_id | 字符串 | 是 | provider 内稳定标识（飞书 open_id、邮箱地址等） |
| external_union_id | 字符串 | 否 | 部分 provider 有跨应用的 union 标识 |
| raw_profile | JSON 字符串 | 否 | 原始 name / avatar / email / department 等，留作展示与未来匹配 |
| password_hash | 字符串 | 否 | 仅 `provider='invite'` 时使用，bcrypt 哈希 |
| bound_at | 时间戳 | 是 | 绑定时间 |

唯一约束：`(provider, external_id)` 组合唯一。

#### 3.1.3 `join_application`：加入申请

审批前的"还不是用户的人"，审批通过后 promote 为 `pivot_user` + `external_binding`。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | 字符串 | 是 | 主键 |
| provider | 字符串 | 是 | 同 `external_binding.provider` |
| external_id | 字符串 | 是 | 同 `external_binding.external_id` |
| external_union_id | 字符串 | 否 | 同上 |
| raw_profile | JSON 字符串 | 是 | 申请提交时的 name / avatar / email / department |
| suggested_match_user_id | 字符串 | 否 | 系统建议的最高置信同人候选，便于排序 |
| status | 枚举 | 是 | `pending` / `approved` / `rejected`，默认 `pending` |
| applied_at | 时间戳 | 是 | 申请时间 |
| reviewed_at | 时间戳 | 否 | 审批时间 |
| reviewed_by | 字符串 | 否 | 审批的管理员 user id |
| reject_reason | 字符串 | 否 | 拒绝原因，仅管理员可见 |

唯一约束：同一外部身份待审批中只能有一条 pending（即 `(provider, external_id)` 在 status='pending' 子集下唯一）。

#### 3.1.4 `invite`：邀请码

管理员发出的一次性加入链接，被邀人接受后创建 `pivot_user` + 邮箱密码 binding。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | 字符串 | 是 | 主键 |
| token_hash | 字符串 | 是 | 链接 token 的 sha256；唯一；明文 token 不入库 |
| email | 字符串 | 是 | 预填邮箱 |
| display_name | 字符串 | 否 | 可选：管理员预填显示名 |
| created_by | 字符串 | 是 | 发邀的管理员 user id |
| created_at | 时间戳 | 是 | 创建时间 |
| expires_at | 时间戳 | 是 | 默认 +7 天 |
| used_at | 时间戳 | 否 | 接受后落值；非空表示已用 |
| used_by_user_id | 字符串 | 否 | 接受后写入新创建的 `pivot_user.id` |

### 3.2 现有表变更

| 表 | 变更 |
|---|---|
| 旧用户表 `users` | 迁移完成后整体删除 |
| 会话表 `sessions` | 用户外键由飞书 open_id 改挂到 pivot_user，重写映射，不清空，**保持登录状态无感** |
| 草稿、阅读状态、收藏、AI 对话、PAT 等 6 张表 | 用户外键统一改挂 pivot_user |
| 飞书通讯录镜像表 `contacts` | **不动**——仍是名字解析的兜底来源（"被 @ 但还没登录过的飞书同事"） |
| 配置表 `settings` | 不动；代码层移除硬编码管理员密码 |

### 3.3 现存 git 仓库内容不变

git 仓库里 post 的 frontmatter 作者标识、commit author 等历史数据**完全不改动**。展示用户名时由后端通过外部身份反查到当前 pivot_user，使用其当前显示名（动态署名）。

## 5. 加入审批

### 5.1 后台 `/admin/applications` 面板

展示 `join_application.status='pending'` 列表，每行：

| 列 | 来源 |
|---|---|
| 来源 | `provider`（飞书 / 邀请） |
| 头像 + 姓名 | `raw_profile.avatar` / `raw_profile.name` |
| 邮箱 | `raw_profile.email`（可空） |
| 组织 / 部门 | `raw_profile.department`（可空，飞书未必返回） |
| 申请时间 | `applied_at` |
| 建议同人 | 多候选 chip 列表，每个带匹配理由（见 §5.2） |
| 操作 | [同意为新用户] / [合并到已有用户] / [拒绝] |

筛选：状态（pending / rejected / approved）、来源、搜索（姓名/邮箱）。

### 5.2 同人匹配建议

申请创建时存一条 `suggested_match_user_id`（最高置信候选）以便排序，**展示时实时多候选计算**（不再读 `suggested_match_user_id`，前端拿全候选）。

**候选规则（每个候选带 reason 标签）：**

1. **邮箱完全匹配**：`raw_profile.email` 非空、小写比较与某 `pivot_user.email` 严格相等
2. **显示名完全匹配**：`raw_profile.name` 严格等于某 `pivot_user.display_name`
3. **全拼匹配**：`raw_profile.name` 经 pypinyin 转拼音后等于 `pivot_user.pinyin`
4. **拼音首字母匹配**：取首字母后等于 `pivot_user.pinyin` 的首字母序列

只对 `status='active'` 的 pivot_user 计算候选。最多展示 5 个候选；超出按规则优先级 + 置信度截断。

候选展示形态示例：
```
建议同人：
  [李伟] 邮箱完全匹配  ←  最强信号
  [李伟](另一个) 显示名完全匹配
  [李未] 拼音首字母匹配
```

管理员看到候选**仅作 hint**，可以全部忽略，按"同意为新用户"创建独立账号。

### 5.3 三个动作的语义

| 动作 | 系统行为 | 通知申请人 |
|---|---|---|
| **同意为新用户** | 用申请的原始资料新建 `pivot_user`（active / member）+ `external_binding` 绑定该外部身份；申请记录置为已批准 | 飞书卡片"加入申请已通过" |
| **合并到已有用户** | 把当前外部身份作为新的 `external_binding` 挂到管理员选定的现有用户；不修改现有用户任何资料；申请记录置为已批准；前置校验：目标用户在同一 provider 下未绑过身份 | 飞书卡片"你的身份已并入现有账号" |
| **拒绝** | 申请记录置为已拒绝；可选记录管理员填写的拒绝原因，仅管理员后台可见 | 飞书卡片"加入申请未通过"（**不携带原因**） |

合并目标的选定：从前面"建议同人"候选 chip 直接选，或打开搜索 dialog 按姓名 / 邮箱 / 拼音 / 拼音首字母检索现有 active 用户。

### 5.4 数据约束

- 同一外部身份在同一时刻只能存在一条待审批记录
- 已被拒绝的外部身份默认阻断再次申请；管理员可在后台主动"解除阻断"放行重新申请
- 同一 Pivot 用户在同一 provider 下最多绑一个外部身份

## 6. 用户管理

### 6.1 后台 `/admin/users` 列表

| 列 | 说明 |
|---|---|
| 头像 + 显示名 | `pivot_user.display_name` |
| 邮箱 | 可空 |
| 外部身份 | chip 显示绑的所有 provider |
| 角色 | `admin` / `member` |
| 状态 | `active` / `suspended` / `deleted` |
| 最近登录 | `last_login_at` |
| 加入时间 | `created_at` |
| 操作 | 见 §6.2 状态机 |

筛选：状态（默认显示 `active+suspended`，`deleted` 默认隐藏可勾选）、角色、搜索框（姓名/邮箱/拼音子串）。

### 6.2 状态机

```
       ┌─────────┐    暂停    ┌────────────┐
       │ active  │ ─────────→ │ suspended  │
       │         │ ←───────── │            │
       └─────────┘    恢复    └────────────┘
            │                       │
            │ 标记 deleted          │ 标记 deleted
            ▼                       ▼
              ┌─────────────┐
              │   deleted   │
              └─────────────┘
                    │
                    │ 撤销 deleted（强二次确认：输入显示名）
                    ▼
                 active
```

**关键规则：**

- 任何状态变更都弹二次确认，可选填 `status_note`
- 标记 `deleted` 时 **必须显式输入用户显示名**才允许提交（避免误点）
- 撤销 `deleted` 同样要求输入显示名
- 任何会导致 `(role='admin' AND status='active')` 计数 == 0 的操作 → 服务端 422 + 前端按钮置灰提示原因
- 管理员**不能对自己**执行 暂停 / 标 deleted / 自我降级（前端置灰 + 服务端 422）

### 6.3 状态对系统行为的影响

| 场景 | suspended | deleted |
|---|---|---|
| 登录入口（OAuth / PAT / 邮箱密码） | 拒绝 | 拒绝 |
| 已有 session 接下来的请求 | 中间件检测后强制登出 | 同 suspended |
| 在 thread 列表 / posts 中作为作者出现 | 正常显示 | 灰名 + tooltip"该用户已停用"（§7） |
| 别人在新内容中 @-mention 该用户 | 不再发飞书 DM；候选下拉不出现 | 同 suspended |
| 历史 mention / 草稿 / 收藏 / read_state | 全部保留 | 全部保留 |
| 联系人选择器候选 | 不出现 | 不出现 |
| 在 admin 后台列表默认筛选 | 显示 | 隐藏 |

### 6.4 删除后再扫码的处理

deleted 用户的 `external_binding` 不删。该外部身份再次扫码 → 入口 1 命中 binding → `pivot_user.status='deleted'` → 拒绝登录，提示"账号已停用，请联系管理员"。**不会**冒出新一条 `join_application`。

### 6.5 用户生命周期状态总表

| 阶段/状态 | 数据位置 | 自己可做的事 | 其他用户感知 | 管理员可操作 | 登录 | 发帖/回复 | 被 @ 候选 | 历史展示 | 收飞书 DM |
|---|---|---|---|---|---|---|---|---|---|
| **申请中** `pending` | `join_application` | 重复扫码看到"等待审批" | 不可见 | 同意为新用户 / 合并到已有 / 拒绝 | ✗ | ✗ | ✗ | 无 | ✗ |
| **已拒绝** `rejected` | `join_application` | 再次扫码被阻断 | 不可见 | 解除阻断 | ✗ | ✗ | ✗ | 无 | ✗ |
| **正常** `active` | `pivot_user` | 系统全部功能 | 名字头像正常显示 | 暂停 / 标 deleted / 升降级 | ✓ | ✓ | ✓ | 正常名 | ✓ |
| **暂停** `suspended` | `pivot_user` | 无（强制登出，登录被拒） | **完全感知不到状态** | 恢复 / 标 deleted / 改 status_note | ✗ | ✗ | ✗ | 正常名 | ✗ |
| **已停用** `deleted` | `pivot_user` | 无 | 灰名 + tooltip"该用户已停用" | 撤销 deleted | ✗ | ✗ | ✗ | 灰名 | ✗ |

### 6.6 角色 × 状态的笛卡尔积

| 组合 | 说明 |
|---|---|
| `member, active` | 普通成员 |
| `admin, active` | 在线管理员（计入护栏） |
| `member, suspended` | 暂停的普通成员；恢复时角色不变 |
| `admin, suspended` | 暂停的管理员；**护栏**：暂停前必须保证还有其他 active admin |
| `member, deleted` / `admin, deleted` | 已停用——已停用管理员仍占着 admin 角色名义但不计入"在线管理员护栏"；撤销 deleted 后角色保留为 admin |

## 7. 历史内容展示规则

### 7.1 名字解析的统一入口

所有展示用户名字头像的地方，都走一个统一的"名字解析"入口，按下面的回退顺序找到当前应展示的形态：

1. 通过外部身份找到对应的 `pivot_user`，使用其当前的显示名、头像、状态
2. 找不到时（仅限飞书 provider），回退到飞书通讯录镜像 `contacts` 表的姓名头像（"被 @ 但没登录过的飞书同事"）
3. 仍找不到时，回退到 git frontmatter 里的原始值，状态标记为"未知"

返回的状态决定前端渲染样式（见下表）。

### 7.2 渲染规则

| 出现位置 | active | suspended | deleted | unknown |
|---|---|---|---|---|
| post header（作者头像+名） | 名字 + 头像 | 名字 + 头像（**无任何标记**） | **灰名 + tooltip"该用户已停用"** + 灰头像 | frontmatter 原始名 + 通用灰头像 |
| 正文 @-mention 高亮 | 蓝色 chip | 蓝色 chip | 灰色 chip + tooltip | 灰色 chip 显示原始名 |
| thread 列表 / 时间线 "by xxx" | 正常 | 正常 | 灰名 | 原始名 |
| 状态变更记录 "marked accepted by xxx" | 正常 | 正常 | 灰名 | 原始名 |
| @mention 候选下拉 | 出现 | **不出现** | **不出现** | N/A |
| Inbox 通知列表的"被点名人" | 正常 | 正常 | 灰名 | 原始名 |

**关键理由：**

- **suspended 不暴露**——状态变更可能是临时性、敏感性的（休假、调查中），让全员看到不合适
- **deleted 暴露**——读者看到旧 post 应能识别"这人已离开"，避免误以为可联系
- **unknown 兜底**——理论不该发生（迁移保证全部覆盖），但保护渲染不崩

### 7.3 名字以谁为准（动态署名）

显示用的 `display_name` **永远取 `pivot_user` 当前值**，不取 git frontmatter 旧名。

- 用户改了显示名 → 所有历史 post 上他的署名一起更新
- Git commit author（`<pinyin>@pivot.local`）**不会变**，git log 仍是当时落地的拼音

这套和现有的 `users → contacts → 原始值` 回退链是同一思路（见 `pivot-memo.md` §5），只是多了 deleted 这个分支。

## 8. 管理员权限管理

### 8.1 升降级动作

后台用户列表上每一行提供 [设为管理员] / [取消管理员] 操作。设为管理员要求目标当前状态为 active；取消管理员把角色改回普通成员。

护栏（服务端硬约束，UI 仅辅助置灰）：

- 取消管理员、暂停在线管理员、把在线管理员标 deleted——任何会让"在线管理员（角色 admin 且状态 active）"计数降到 0 的操作一律拒绝
- 管理员**不能对自己**执行 取消管理员 / 暂停 / 标 deleted（避免间接绕过护栏；要离任由其他管理员处置）

### 8.2 移除硬编码 `000123`

代码层移除当前所有 `X-Admin-Password` 头校验与 `ADMIN_PASSWORD` 常量。原本受这把锁保护的接口改为根据 `pivot_user.role='admin' AND status='active'` 判断是否放行。涉及现有的 AI 设置、通讯录同步、数据仓库配置等管理员接口，以及本次新增的所有管理员后台接口。

身份不足时统一返回 403 + 错误码（语义从"密码缺失"调整为"身份不足"）。

## 9. 迁移方案

### 9.1 迁移脚本步骤（按顺序，单事务包裹）

| 步骤 | 内容 |
|---|---|
| 1 | 建立 4 张新表 |
| 2 | 遍历现有飞书用户：每位生成新的 Pivot 用户主数据 + 飞书外部身份绑定，并维护 open_id ↔ pivot_user_id 映射 |
| 3 | 把所有依赖用户身份的表（session、草稿、阅读状态、收藏、AI 对话、PAT）的外键从飞书 open_id 改为 pivot_user_id；过程中检测无映射的孤儿数据，发现即整体回滚 |
| 4 | 校验：新 Pivot 用户行数与原飞书用户行数一致；下游表无空外键 |
| 5 | 把 `--initial-admin` 参数指定的拼音匹配为某位 Pivot 用户，并把其角色置为 admin；命中失败则中止迁移 |
| 6 | 删除旧 `users` 表 |

整个流程要么全部成功提交，要么全部回滚，不会留下半成品状态。

### 9.2 命令行参数

| 参数 | 必填 | 说明 |
|---|---|---|
| `--db <path>` | 是 | SQLite 路径 |
| `--initial-admin <pinyin>` | 是 | 指定首位管理员的拼音；脚本会找现有用户中 pinyin 严格相等者，命中失败则中止 |
| `--dry-run` | 否 | 跑前 4 步并输出对账报告（计数、孤儿、初始管理员是否能匹配），最后回滚不写入 |

### 9.3 部署 runbook（高层）

1. 停服务
2. 备份现有 DB 文件
3. 先用 `--dry-run` 跑一次，看对账报告是否全部 OK
4. 跑正式迁移
5. 启动新版本，用 `--initial-admin` 指定的飞书账号扫码登录、验证后台可访问
6. 如出问题：停服 → 用备份覆盖 DB → 回滚代码版本 → 起服

详细命令由 plan 文档或 runbook 给出。

### 9.4 测试策略

- 单测覆盖：含若干用户与各下游表数据的临时 DB，跑完迁移断言行数对齐、外键全部有效、孤儿为空、初始管理员角色正确；`--dry-run` 模式断言数据未变
- 回归：迁移后跑全量 pytest baseline
- 真实数据演练：可在 dev DB 上跑一次 `--dry-run` 预演

### 10.1 新增飞书卡片通知场景

| 触发 | 接收人 | 内容 |
|---|---|---|

### 10.2 不通知的场景（明确）

- 用户被暂停 / 恢复 / 标 deleted：不通知本人，不通知其他人
- 角色变更（升降级）：不通知本人，不通知其他人
- 邀请链接生成 / 失效：不发卡片（管理员自己复制链接）

理由：这些是治理动作，避免给"被处置者"和"全员"造成不必要的信号；需要时管理员自行沟通。

## 12. 本期不做的范围

| 不做项 | 原因 | 何时该做 |
|---|---|---|
| 菜单权限 / 数据权限（角色对 thread / category 的可见与可写） | 当前研发团队场景下不存在跨部门隔离需求；缺乏真实场景下的设计反馈，过早设计极易猜错；未来真正引入"财务、人力等部门加入协作"时再单独立项 | 引入新部门或客户协作时 |

| 完整审计日志（含登录、内容操作、PAT 等） | 本期采用行内 `status_changed_at/by` + `reviewed_at/by` 已能回答"谁在何时把谁怎么了"；完整审计日志是更大的合规专题 | 安全合规要求出现时 |
| 忘记密码自助流程 | 邀请码用户的密码由管理员后台一键重置（低频）；做忘记密码需要邮件服务、邮件模板、防刷、重置 token | 邀请码用户规模超过管理员手动维护能力时 |

| 邀请链接的邮件自动发送 | 管理员复制链接自己发；做发件需要 SMTP 配置 | 邀请使用频度高时 |
| 暂停/删除/升降级的批量操作 | 本期单个用户操作；批量需要更严的二次确认 + 进度反馈 | 实际运维有批量需求时 |

## 13. 为未来权限模型预留的架构空间

| 选择 | 设计依据 | 未来如何扩展 |
|---|---|---|
| `pivot_user.role` 用字符串而非 bool | 当前只有 `'admin' | 'member'`；不用 `is_admin: bool` 是为了未来加 `'finance'`、`'hr'`、`'auditor'` 等角色不需改 schema 主键/约束 | 直接 `ALTER` 不需要；新增 role 名即可 |
| 不在 `pivot_user` 上加 `permissions: JSON` | 权限表达力差异大，早期格式化只会束手束脚 | 真要做时新加 `user_role(user_id, role)`、`permission_grant(...)` 等关系表，与 `pivot_user.role` 共存 |
| `external_binding.raw_profile` 保留 JSON 而非展平字段 | 不同 provider 字段差异大（飞书 `department_ids` / 钉钉 `dept_id_list` / 企微 `department`）；展平到表会绑死 schema | 未来如果按部门做权限，加独立 `org_membership(user_id, dept_id, source_provider)` 表，从 raw_profile 提取/同步，不污染身份层 |
| `external_binding.provider` 是字符串而非 SQL 枚举 | 当前 `'feishu' | 'invite'`，未来加 `'dingtalk'` / `'wecom'` 不需改约束 | OAuth 路由层加分支即可，binding 表不动 |
| `posts` / `threads` 不引入 `visibility` 字段 | 当前所有内容对所有 active 用户可见；引入字段会绑死可见性的表达模型 | 真做隔离时加 `thread_acl(thread_key, ...)` 关联表；现存数据不需回填——查询层默认"无 ACL = 全部可见" |
| `pivot_user` 不内嵌部门/职级 | 这些是 provider 层属性，可能跨 provider 不一致、可能频繁变更 | 用独立 `org_membership` 表表达；用户主数据保持稳定 |
| 唯一 admin 护栏写在服务端 | UI disable 只是装饰，真正不变式 (`admin AND active 数 ≥ 1`) 在 API handler 层判断 | 未来角色多了，类似不变式（"至少 1 个有 X 权限的人"）继续在服务端 |
| `join_application` 与 `pivot_user` 严格分开 | 申请期是"还不是用户的人"——和用户主数据隔离 | 未来需要"邀请待激活"、"自助申请等待"等更多预登记态时，扩展 `join_application` |
