# 新需求-pivot用户管理功能的增强

```md
## 背景

目前 Pivot 的登录和账号创建主要依赖飞书扫码。用户扫码后，系统可以无缝创建 Pivot 账号并完成登录。这对早期内部使用是高效的，但随着 Pivot 进入产品化阶段，仅靠“扫码即加入”的模式已经不够。

未来 Pivot 不应只绑定飞书。来自飞书、钉钉、企业微信等不同组织协作工具的扫码身份，都应该能够被归拢到 Pivot 自己的用户体系中。同时，用户是否能加入当前 Pivot，也不应完全由扫码行为自动决定，而应进入可审批、可管理的用户管理流程。

## 目标

增强 Pivot 的用户管理能力，建立 Pivot 内部自己的用户主数据和管理员治理机制。

核心目标包括：

- 建立 Pivot 内部用户数据结构，不直接把飞书、钉钉、企业微信账号等同于 Pivot 用户。
- 支持来自不同外部身份源的扫码登录请求，并归并到同一个 Pivot 用户。
- 新用户扫码后，需要管理员审批后才能正式加入当前 Pivot。
- 管理员可以管理已加入用户，包括暂停登录、恢复登录、标记为永久删除等操作。
- 移除系统中写死的管理员账号密码，改为初始化和后台可配置机制。
- 支持管理员之间互相调整管理员权限。

## 需求描述

### 1. 建立 Pivot 内部用户体系

Pivot 需要有自己的用户数据结构。

飞书、钉钉、企业微信等外部系统账号，只应作为某个 Pivot 用户的外部身份绑定，而不是直接等同于 Pivot 用户本身。

同一个 Pivot 用户未来可能绑定多个外部身份，例如：

- 一个飞书扫码身份。
- 一个钉钉扫码身份。
- 一个企业微信扫码身份。

系统需要能够把这些外部身份归拢到同一个 Pivot 用户之下，避免因为不同登录来源产生多个割裂账号。

### 2. 新用户加入审批

当来自飞书、钉钉或企业微信的用户扫码登录时，如果该外部身份尚未绑定到当前 Pivot 的有效用户，系统不应直接允许其加入。

扫码后应进入申请加入流程。

管理员后台需要展示待审批用户，包括：

- 外部身份来源，例如飞书、钉钉、企业微信。
- 外部身份中的姓名、头像、组织、部门等可用信息。
- 申请加入时间。
- 该用户是否可能和已有 Pivot 用户匹配。
- 管理员可执行的操作：同意或拒绝。

管理员同意后，该用户正式成为当前 Pivot 用户，并可以登录系统。

管理员拒绝后，该用户不能进入当前 Pivot，但系统应保留必要的申请记录，便于后续处理。

### 3. 已加入用户管理

对已经加入 Pivot 的用户，管理员需要有基础管理能力。

至少包括：

- 查看用户列表。
- 查看用户状态。
- 暂停用户。
- 恢复用户。
- 标记用户为永久删除。

暂停用户后，该用户暂时无法登录系统，但其历史发布内容和时间线记录不受影响。

永久删除用户也不应真正删除数据。这里的“永久删除”只是将该用户账户状态标记为永久删除，使其不能再登录或继续使用当前 Pivot。该用户的历史身份、历史行为、已发布文档和时间线记录都继续保留。

也就是说，用户管理操作只能改变账户当前状态，不能抹除历史数据。

### 4. 管理员初始化

当前系统中写死的管理员账号密码需要移除，改为系统初始化流程。

系统首次启动、尚不存在管理员时，进入初始化界面。该界面需要明确提示：

> 如果你要初始化这个系统，你将成为当前 Pivot 的管理员。
>
> 请确认你有权限执行初始化操作后再继续。否则请联系系统管理员或你的上级。

用户确认后，完成登录或身份绑定，并成为初始管理员。

目前不需要设计复杂的防抢占机制。如果管理员发现系统被错误的人抢先初始化，可以通过重新安装系统来解决。这个简单机制可以满足当前阶段需求。

### 5. 管理员权限调整

管理员之间可以互相调整权限。

一个管理员可以：

- 将普通用户提升为管理员。
- 将其他管理员移除管理员权限，使其变为普通用户。
- 管理用户加入审批。
- 管理用户状态。

需要避免系统进入“没有任何管理员”的状态。因此在移除管理员权限时，系统应保证至少保留一个有效管理员。

## 暂不包含

本需求暂不提供完整审计日志功能。

管理员审批、暂停、恢复、永久删除、权限调整等操作，未来如果需要审计能力，可以作为统一的审计日志体系再单独设计。

## 期望结果

Pivot 从“依赖飞书扫码自动创建账号”的早期模式，升级为真正的产品化用户管理体系。

最终效果是：

- Pivot 有自己的用户主数据。
- 外部扫码身份只是登录来源，不是用户体系本身。
- 新用户加入当前 Pivot 需要管理员审批。
- 已加入用户可以被暂停、恢复或标记为永久删除。
- 被永久删除的用户数据不会被物理删除，只是账户状态变为永久删除。
- 管理员账号不再写死在系统中。
- 管理员权限可以在后台配置和调整。
- 历史发布记录不会因为用户状态变化而被破坏。

## 交给实现同学思考的问题

- Pivot 用户、外部身份绑定、加入申请、用户状态、管理员权限应如何拆分数据表。
- 飞书、钉钉、企业微信的外部身份字段不完全一致时，应该如何设计统一身份映射。
- 新扫码用户如何判断是否可能和已有 Pivot 用户属于同一人。
- 用户被标记为永久删除后，历史文档中的作者信息应该如何展示。
```
