# 新需求-作者在pivot中失效自己的文档

```md
## 背景

目前可以把 Pivot 理解为一个只读的协作数据库：每个 matter 时间线上发布过的文档，一旦写入就不可修改。这是为了真实记录围绕一个 matter 发生过的所有思考、行动、判断和结论。

这个原则很重要。未来如果 Pivot 要进一步和不可逆、可验证的记录机制结合，例如区块链式的时间线记录，也应当保持“原始发布记录不可被篡改”的基本设计。

但现实中仍然会出现误操作或异常发布。例如用户误发了内容，发布的信息严重失真，或者文档中包含不应继续展示的敏感信息。在这种情况下，系统需要允许作者对已发布文档进行“失效标记”，但不能直接修改或删除原始文档。

## 目标

在保持 Pivot 时间线不可篡改的前提下，增加“标记文档失效”的能力。

这个能力不是删除文档，也不是编辑原文，而是在 index 和时间线上明确记录：某个作者在某个时间点，基于某个理由，将自己发布过的某篇文档标记为失效。

## 需求描述

用户可以将自己发布过的某篇文档标记为失效。

标记失效时，用户必须选择以下的失效理由之一：

- 误操作发布。
- 输入内容严重失真或错误。
- 包含不应公开展示的敏感信息。

只有文档作者本人可以将自己的文档标记为失效。其他用户不能对非本人发布的文档执行失效操作。

文档被标记为失效后，不应继续出现在普通 UI 展示界面中。但原始文档文件仍然保留在内部数据仓库中，不进行物理删除。

## 数据记录要求

一次“标记失效”操作需要同时更新 index 中的两个位置。

第一，时间线上新增一条记录，用来表示失效事件本身。

这条记录需要表达：

- 谁执行了失效操作。
- 在什么时间执行。
- 哪一篇文档被标记为失效。
- 失效理由是什么。

第二，原有文档在 index 中的文件属性需要被更新。

每个文件记录中应增加失效相关属性，例如：

- 是否已失效。
- 失效时间。
- 失效原因。
- 执行失效操作的用户。

也就是说，失效操作不是覆盖原始文档，而是在时间线上新增一个可审计事件，同时在原文件索引属性上标记当前状态。

## UI 行为

被标记为失效的文档，默认不出现在普通用户界面中。

但系统内部仍然保留完整数据。后续如果需要审计、回溯、AI 分析或管理员检查，仍然可以找到原始文档、失效事件和失效理由。

## AI 处理要求

AI 沿着 matter 时间线搜索和处理文档时，不能简单地完全跳过失效文档。

更合理的处理方式是：

AI 需要知道该文档已经被标记为失效，并阅读原文内容、失效时间和失效理由，再进行综合判断。

例如，某篇文档虽然已经失效，但它可能仍然说明当时发生过一次误判、一次错误输入、一次被撤回的想法，或者一次敏感信息泄露风险。AI 在分析 matter 的发展过程时，应当把这些信息作为上下文的一部分，而不是当作从未发生过。

## 期望结果

Pivot 能够在不破坏时间线可信度的前提下，处理误发、错误内容和敏感内容等现实问题。

最终效果是：

- 原始发布记录仍然不可修改、不可抹除。
- 用户可以对自己发布的错误文档进行失效标记。
- 普通 UI 不再展示已失效文档，避免继续误导团队。
- 内部数据仓库保留完整记录，支持审计和 AI 分析。
- AI 在处理时间线时能够理解“这篇文档曾经存在，但后来被作者声明失效”这一事实。

## 其他一些想法

- 失效事件在 timeline 中不应使用独立的文件类型，而是作为现有 timeline event 的一条记录。
- 普通用户界面无法看到失效文档，但未来的内部审计和管理视图可以看到。
- 已失效文档允许再次恢复展示；如果允许，恢复行为也应当作为新的时间线事件记录。
```
