# 反馈：失效的边界与可见性，建议两处对齐

读完 001 与 002，结合团队对「帖子权限」单独立项的预期，002 有两处需要在评审会上校正。

## 一、`sensitive` 不应作为失效理由

`sensitive`（敏感信息）本质是访问控制问题，应由即将单独立项的「帖子级权限」承接，不属于失效能力的范围。

把它塞进失效会有逻辑漏洞：

- 失效只是「在 timeline 上加标签」，不做物理隐藏
- 一旦把 sensitive 当作隐藏工具，要么用户的隐藏诉求落空（标了等于没标），要么倒逼我们在权限体系出来前先造一套半成品权限

**建议 v1 失效理由收敛为：`misposted` / `inaccurate` / `restored` 三值。** 001 §需求描述列出的三种是需求初稿，`sensitive` 这条建议直接剔出，归入未来的「帖子级权限」项目。

遗留问题：在帖子权限上线前，如果用户误粘贴了密钥/PII 怎么办？建议走 admin 介入的紧急 redact 路径，作为独立紧急通道，不要混进 v1 失效流程。

## 二、可见性应简化为单一视图

002 §四提出的「普通用户隐藏 / 作者带标识 / Admin 审计视图」三态规则，与团队对失效的真实理解不一致。

**团队真实诉求**：失效是 timeline 上的一种元信息标注，对该 matter 有访问权限的所有用户和 AI 都可见，只是 UI 上明确标记"已失效（理由）"。

理由：

- 失效是「声明式撤回」，不是「删除式撤回」。团队需要看到「曾有过这个判断、后来被作者撤回」——这是 matter 演进过程的一部分。
- AI 在出方案、做评估时必须知道哪些方向已经被作者主动否掉。否则 AI 会反复给出已被否掉的建议，反而降低协作效率。
- 谁有权访问这个 matter 或这条帖子，由「帖子级权限」单独控制，失效能力不替它承担。

**建议 §四 改为：**

| 角色 | 失效文档可见性 |
|---|---|
| 对 matter 有访问权限的所有用户 | 可见，UI 标记"已失效（理由）"，原文可展开 |
| AI / 外部 agent | 可见，沿 timeline 读取原文 + 失效元数据，作为"已被撤回的判断"上下文 |
| 作者本人 | 同上，附带"恢复"入口 |

`is_visible_to` 因此可以退化——失效不再参与可见性判断，只参与渲染层标记。可见性继续由「matter 访问权限」单独控制。

注：001 §UI 行为里有「默认不出现在普通 UI 展示界面」的表述，002 据此走了「完全隐藏」路线。本反馈是对 001 这条措辞的进一步明确——「不出现」应理解为「不以正常发布形态出现」（即必须带失效标记），而不是「列表里直接消失」。

## 三、连带影响

- §三 校验规则不需要因可见性收紧而新增规则
- §5.3 引用阻断仍然成立——已撤回的判断不应被新文档 quote / refer
- §5.4 unread 计数减 1 仍然成立——已撤回内容不应作为"待读"催促用户
- §八 `is_visible_to` 与三态过滤逻辑可以大幅简化
