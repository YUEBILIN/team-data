# 需求：员工评价体系需求与设计

## 1. 背景

Pivot 的 Matter 时间线里已经沉淀了很多协作证据：

| 来源        | 说明                                               |
| ----------- | -------------------------------------------------- |
| `act`     | 谁负责了什么行动项，是否推进到可验收               |
| `verify`  | 谁验收了行动项，验收结果和评价是什么               |
| 新增评论    | 协作过程中对人、事、方案的反馈；可以不选择被提及人 |
| `result`  | Matter 最终结果、是否收口、是否达成目标            |
| `owner`   | 谁承担主责，谁推进事项                             |
| `insight` | 事后复盘和长期沉淀                                 |

这个需求不是让 AI 自动做绩效裁决，而是让 Matter 进入 `result` 后，管理员能看到本 Matter 中相关人员的评分、理由和证据。

## 2. 产品目标

| 目标                      | 说明                                                |
| ------------------------- | --------------------------------------------------- |
| Result 后生成 Matter 评分 | Matter 进入 `result` 后，生成本 Matter 的人员评分 |
| 管理员可查看              | 管理员能看到这个 Matter 中每个人得了多少分          |
| 可配置可见范围            | 管理员可配置评分结果指定哪些人可见                  |
| 说明为什么                | 每个人的分数必须说明原因，并引用证据                |
| 证据可追溯                | 每条评分理由都能回到新增评论、verify、act 或 result |
| 证据充分才评分            | 证据不足时不强行给高置信度评分                      |

## 3. 不做什么

| 不做项           | 原因                                |
| ---------------- | ----------------------------------- |
| 不做自动绩效裁决 | AI 结果不能直接决定绩效、奖金、晋升 |
| 不做黑盒总分     | 不能只给一个分数，必须展示证据来源  |
| 不评价私人态度   | 只评价工作行为、协作过程和交付结果  |

## 4. 核心机制

证据链在 Matter 进入 `result` 后统一生成。平时新增评论、verify、act 变化时不单独生成证据。

完整流程：

```text
Matter 进入 result
  ↓
系统触发 AI 评分任务，并提供 Matter 完整时间线
  ↓
AI 生成证据链、评分和理由
  ↓
进入 Matter 人员评分
```

核心原则：

| 原则            | 说明                                                       |
| --------------- | ---------------------------------------------------------- |
| 证据链必须生成  | 不允许 AI 只给分不给证据                                   |
| Result 后才评分 | 事项没有结果前，不生成评分                                 |
| 平时动作不评分  | 新增评论、verify、act 变化时只保留原始记录，不生成评分     |
| AI 不直接拍结论 | AI 必须输出证据、理由、置信度                              |
| 证据可追溯      | 每条证据必须绑定新增评论 / verify / act / result 的原始 ID |
| 证据不足不强判  | 证据不足、对象不明确、冲突严重时，降低置信度或不评分       |

## 5. AI 评分规则

系统不单独做证据规则引擎。Matter 进入 `result` 后，系统把完整时间线提供给 AI，由 AI 统一完成证据识别、归因、评分和理由生成。

下面规则用于约束 AI 的判断和输出。

### 5.1 证据判断规则

| 原文类型                             | 是否入链           | 处理方式                                           |
| ------------------------------------ | ------------------ | -------------------------------------------------- |
| 明确指向某个人，并描述工作行为或结果 | 入链               | 生成正向或负向证据                                 |
| 有事实、有结果、有影响               | 入链               | 作为较强证据                                       |
| 只有“不错”“很差”，但没有上下文   | 弱入链或不入链     | 降低置信度                                         |
| 只有情绪化表达，没有事实             | 不入链             | 不作为评分依据                                     |
| 同一句里有褒有贬                     | 入链               | 拆成多条正负证据                                   |
| 评价对象不明确                       | 不入链或等待上下文 | 不强行归因                                         |
| 新增评论没有选择被提及人             | 可以入链           | 优先参考评论所在帖子的作者；仍需文本语义能支撑归因 |
| 同一人重复评价同一对象和同一事实     | 合并入链           | 不重复累计权重                                     |
| 与 Matter 无关                       | 不入链             | 忽略                                               |

示例：

| 原文                                               | 处理                                           |
| -------------------------------------------------- | ---------------------------------------------- |
| “张三这次交付很快，客户当天就验收了”             | 入链，正向，交付质量 / 责任闭环                |
| “张三前期没确认字段，导致接口返工”               | 入链，负向，判断质量 / 交付质量                |
| “张三真离谱”                                     | 不入链，只有情绪                               |
| “方案不错”                                       | 如果能从上下文确定 owner，可弱入链；否则不入链 |
| 在张三发布的 act 下评论“这个交付很快”            | 可归因给该 act 作者张三，但置信度不高于 medium |
| “前面考虑粗了，但后面补救很快”                   | 拆成负向证据和正向证据                         |
| 同一个人连续说“张三交付很快”“张三这次确实很快” | 合并为一条同类正向证据，不重复加权             |

### 5.2 评分维度

评分使用 5 个维度，避免过度复杂。

| 维度     | 看什么                             | 正向信号                                   | 负向信号                   |
| -------- | ---------------------------------- | ------------------------------------------ | -------------------------- |
| 交付质量 | 负责事项是否通过验收、结果是否达成 | passed、客户认可、返工少                   | failed、返工、验收意见严重 |
| 责任闭环 | 是否把事情推进到可验收、可收口     | 主动推进、及时补救、完成闭环               | 长期无进展、缺少收口       |
| 协作贡献 | 是否帮助他人推进事项               | 有价值新增评论、有效响应协作请求、协助验收 | 被多次提醒仍无响应         |
| 判断质量 | 判断、方案、验收是否准确           | 提前识别风险、判断被结果验证               | 判断与后续事实明显相反     |
| 过程规范 | 是否留下必要过程证据               | act / verify / result 记录清楚             | 线下完成但无记录、补录混乱 |

## 6. Result 后生成什么

Matter 进入 `result` 后，系统触发 AI 评分任务，并向 AI 提供 Matter 完整时间线。AI 负责从时间线中生成证据链、评分和理由，系统负责保存结果，管理员在后台评分管理页面查看和配置评分结果。页面示例：

| Matter 名称      | 人员 | Matter 内评分 | 为什么                                                      | 主要证据                         | 时间             |
| ---------------- | ---- | ------------: | ----------------------------------------------------------- | -------------------------------- | ---------------- |
| 客户验收流程优化 | 张三 |           4.2 | 负责的行动项按期完成并通过验收，result 中确认客户已接受交付 | 可点击：verify 通过、result 完成 | 2026-04-29 18:30 |
| 客户验收流程优化 | 李四 |           3.6 | 多次参与方案讨论并补充风险，但没有直接承担交付 owner        | 可点击：新增评论协作记录         | 2026-04-29 18:30 |
| 客户验收流程优化 | 王五 |           2.5 | 前期字段确认不充分导致返工，后续有补救                      | 可点击：负向评论、补救记录       | 2026-04-29 18:30 |

证据弹窗：

| 内容         | 说明                              |
| ------------ | --------------------------------- |
| 原始帖子标题 | 展示证据来自哪个帖子              |
| 原始帖子类型 | 新增评论 / verify / act / result/ |
| 原始作者     | 展示该帖子作者                    |
| 发布时间     | 展示原始发布时间                  |
| 原始内容     | 展示对应帖子的原文内容            |

评分规则：

| 分值 | 含义                         |
| ---- | ---------------------------- |
| 5    | 明确优秀，有多条强证据支撑   |
| 4    | 表现良好，有明确正向证据     |
| 3    | 正常完成，证据有限或表现中性 |
| 2    | 有明显问题或缺口             |
| 1    | 严重影响事项推进或结果       |

## 7. 管理员配置

管理员后台需要提供评分配置页面，用于控制评分结果的可见范围。

| 配置项       | 说明                           |
| ------------ | ------------------------------ |
| 评分功能开关 | 控制 result 后是否自动生成评分 |

## 8. 开放问题

| 问题             | 说明                                                                             |
| ---------------- | -------------------------------------------------------------------------------- |
| 是否需要人工修正 | 如果 AI 对证据理解错误、归因错误或分数明显异常，是否允许管理员人工修正分数或证据 |
| 是否推送 IM 评分 | 是否在评分生成后通过 IM 推送给指定用户或角色                                    |
