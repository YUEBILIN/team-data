# 需求：Pivot 数据隐私功能需求与设计

## 1. 背景

当前 Pivot 的 Matter 默认对所有登录用户可见，并且新建、更新、状态变化会通过现有 IM 群通知扩散。进入真实业务协作后，需要支持“部分用户/角色可见，其他人完全不可见”的场景。

本需求不建设用户、角色、权限管理后台。用户、角色、权限配置由“新需求-pivot用户管理功能的增强”提供；本需求只消费那边的权限系统，并在 Pivot 的数据出口执行权限判断。

需要保护的数据出口：

| 出口               | 要求                                         |
| ------------------ | -------------------------------------------- |
| Matter 列表 / 详情 | 不可见用户看不到 matter，也不能通过 URL 访问 |
| 搜索 / AI / MCP    | 不可见用户搜不到；直接读取时返回无权限 |
| SSE                | 不可见用户收不到该 matter 的事件             |
| IM 通知            | 非完全公开场景不发群，只私信 owner 和 @ 用户 |
| @ 选择器           | 只出现具备 matter 读权限的人                 |

## 2. 最终目标

最终实现 Category + Matter 双层数据隐私控制。

| 层级     | 定位                         | 可见类型                 | 授权粒度            | 规则                                          |
| -------- | ---------------------------- | ------------------------ | ------------------- | --------------------------------------------- |
| Category | 业务域 / 项目域 / 客户域边界 | 公开 / 指定角色可见      | 粗粒度，只按角色    | 不做个人例外                                  |
| Matter   | 具体事项边界                 | 公开 / 指定角色+用户可见 | 细粒度，角色 + 用户 | 只能在 Category 范围内收紧，不能越过 Category |

典型场景：

| 场景                       | Category     | Matter            | 谁能看到 Matter                         | 通知                                   |
| -------------------------- | ------------ | ----------------- | --------------------------------------- | -------------------------------------- |
| 全员公开协作               | 公开         | 公开              | 全部 active 用户                        | 保持现有群通知和私信逻辑               |
| 公开分类里的敏感事项       | 公开         | 指定角色+用户可见 | Matter 指定角色或用户                   | 不发任何 IM 群；只私信 owner 和 @ 用户 |
| 指定角色分类里的普通事项   | 指定角色可见 | 公开              | Category 授权角色                       | 不发任何 IM 群；只私信 owner 和 @ 用户 |
| 指定角色分类里的更敏感事项 | 指定角色可见 | 指定角色+用户可见 | Category 范围内的 Matter 指定角色或用户 | 不发任何 IM 群；只私信 owner 和 @ 用户 |

横向规则：

| 规则          | 说明                                                                            |
| ------------- | ------------------------------------------------------------------------------- |
| 不做 deny     | 只做公开、指定角色、指定用户                                                    |
| Matter 不越界 | Matter 可见范围不能超过所属 Category 可见范围                                   |
| 多角色并集    | 一个用户可以属于多个角色；命中任一授权角色即可获得对应 Category / Matter 权限   |
| 多授权并集    | 一个 Category / Matter 可以授权多个角色；一个 Matter 可以额外授权多个用户       |
| 角色 + 用户   | Matter 可见范围 = 授权角色成员 ∪ 授权用户，但仍必须在 Category 可见范围内       |
| 当前权限实时生效 | 权限始终按当前角色计算；用户从角色移除后，后续列表、详情、搜索、AI、MCP、SSE 都不能再访问 |
| 新建 Category | 新建 Matter 时如果同时创建 Category，必须先确定 Category 是公开还是指定角色可见 |
| @ 候选过滤    | 不可见用户不出现在 @ 列表和搜索结果中                                           |
| 后端兜底      | 伪造请求或草稿恢复出不可见 @ 用户时，发布必须失败                               |
| 私信校验      | owner 和 @ 用户发私信前也必须通过 `matter.read`                               |

## 3. 实现要点

### 3.1 后端

| 模块                      | 要做什么                                                                                                                                                                               |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 权限服务                  | 接入用户管理权限系统，提供 `can_read_category`、`can_create_matter_in_category`、`can_read_matter`、`can_write_matter`、`filter_visible_matters`、`list_mentionable_users` |
| `GET /api/matters`      | 过滤不可读 matter                                                                                                                                                                      |
| `GET /api/matters/{id}` | 不可读返回无权限                                                                                                                                                                      |
| `POST /api/matters`     | 支持公开 / 指定角色或用户可见，并把授权范围提交给用户管理权限系统                                                                                                                      |
| 新建 Category             | 如果新建 Matter 时同时创建 Category，必须同步设置 Category 为公开或指定角色可见                                                                                                        |
| 追加文件 / 评论 / result  | 需要 `can_write_matter`                                                                                                                                                              |
| 已读 / 收藏 / AI 会话     | 不可读返回无权限                                                                                                                                                                      |
| 联系人搜索 / @ 选择       | 只返回当前 matter 可读用户                                                                                                                                                             |
| SSE                       | live 和 replay 都按 `matter.read` 过滤                                                                                                                                               |
| IM 通知                   | 完全公开保持现有；非完全公开不发群，只私信 owner 和 @ 用户                                                                                                                             |

### 3.2 AI / MCP

MCP 走 REST API，REST 加权限后自然继承。

Web AI tools 会直接读 index 和 Markdown，必须单独加权限：

| 工具                  | 权限要求                                       |
| --------------------- | ---------------------------------------------- |
| `list_matters`      | 只列可读 matter                                |
| `search_indexes`    | 只搜索可读 matter                              |
| `read_matter_index` | 先校验 `can_read_matter`                     |
| `read_post`         | 通过路径反查 matter 后校验 `can_read_matter` |

### 3.3 前端

| 页面/组件     | 要做什么                                                              |
| ------------- | --------------------------------------------------------------------- |
| 新建 Matter   | 增加“公开 / 指定角色+指定人员可见”                                  |
| 新建 Category | 在新建 Matter 过程中创建 Category 时，要求选择“公开 / 指定角色可见” |
| 可见人员选择  | 创建者和 owner 默认加入                                               |
| @ 选择器      | 只从可见人员中选                                                      |
| Matter 列表   | 指定范围可见 Matter 显示锁/指定范围标识                               |
| Matter 详情   | 无写权限时隐藏追加和评论入口                                          |

## 4. 验收标准

核心验收只看一条主线：

> A 创建一个只给 B 可见的 Matter 后，C 在列表、搜索、SSE、IM 通知中看不到；直接访问 URL 或通过 AI/MCP 读取时返回无权限。

具体验收：

| 场景                                | 预期                                  |
| ----------------------------------- | ------------------------------------- |
| A 创建公开 Matter                   | B/C 都能看到；IM 保持现有逻辑         |
| A 创建指定用户可见 Matter，只授权 B | B 能看到，C 看不到                    |
| C 直接访问指定用户可见 Matter URL   | 返回无权限                            |
| C 搜索标题                          | 无结果                                |
| C 追加文件或评论                    | 失败                                  |
| C 使用 Web AI / MCP 读取            | 读不到，返回无权限                    |
| C 在线订阅 SSE                      | 收不到该 matter 事件                  |
| A 在指定用户可见 Matter 中 @ C      | C 不出现在 @ 候选；伪造请求时后端阻断 |
| 指定范围 Matter 通知                | 不发 IM 群；只私信 owner 和 @ 用户    |

## 5. 开放问题

### 5.1 超级管理员是否可看全部

是否需要一个“超级管理员可查看所有 Category / Matter”的角色，需要产品和安全口径先确认。

两种方案：

| 方案           | 说明                                                 | 风险                                 |
| -------------- | ---------------------------------------------------- | ------------------------------------ |
| 无全局可读超管 | 管理员可管理用户、角色、权限，但不能默认读取所有内容 | 更符合隐私预期，但排障时不方便       |
| 有全局可读超管 | 某些超级管理员可读取所有 Category / Matter           | 隐私承诺变弱，必须配套审计和显式提示 |

建议默认采用“无全局可读超管”。如果未来必须支持，需要由用户管理权限系统单独提供 break-glass 能力，并记录审计日志，包括谁在何时因为什么原因查看了什么 matter。

### Category 专属群

当前没有 Category 专属群，也没有群成员和 Category 权限范围的一致性校验。因此本期不支持 Category 群通知。

如果未来要支持，需要单独设计：

| 问题                    | 说明                                            |
| ----------------------- | ----------------------------------------------- |
| 群如何绑定 Category     | 一个 Category 一个群，还是多个群                |
| 群成员如何校验          | 确保群成员不超过 Category 可见范围              |
| 成员变化如何同步        | 群新增成员后是否自动影响权限                    |
| Matter 更小范围时怎么办 | 即使 Category 群合法，Matter 也可能只给少数人看 |

在这些问题解决前，非完全公开场景一律不发 IM 群。
