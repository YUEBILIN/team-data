# OPC 数字员工服务台 · PRD

> **版本**：v1 · 2026-04-25
> **关联**：竞品调研 `opc-竞品调研报告.md` · 对外稿 opc.enclaws.com · 技术方案 Pivot thread3 · 商务决策 thread1

---

## 一、定位

面向一人公司老板的**运营大脑**。登录 → 秘书对话 → 自动配一套数字员工班底 + 主营业务工作流 → Cron 跑起来 → 老板只审关键节点 → 业务沉淀成企业档案。**一次告诉秘书，全班底都懂**。

---

## 二、产品设计

**价值链闭环**

| 方       | 痛点           | 价值                            |
| ------- | ------------ | ----------------------------- |
| OPC 老板  | 散装工具 + 孤立找活  | 班底跑闭环 + 接入订单池 → 省 6h/天 + 稳定单量 |
| 园区（政府）  | 企业自报不可信、补贴粗放 | 真实交易+信用数据 → 精准补贴              |
| 企业（发包方） | 挑外包难、交付难管    | 按交易信用分级推荐 + 履约沉淀 → 采购降本       |
| 质数（技术方） | —            | 卖"运营中枢+共享记忆壁垒"，不是卖工具          |

**场景链闭环**：Phase 1 深做内容工作室（1 个），其他 5-7 行业共用同一 6 员工框架（业务语义映射 + Skill 差异化）。

**业务链闭环**：入驻（虚拟/待注册/真实三态）→ 班底（秘书对话配菜）→ 决策（业务节点走预览卡；秘书建议走通知+三选一+两段拍板）→ 数据（共享记忆层三机制）→ 生态（资源中心 + 交易信用 + 分层认证 + 园区看板）

**四方边界**：质数 / 园区 / 企业 / OPC。数科院及其他合作/咨询/渠道方**不进产品结构**。

## 三、产品架构

### 3.1 四方入口

| 方       | 菜单                                    |
| ------- | ------------------------------------- |
| **OPC** | 我的秘书 / 今日排班 / 我的团队 / 企业档案 / 资源中心 / 账户 |
| **园区**  | 入驻管理 / 生态看板 / 政策匹配 / 审计               |
| **企业**  | 需求发布 / OPC 筛选 / 订单验收 / 结算             |
| **质数**  | 后台（用户不直接用）                            |

三方前端，同一 EC 底座（共享 `tenant.*` RPC 和 workspace 数据层）。

### 3.2 秘书 agent

常驻对话入口。三段职责：入驻期（商业画布对话 → 自动配菜 + 解释为什么）/ 运营期（日常问答 + 主动建议）/ 演进期（业务转型时建议重配）。**事件触发**不定时：用户输入、数据异常、套餐触顶、月度复盘。**两段拍板防失控**：秘书提建议 → 老板同意 → 秘书生成执行清单 → 老板 review → 才真执行。秘书永远不直接动班底。

### 3.3 共享记忆层

**EC 底层已实现**（`src/memory/`：SQLite 向量 + 多供应商 embedding + FTS + MMR + 时间衰减；3 个 agent memory tool 分 user/tenant/通用）。**本期补产品层**（三机制）：

- **存储**：`company-profile/` 档案（长期，秘书沉淀）+ `operations/` 员工接力（张菠 thread3 §1.8）+ `_index/`
- **交互（核心差异化）**：cc 风格——**写前 preview + 老板拍同意才入库 + 档案视图可直接编辑 MD + 员工产出透明溯源 + 每条可否决**
- **迭代**：秘书归纳 → 老板拍板 → 入库 → 修改历史可回滚

### 3.4 数字员工班底

内容工作室 6 员工（张菠 thread3 §1.4）：**选题 / 创作 / 分发 / 社区 / 商务 / 财务**。跨行业复用同框架，按行业做语义映射（内容"选题"≈电商"选品"），每行业加 2-3 个专属 Skill。协作靠 Cron + workspace 文件接力（frontmatter `status` 传递），不用事件总线。发布策略分档：公众号全自动、小红书等半自动（生成发布包老板粘贴），不赌逆向封号。

### 3.5 资源中心（商机+交易信用+分层认证）

**企业侧**：需求发布 → 订单分级（A 5-20万 / B 1-5万 / C ≤5千，借青羊区 PPT）→ OPC 筛选 → 验收 → 结算。
**OPC 侧**：商机池 → 强机会 → 已报价 → 已成交（借竞品状态机）。
**商机来源**：①政府/事业单位公开招投标抓取（竞品验证路径）+ ②青羊区等政府推送订单 + ③企业自发发单 + ④OPC 互包（低优）。
**两种信用**：分层认证（入驻资质 T1/T2/T3，门槛）≠ 交易信用（动态履约表现，权重）。佣金 6-10%。

### 3.6 园区运营看板

四个维度（Daisy thread2 v0.2 §五）：单 OPC 业务视图 / 就业转化量化 / 产业热力图 / 合规审计。**用途是补贴决策依据**，不作业务介入（园区不动 OPC 班底）。

## 四、数据层

文件系统协作（MD+frontmatter+append-only），租户沙箱隔离。

```
workspace/
  company-profile/        秘书沉淀的企业档案（核心差异化）
  _config/
  inspiration/ topics/ drafts/ published/ publish_packs/ analytics/ assets/
  messages/{inbox/raw, inbox, replies, leads, spam}/
  deals/{inquiries, negotiating, signed, completed, daily}/
  income/{raw, daily, monthly}/
  schedule/ _index/
  agent-suggestions/{pending, accepted, rejected, snoozed}/
```

后端 RPC：`workspace.*` / `tenant.companyTemplate.*` / `tenant.opcEmployee.*` / `notification.dispatch`。Agent 用一次性 JWT。

## 六、分期（3-4 天 MVP，按反馈迭代）

**目标**：V1 在 3-4 天内上线，Daisy 种子可用。砍到最小闭环，其他按反馈迭代。

**V1 · 3-4 天 MVP · 内容创作官单员工闭环**

| Day | 交付 |
|---|---|
| D1 | 选内容工作室模板 + workspace 初始化 + 内容创作官 agent 启用（复用 EC `tenant.agents.create` + prompt 模板） |
| D2 | 1 条端到端工作流：老板丢主题 → LLM 生稿 → 老板审 → 公众号 API 自动发（复用 EC 公众号 skill） |
| D3 | 极简 UI：单屏含"我的班底 + 主题输入 + 审稿区"（前端 ECR MVP） |
| D4 | Daisy 真实内容跑一天，bug 修复 + 打磨 |

**V2 迭代**（1-2 周，按 V1 反馈驱动）：其他 5 员工（选题/分发/社区/商务/财务）/ 小红书半自动 / 薪资面板 / 预览卡机制 / 企业档案手填 UI。

**V3 迭代**（按需）：秘书 agent / 共享记忆层产品层 cc 风格 / 资源中心 / 园区看板 / 分层认证+交易信用 / 多行业扩展 / 私有化部署。
