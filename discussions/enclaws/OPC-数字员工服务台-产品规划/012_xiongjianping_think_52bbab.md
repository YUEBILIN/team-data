# 产品顶层叙事 · 同舟 - OPC 协作平台

> 给团队对齐认知用。
> 
> **平台名（团队拍板前的候选清单）**：同舟（默认采用）/ 百业 / 派客 / 匠云。本文行文统一用"同舟 - OPC 协作平台"占位，定稿后做一次性 replace。

---

## §一 产品定位

> **同舟是 OPC 协作事项的完整时间线记录器 + 生态调度器。**

三个关键词：

- **协作事项** —— 不仅仅是"项目"，是从一句话需求开始到验收复盘结束的完整工作主体；覆盖任意业态（设计/技术/内容/咨询/数据），不为任何业态画专属流程。
- **时间线** —— 一条事项从立项到沉淀是同一根线上的演进，不是 5 个独立 tab。借鉴 Pivot 的 matter 模型。
- **生态调度器** —— 平台不只做双方撮合，还主动调度 AI 员工 / 资源 / 政策 / 合规，让事项推进顺畅。

## §二 给谁用、为什么用

| 谁                | 为什么用                      | 我们怎么支持                                                          |
| ---------------- | ------------------------- | --------------------------------------------------------------- |
| **链主企业**（发包方）    | 把碎活外包给 OPC，省内部用工，又要敢放更大的单 | AI 量化 OPC 产能（动态评级 + 验收数据 + 申诉记录）+ 园区/政府背书 + AI 监控贯穿履约 + 履约保证金兜底 |

| **平台运营**（园区/研究院） | 数字化运营 OPC 生态，可见可控可决策      | 全链路漏斗 + 多维评级（持续标准）+ 政策执行抓手 + 公共活动管理工具                           |

## §三 三视角的工作动线

### §3.1 平台运营 · 运营驾驶舱

**设计原则**（蒋鲲反馈核心）：**可见可控（驾驶舱）+ 全链条评估（漏斗）+ 持续标准（动态评级，不一次性认证）**。明确反对"首页=待办"——任务管理本身就难做好，运营人不靠 todo 推进工作。

**完整动线**：

```
登录
  → 看运营驾驶舱（健康度 / 漏斗 / 待决策项）
    → 处理具体决策（OPC 准入审批 / 评级调整 / 三个月无经营淘汰）
      → 组织公共活动（链主-OPC 沙龙 / 培训 / 路演）
        → 复盘活动转化和生态指标
```

### §3.2 发包方 · 从注册到放款的完整闭环

```
注册 → 登录 → 创建公司 → 配置 AI 员工
  → 发起需求 → AI 多轮对话产 SOW → 智能匹配候选 OPC
    → 招募 → 报价审阅 → 选定中选方 → 合同签署
      → 履约监控（AI 监控员）→ 里程碑验收 / 终验
        → 放款 → 复盘 → 沉淀（信任池 / SOW 模板）
```

**3 个关键决策点**：选哪家 OPC（AI 给数据、人拍板）/ 验收阶段放不放款（多维评分 + 冷静期）/ 出现争议怎么办（SOW 锚定异议 + 平台仲裁）。

### §3.3 接单方 · 从注册到沉淀的完整闭环

```
注册 → 登录 → 创建公司（可先模拟运营）→ 配置 AI 员工
  → 项目大厅找单 → 接包风控评估 → 提交报价
    → 等评选 → 中选 → 履约（AI 监控员）
      → 交付（AI 验收）→ 收款 → 复盘 → 沉淀（画像评级提升 / 案例库）
```

**3 个关键决策点**：接不接（接包风控 + 自身画像匹配度）/ 怎么报（市场参考价 + 历史成交分布，AI 不替你定价）/ 如何交付（M3 整改建议 + 验收预演）。

### §3.4 横切支撑动线（贯穿三视角）

**做什么**：法务 / 财税 / 政策红利申报由专属 AI 员工承担（法务助理 / 财税助理 / 政策对接员），在关键业务节点**主动触发**，不靠用户自己想起来用。

**目标和价值**：OPC 创业者很多是首次创业，法务/财税/政策不熟；即使配了 AI 员工，如果用户不主动调用就形同虚设。必须由秘书在关键节点主动推送，把"非业务支撑"嵌进业务动线里。

**触发节点**：

| 节点      | 触发的支撑动作            |
| ------- | ------------------ |
| 注册创建公司  | 一人公司注册引导（含工商/税务登记） |
| 合同签署前   | 法务条款审核             |
| 中选 / 收款 | 开票、入账、税务提醒         |
| 季度 / 年度 | 财税申报清单生成           |
| 政策窗口期   | 算力券、补贴申报提醒和指导      |

**入口**：AI 工作台秘书主动推送 + 用户中心"我的支撑"独立页面。

## §四 原型设计要点

> 9 条贯穿全产品的设计原则，每条标 ✅ 已做到 / 🚧 计划修改。

### 1. tab 按用户业务对象划分，不按 AI 能力划分 ✅

**做什么**：tab 命名以"项目大厅 / 我的申请 / 项目监控 / 需求管理"等业务对象组织，AI 能力嵌在每个业务对象内部。

**目标和价值**：用户找东西的心智模型是"我的项目在哪"，不是"AI 澄清在哪 tab"。原本"AI 需求澄清 / 智能匹配 / 报价审阅 / AI 监控"5 个独立 tab 的方案被否决——会让用户每完成一步就要切 tab、丢上下文。

### 2. AI 嵌入每个高摩擦节点 ✅

**做什么**：在传统外包 5 大痛点节点都嵌入 AI ——

- **需求澄清**：AI 多轮对话产 SOW
- **智能匹配**：7 维加权 + 候选漏斗
- **投标反同质化**：NLP 识别 AI 模板/批量海投
- **履约监控**：实时扫代码与沟通 → 提前 2-3 周预警 + 自动整改建议
- **多维度验收**：5 维 AI 质检 + SOW 锚定异议

**目标和价值**：传统外包 90% 纠纷源于"信息不透明 + 标准不可量化"，AI 在每个高摩擦点把客观数据摆出来。

### 3. AI 是地图，不是裁判 ✅

**做什么**：所有 AI 决策入口都给数据 + 让用户拍板，不替任何一方做商业决策。

**目标和价值**：避免 AI 越位——不替链主拍价格、不替 OPC 砍需求、不替任何方下结论。决策权始终在人，避免纠纷把锅甩到平台。

**怎么做**：所有 AI 输出都附"客观依据"（评分维度、对比基准、历史成交分布），用户在地图上自己选路径。比如智能匹配给出 7 维加权得分但不给"推荐选哪家"；价格参考给出市场分布但不给"建议报价"。

### 4. AI 会话标签页最前 + 待办/定时任务融入会话 🚧

**做什么**：AI 会话作为最前置的 tab；待办、定时任务不占独立标签页，融入 AI 会话作为秘书的主动推送内容。

**目标和价值**：AI 是产品定位里的"调度器"，自然要做最前置入口；待办/定时任务作为独立 tab 不容易做好（任务管理本身就难），由秘书在合适的时间主动提醒比让用户自己开 tab 维护更顺。

**怎么做**：AI 会话主区 + 侧栏 widget（事项时间线/待办/定时任务）。具体布局留待原型迭代验证，不在叙事固化。

### 5. 事项时间线视图 · 让"大脑-手脚"层级可见 🚧

**做什么**：详情页加"时间线"视图，纵向 activity feed 呈现事项从立项到沉淀的全过程；**每张卡片明确标注发起主体**——用户 / OPC 秘书 / 某个 AI 员工（监控员 / 验收员 / 法务 / 财税 ...）/ 系统。

**目标和价值**：

- 一个事项的完整上下文（SOW / 匹配 / 报价 / 履约 / 验收 / 复盘）不被切碎到 5 个独立 tab；用户和 AI 都能"通读时间线 → 理解状态 → 推动下一步"。这是借鉴 Pivot matter 模型，反对碎片化多 tab 切换。
- **更核心的价值**：让用户一眼看清"**OPC 秘书是大脑、AI 员工是手脚**"的层级关系。OPC 必须把这层关系做得同样可见，否则即使把秘书 + AI 员工都做了，用户仍会把它读成一堆孤立 AI 工具的拼盘。

**怎么做**：

- 纵向倒序的 activity feed（卡片列表），每张卡片含：**发起主体徽标**（用户头像 / 秘书图标 / AI 员工角色图标 / 系统）+ 行动内容 + 产物链接 + 可展开详情
- **秘书的调度行为本身要作为可见卡片**：例如 "OPC 秘书 14:23 → 调度 AI 监控员检测 M2 进度风险" 紧接着 "AI 监控员 14:25 → 输出 M2 风险报告（红：进度滞后 2 天）"；两张卡片视觉上有连接线/缩进，呈现"大脑命令 → 手脚执行 → 产物回填"的因果链
- **不做 dot 时间轴** —— OPC 协作跨度数周-数月、节点 10-15 个，dot 密度太低、信息密度差

### 6. 多公司切换 + 秘书会话按公司隔离 🚧

**做什么**：header 切公司，秘书随之切换上下文焦点。

**目标和价值**：一个用户管多家公司是 OPC 形态常见场景（同时是发包+接单，或管多家）；AI 秘书要能感知公司上下文，但**绝不能串味**。

**怎么做**：MVP 期"切公司 = 切会话"——每家公司一个独立会话栈，秘书还是一个但不跨公司延续上下文；跨公司上下文延续 v2 再做（等 RAG / 多 agent 隔离方案成熟）。

### 7. 验收兜底组合 ✅

**做什么**：组合 5 个机制保证验收公平：AI 多维评分 + SOW 锚定异议 + 冷静期 + 履约保证金极简版 + 平台仲裁。

**目标和价值**：传统外包"一手交钱一手交货"做不到，弱势方吃亏；OPC 平台必须给可信的兜底机制，让链主敢付款、OPC 敢交付。

### 8. 业态适应：主流程通用 + AI 按业态动态生成 🚧

**做什么**：主流程骨架（澄清→匹配→报价→选定→履约→验收）所有业态共用，**不为业态画专属流程图**；业态差异由 AI 在每个节点动态适配。

**目标和价值**：蒋鲲反馈"OPC 是组织模式不是产品，要对各种业态适应"。如果为每个业态预设一套流程，平台扩业态成本爆炸；如果只做通用流程，又对不上业态特性。解法是把"业态智能"放在 AI 节点里。

**怎么做**：

- **业态字段**贯穿事项始终（餐饮 SaaS / 文创 IP / 数据标注 / 咨询 / ...）
- **SOW 模板**按业态生成（AI 拉取业态标准/案例库）
- **匹配权重**按业态调整（开发重 GitHub 评分、设计重作品集）
- **验收 5 维**按业态可换（开发：功能/性能/可维护/安全/兼容；设计：美感/可用性/创意/品牌契合/源文件可交付）
- **AI 员工技能**有业态标签，业态决定调度哪个

**新业态接入成本**：不重画流程图，只给 AI 喂业态参考案例和标准 → 自动产出适配版 SOW / 评分维度 / 监控点。

### 9. 流程设计站在经营逻辑，而非 AI 技术能力 🚧 持续校准

**做什么**：每个节点先问"链主 / OPC 在这一步要做什么决策、有什么顾虑"，再决定 AI 在这里放什么能力；不先问"我们这里能放什么 AI"。

**目标和价值**：蒋鲲反馈"AI 对场景融入快，要站在经营逻辑做流程设计"。技术驱动思维容易堆功能（"我们能做 X 就放 X"），业务驱动思维才能让流程丝滑。

**怎么做**：每次设计/修改节点前先做"经营逻辑梳理"——这一步用户的目标/障碍/常见决策路径是什么；AI 能力作为这些经营动作的辅助工具，不是流程的主角。
