# OPC 产品讨论 · 四层拆分

> 回应 015 dengke：产品讨论拆为产品宏观定义、五一前 MVP、两周迭代、长期规划四层。V2 回应 017 dengke：补齐数科院运营方场景。V3 修正：区分"原型有这个页面"和"MVP 真交付"，按时间/质量/优先级合理分级。

---

### 第一层：产品宏观定义（对外沟通版）

**定位**：面向"一人公司（OPC）"的协作平台，以园区运营方为生态组织者（组织链主释放订单、管理 OPC 准入与评级），通过 AI 工具+平台生态让 1 人完成原本 5 人团队的工作，安全高效地匹配链主订单与 OPC 产能。

**三类核心用户 & 关键动线**

| 用户 | 核心痛点 | 关键能力 | 主操作动线 |
|---|---|---|---|
| **链主企业（发包方）** | 碎活外包易失控、验收标准模糊 | AI 生成标准化 SOW → 智能匹配 OPC → 全程履约监控 → 多维验收 | 发起需求 → AI 产 SOW → 选定 OPC → 监控/验收 → 放款复盘 |
| **OPC 创业者（接单方）** | 获单不稳定、产能受限、合规门槛高 | 项目大厅 → AI 接包风控 → AI 秘书调度 → 政策/财税/法务支持 | 找单 → 风控评估 → 报价/中选 → AI 辅助履约 → 交付收款 |
| **平台运营（数科院/园区）** | 生态运营黑盒、缺乏决策抓手 | 运营驾驶舱 → OPC 准入/评级/淘汰 → 公共活动与转化追踪 | 看数据大盘 → 审批/干预 → 组织活动 → 复盘生态健康度 |

**核心设计原则（对外沟通精简版）**

- 流程按业务对象组织，不按 AI 能力切分
- AI 是"地图"不是"裁判"：所有 AI 输出附客观依据，最终决策权归人
- 一事项一时间线：全生命周期记录不被切碎到独立页面

### 第二层：五一前 MVP 基线（交付边界 & 验收标准）

**目标**：跑通"发单→接单→履约→验收"主链路骨架，并给数科院一个可见、可汇报的最小运营视图。

#### 发包/接单主链路

| 页面 | 状态 | 说明 |
|---|---|---|
| 需求创建页 | 🔵 真功能 | 基础表单（标题/简述/预算/周期/标签） |
| 项目大厅 | 🔵 真功能 | 列表展示已发布需求，接单方入口 |
| 需求详情页（时间线） | 🔵 真功能 | 纵向 Activity Feed，核心载体 |
| 报价/接受/完成操作区 | 🔵 真功能 | 交易核心动作 |
| 角色切换（三视角） | 🔵 真功能 | 前端下拉模拟，bypass 鉴权 |
| AI 需求澄清 | ⚠️ 演示态 | 占位 UI，首期不接 LLM |
| 智能匹配 OPC | ⚠️ 演示态 | 静态 mock 卡片，无真实画像库 |
| AI 监控/验收/法务/财税 | 🚫 不做 | 强依赖 Agent，短期不可交付 |

#### 运营方（数科院）视角

| 能力 | 状态 | 说明 |
|---|---|---|
| AI 工作台 · 运营 widget | 🔵 真功能 | 3 组 KPI 卡片（健康度 92 / 活跃 OPC 237 / 待决策 3 项 / 首单转化 / 无经营淘汰），静态数字但覆盖核心关注点 |
| 运营事项总览 | 🔵 真功能 | 跨所有发包方的事项列表，含状态筛选（招募中/履约中/已完成），可钻入任一事项目时间线。这是 dengke 017 要求的"最小运营场景"——能看、能汇报 |
| 运营管理（入驻审批+工位+会议室） | ⚠️ 演示态 | 原型已有完整 UI 和 mock 数据（5 家待审+AI 预审+80 工位图+会议室），但数据与交易链路无关。MVP 期仅作演示，向数科院展示运营管理页面的设计方向 |
| OPC 评价中心（漏斗+评分卡） | ⚠️ 演示态 | 原型有全链路漏斗和评分卡 UI，但全为静态 mock。MVP 期展示概念，不标"已交付" |
| 治理中心（风控+保证金） | ⚠️ 演示态 | 静态卡片，MVP 期仅演示概念 |
| 社区广场（feed + 公告框架） | ⚠️ 演示态 | 基础 feed 可用，但置顶/发起活动按钮为占位 |

**运营方 MVP 的诚实表述**：数科院切到平台视角后，能看到 KPI 卡片 + 全平台事项列表 + 钻入时间线。这满足了"可见、可汇报"的最小要求。ops/评价中心/治理中心三个 tab 在原型里可以点开看 UI 概念，但在 MVP 基线中标记为演示态——它们不连接真实数据流。

#### 已知缺陷

1. 无真实 AI：SOW 全靠手填
2. 无权限隔离：三视角靠前态切换，无后端租户
3. 匹配逻辑缺失：OPC 推荐为静态数据
4. 运营数据为静态 mock：KPI 数字、事项列表均非实时
5. 无文件/合同流：不支持附件留存
6. 验收纯手动：点"完成"即结束

#### 验收标准

- 发包/接单主链路跑通：创建→发布→大厅可见→报价→接受→履约中→完成
- 运营方可看：切平台视角 → KPI widget 可见 → 事项总览列表可筛选 → 钻入时间线
- 无阻断缺陷：无 500/白屏/表单丢数据
- 首屏 <2s
- 数科院代表 5 分钟内完成：看 KPI → 看事项列表 → 点进一个事项 → 看时间线

### 第三层：两周迭代优先级

| 优先级 | 能力 | 面向角色 | 说明 |
|---|---|---|---|
| **P0** | AI 需求澄清（单轮 LLM） | 发包方 | 替代手填 SOW，调 API 不涉 Agent，产品核心价值点 |
| **P1** | 完整事项时间线 | 三方 | 补齐报价/接受/里程碑/验收等卡片类型，三方都受益 |
| **P2** | OPC 业务分析看板 | 运营方 | 活跃度/业务量/资源占用基础统计，识别"持续无经营"和"活跃应扶持"——这是数科院刚需，比 UI 打磨更重要 |
| **P3** | 入驻审批联动真实数据 | 运营方 | 把 ops 从静态 mock 改为接入注册数据流，让入驻审批从"演示"变"可用" |
| **P4** | 活动公告/组织/报名 | 运营方+OPC | 培训、沙龙、路演、对接活动的发布与报名闭环 |
| **P5** | AI 输出人工确认/编辑 | 发包方 | SOW/匹配结果可二次修改，解决 009 的 AI 主观性风险 |

**优先级逻辑**：P0-P1 补齐交易链路核心体验 → P2-P4 把数科院从"旁观"升级为"能用" → P5 解决 AI 可信度。两周后运营方不再是只看 KPI 卡片，而是能做入驻审批、看业务分析、发活动。

### 第四层：长期产品规划

**不作为短期承诺，仅作战略方向同步**：

- **运营驾驶舱**：全链路漏斗、健康度指标、待决策项聚合（原型 opc-rating 的漏斗+评分卡在长期才真正数据化）
- **动态评级体系**：基于履约/验收/申诉的持续评级，替代一次性认证
- **AI 员工市场**：法务/财税/政策/监控/验收等角色动态调度
- **多业态适配**：主流程通用 + AI 按业态生成 SOW/监控点/验收标准
- **仲裁与保证金**：SOW 异议处理 + 极简兜底
- **多公司切换 & 上下文延续**：RAG/多 Agent 隔离成熟后
