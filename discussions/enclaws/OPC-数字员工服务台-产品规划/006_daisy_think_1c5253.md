# OPC 真实需求调研简报

**版本** v1 · 2026-04-27
**作者** Daisy
**性质** 多源交叉验证调研（11 个一手来源）
**用途** 为 OPC 数字员工服务台 PRD 产品方向决策提供市场数据依据

---

## 调研背景

OPC 数字员工服务台 PRD 推进过程中出现两个关键分歧：

1. 张菠在 thread3 提议 V1 跑「订单履约闭环」，把产品做成企业供需协作平台
2. 团队内部对"易用 AI 工具是否是 OPC 刚需"存在不同判断

为避免方向决策建立在直觉而非数据上，本次围绕"OPC 真实痛点 / 入园动机 / 工具使用现状 / 商业模式"做了一轮市场调研，从 11 个一手来源（含杭州 1500+ 份问卷、澎湃/36 氪/四川经济网现场访谈、网易争议报道、OPC 圈社区高频帖等）做了交叉验证。

**结论一句话**：OPC 老板的痛点优先级是「**获客 > 孤独 > 资源对接 > 补贴流程 > 工具效率**」，"易用 AI 工具"是 P4 级需求。做"工具集"路线大概率打不到 OPC 真刚需。撮合平台路线在外部企业侧也无市场数据支撑。

---

## 一、五个核心发现

### 发现 1 · OPC 痛点排序：孤独和获客是 #1，工具垫底

**杭州 1500+ 份问卷调研**：
> 75% 非技术背景，月入 5 万也有焦虑，**孤独与获客是最大隐形挑战**

**澎湃新闻三位创业者访谈共识**：
> 工具费便宜，几百块、一两千一个月就能顶上过去请不起的人。但**获客、投流、公司运营，花钱的地方有的是**

**36 氪 1000 万营收案例（素源矩阵）总结**：
> **有需求的人找不到技术，有技术的人不知道需求在哪里**

排序确凿。这不是猜测，是真实创业者用钱投票投出来的结论。

---

### 发现 2 · OPC 工具已经"够便宜够用"，甚至订阅过剩

**苏魁（龙骨动画，36 氪封面案例）成本结构**：
> 采用腾讯 Cloud Studio、DeepSeek、豆包等 AI 工具辅助……公司注册 2000 多元、代理记账一年 1000 多元、服务器成本可忽略不计、**AI token 消耗 1 个月几十元**，**一年成本不到 5000 元**

**OPC 圈社区（最大 OPC 站）真实高频帖**：
> 最近发现自己**订阅了太多东西**：AI 工具、网盘、会员、云服务……有些已经很久没用了，**但还在扣费**

两个含义：
- 工具已经过剩，不是稀缺。OPC 老板真实抱怨是订阅太多取消不过来
- token 消耗一个月几十元——如果按 token 量给政府补贴，单 OPC 一年也就上千块基数，**这个数据维度撑不起补贴依据**

---

### 发现 3 · OPC 入园核心动机：社群 + 资源对接 + 注册降本，不是工具

**成都首个 OPC 社区现场报道**：

史炳诚（赛博西游开发者）：
> 在家最大的困难是**很"闷"**，一个人没头没尾地开发，也没多少人交流……到这里来能有一个集中的地方可以**获取资源对接**，比一个人到处跑强太多了

袁欣（剧灵绘）：
> 这意味着我们可以把原本要交房租的钱，**投入到全栈工程师的薪资和高质量数据的采购**上

**隔墙对接模式**：
> 一个 AI 数字人创意，可以直接向隔壁的数字天空技术团队演示

**网易 OPC 社区争议报道**：
> 注册一家企业对于大部分起步阶段的 OPC 是没必要的，尤其是做服务业务的个体户

OPC 入园拿三件事：**省钱（免租）+ 解闷（社群）+ 撞机会（隔墙对接）**。工具排不上号。

---

### 发现 4 · 真正赚到钱的 OPC，赚的不是 AI 工具的钱，是垂直行业 know-how 的钱

**36 氪「一人公司年入千万」案例汇总**：

| 案例 | 团队 | 怎么赚的 |
|---|---|---|
| Medvi 线上医疗 | 2 人 | 营收 4 亿美元，预期 18 亿 |
| 素源矩阵 工业 AI | 7 人 | 预计 1000 万，找水泥/钢铁/化工 |
| 声序智能 | 1+ | 民乐五线谱→简谱，B 端切入 |
| 画宗 | 1+ | AI 短剧创作 |

共同点：**全部靠"找垂直小场景"赚钱，没有一个靠"用 AI 工具"赚钱**。

**苏魁现状（36 氪封面案例）**：
> 在 B 站做了 16 期教程，420+ 粉丝……目前付费用户约 300 人,**总收入近万元**（注：从 2025 年 5 月做到 2026 年初，约 9 个月）

苏魁是封面，但严格意义上是个挣扎样本——不是反例，是常态。**MIT 报告：AI 创业失败率 95%。**

---

### 发现 5 · 政府和园区运营方利益不一致（隐藏张力）

**网易 OPC 社区争议报道**：
> 政府肯定希望政策以**更低成本落地，让更多人入局**，门槛就必须足够低……但是 OPC 社区是靠**收取租金盈利**的，因而希望让具备**高潜力的项目入驻社区**

这条很关键。它意味着「政府」和「园区运营方」不是同一个客户：

- **政府要的**：覆盖广、入驻多、统计漂亮、补贴有据可发
- **园区要的**：质量高、租金稳、KPI 可持续、运营有抓手

我们做产品时**到底站谁那边**，决定了产品形态完全不同。

## 二、对当前 PRD 方向的四个含义

### 含义 1 · EC「易用 AI 工具集」打的是 P4 级痛点

P0–P3 都是 OPC 老板更痛的事，且基本都不能用 SaaS 工具解决：

- **P0 获客** = 销售/品牌 know-how，不是工具问题
- **P1 孤独** = 物理空间和社群，工具替代不了
- **P2 资源对接** = 园区线下生态在做
- **P3 补贴流程** = 政府办事窗口

EC 做得再好，**单 OPC 付费意愿也不会高**——他们已经在用免费的 DeepSeek/豆包/Cloud Studio。

### 含义 2 · 单 OPC 的 token 消耗量不足以支撑"按 token 发补贴"的逻辑

苏魁案例：月 AI token 消耗几十元。一个园区 100 个 OPC 全年总消耗对政府而言是个**很小的数字**。

政府在政策文件里写的"算力补贴最高 500 万"是给做 AI 模型/AI 平台的中大型团队的，不是给单个 OPC 的。我们想用"OPC token 消耗"作为补贴依据这条路在数据量级上**站不住**。

### 含义 3 · 政府能用的"真实数据"目前只有营收和就业是有量级的

苏魁数据回看，政府能用的字段实际只有：

- ✅ 在营 OPC 数（注册了就有）
- ✅ 月度营收（自报或票据）
- ✅ 雇佣人数（社保挂靠）
- ⚠️ Token 消耗（量级太小）
- ⚠️ 履约率/客户数（对纯产品型 OPC 不适用）

**最简的看板字段就是营收 + 主体数 + 雇佣**，剩下都是装饰。

### 含义 4 · 张菠提议的"双边撮合"在数据上不被支持

调研里没有任何数据支持"企业会主动到 OPC 平台发布需求"这个假设。OPC 老板的获客渠道是 TikTok / 小红书 / Hacker News / 私域 / Google SEO 等已成熟生态。"OPC"在发包方眼里是劣势标签不是优势标签——这一判断被全部一手来源印证。

## 三、产品方向重新选型（三条候选）

### 候选 A · 工具站（当前 EC 方向）

| | |
|---|---|
| **做什么** | 5 场景 AI 员工（视频/设计/内容/分发/...） |
| **真买家** | OPC 老板（付费意愿低） |
| **风险** | 打 P4 级痛点，红海，单 OPC 付费弱 |
| **判断** | **不建议作为主线** |

### 候选 B · 园区运营 SaaS（B 端转向）

| | |
|---|---|
| **做什么** | 给园区的「OPC 入驻审核 + 阶段评审 + 出勤/营收记录 + 路演活动管理 + 政府数据看板」 |
| **真买家** | 园区运营方 + 政府 |
| **OPC 使用动机** | **为了拿补贴必须维护**（不需要"觉得好用"才用） |
| **和园区关系** | 赋能而非竞争 |
| **风险** | 销售周期长，客单价高但客户数少 |
| **判断** | **最贴合政府"工具+数据"双重需求**，且与现有 OPC 生态最互补 |

### 候选 C · 获客放大器（直击 P0）

| | |
|---|---|
| **做什么** | OPC 个人 IP 卡片 + 多平台内容分发 + 接通蒲公英/通告平台 |
| **真买家** | OPC 老板 |
| **风险** | 竞品多，技术门槛低，差异化难 |
| **判断** | **直击真痛点但红海** |

## 四、待团队决策的三个核心问题

### Q1 · 我们的真买家到底是 OPC 老板，还是园区/政府？

这决定了候选 A vs B 的选择，也是 PRD 里"四方边界"需要明确的核心。本次调研倾向于**B 端转向**（卖给园区/政府），因为：

- OPC 老板付费意愿弱
- 政府和园区有真预算
- OPC 使用动机不需要培育（为拿补贴必须用）
- 和现有 OPC 生态互补不竞争


### Q2 · Daisy 跑 Phase 1 种子时，验证什么假设最有价值？

- 选 A：验证"OPC 老板愿意为易用工具付费" — **市场数据已经预示答案是否定的**
- 选 B：验证"园区运营方愿意采购 SaaS 来管理 OPC 和出补贴看板"
- 选 C：验证"我们能帮 OPC 多接到一个客户"

## 五、附录：被验证的判断清单

| 假设 | 验证结果 |
|---|---|
| OPC 在园区主要是为了拿工具支持 | ❌ 主要为社群、对接、降本 |
| 企业会主动到 OPC 平台发包 | ❌ 没有任何数据支持 |
| 撮合交易是政府/园区的需求 | ❌ 政策文本里没有这条 |
| OPC 用 AI 工具不熟练 | ✅ 部分群体确实如此（75% 非技术背景） |
| OPC 工作效率提升是政府刚需 | ⚠️ 是政府叙事的需求，但单 OPC 维度的提效数据量级太小，撑不起补贴依据 |
| 营收/就业事实是政府刚需 | ✅ 最直接、最可量化、最能撑起补贴依据 |
