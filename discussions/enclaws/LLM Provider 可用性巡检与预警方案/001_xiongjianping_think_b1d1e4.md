# LLM Provider 可用性巡检与预警方案

> 日期：2026-04-16
> 作者：xiongjianping

---

## 一、背景

最近 CS widget 在 dev 部署时踩了 "400 Model Not Exist"——租户 agent 配置的 model 在运行时不可达，widget 气泡出现但点进去就报错。直接原因是几个前端/后端 bug（已在 hotfix 修复），**深层原因是 EC 平台缺少「主动感知模型可用性」的基础设施**：运维和租户在真正调用前都不知道哪个模型已失效、余额耗尽、订阅到期。

本 proposal 规划这套基础设施。

---

## 二、设计原则

### 2.1 不用 test prompt 消耗 token 验证

最直接的健康检查做法是"发个 'ping' 看回响应"——但在 SaaS 规模下这会变成 memory-leak 式不可控成本：

- 每次页面刷新巡检一次：多用户场景下 N 倍放大
- 巡检周期设短：总成本线性增长
- 巡检周期设长：失效感知不及时
- 各租户各模型组合数量增长：成本组合爆炸

token 消耗型健康检查**不具备 SaaS 规模下的经济可持续性**。

### 2.2 用厂商已有 API 巡检

大多数 LLM provider 都提供余额 / 订阅 / 额度查询 API，**调用这些 API 本身不消耗对话 token**。核心思路：

- 把"模型能不能用"转化为"**账户有没有钱、订阅有没有到期、API Key 有没有失效**"
- 这些信号更本质——真实失败场景里，绝大多数是账户问题而非偶发调用失败
- 巡检成本几乎为零，可以高频运行

### 2.3 能力归属 EC 底层

这不是 CS 模块的功能——CS 只是**第一个**暴露需求的消费方。后续多个上层业务都会依赖：

- CS widget gate 判定（配置完整才显示气泡）
- Agent 管理页"测试"按钮（不再即席 ping）
- 套餐预检 / 部署就绪检查
- 平台侧运维巡检告警

能力应由 EC 底层提供统一 primitive，上层业务只需要读快照。

### 2.4 合格性由单一事实源定义

"一个租户/环境是否合格（可用）"这个判断，**只能有一个权威定义**。现阶段这个定义就是连通性测试（readiness checks）的结果——所有检查项全部通过即合格。

- 连通性测试持有合格性的完整定义，随业务演进自然扩展检查项
- Gate、UI、套餐预检等所有消费方**引用** readiness 结果，**不自己定义**什么是合格
- 加新检查项（如本方案的健康快照）只改一处 readiness 逻辑，所有消费方自动跟进

这是保持演进不发散的底线——避免"gate 认为合格但 UI 显示未就绪"这类事实源分裂问题。

## 三、厂商 API 清单（需团队补充完善）

初步调研，具体 endpoint 细节各家独立调研：

| Provider | 可用 API（示意） | 返回信号 | 支持度 |
|---------|---------------|---------|------|
| OpenAI | `/v1/organization/costs` / billing subscription | hard_limit / 已用额度 / 到期时间 | 应有 |
| Anthropic | `/v1/organizations/usage_report` | 日用量 / 月成本 | 待确认 |
| DeepSeek | `/user/balance` | 余额 / 赠送余额 | 应有 |
| Qwen / 百炼 | 账单 API | 订阅到期 | 待查 |
| Moonshot | 余额 API | 余额 | 应有 |
| Gemini | Google Cloud billing（间接） | 项目级账单 | 需走云账号 |
| 豆包 | 火山引擎账单 API | 账户余额 | 待查 |

**厂商不支持的降级方案**：

- 无查询 API 的 provider 提供"手工维护到期日"输入——租户在模型管理页填入到期日，系统按日历提醒
- 调用 API 失败时标记为"未知"状态——不阻塞业务，但 UI 做灰色提示

## 四、巡检机制

| 属性 | 说明 |
|------|------|
| 触发 | 平台级 cron 任务（复用现有定时任务系统），默认每 4 小时一次 |
| 粒度 | 以 `tenant_models` 表的每条记录为单位 |
| 去重 | 同 provider + apiKey 组合只查一次（多 model 共享同 key） |
| 存储 | 新增 `model_health_snapshots` 表：tenant_id / provider / api_key_fingerprint / balance / expires_at / last_check_at / status / error_message |
| 频率上限 | 同一 provider+key 组合最多每小时 1 次（防 rate limit） |
| 失败重试 | 连续 3 次失败标记为 `status=unhealthy`，触发告警 |
| 缓存失效 | 租户更新模型配置时主动失效对应快照，立即触发一次巡检 |

## 五、预警与展示

**统一复用公告通知基础设施**（见 `enclaws-saas版的菜单及功能调整规划` §5.7 定义的"顶部横幅 + 铃铛 + 公告中心"）。

告警和公告共享 UI，只是数据源不同：

- 公告 = 平台**主动发**的（维护通知、新功能、政策）
- 告警 = 系统**自动检测**触发的（余额不足、到期、巡检失败）

### 5.1 告警分发矩阵

| 告警类型 | 受众 | UI 通知 | 频道通知（如配置了） |
|---------|------|---------|-----------------|
| 平台共享模型告警 | 平台管理员 | 平台视图顶部横幅 + 铃铛 | 平台默认频道（运维飞书群） |
| 租户自有模型告警 | 租户管理员 | 租户视图顶部横幅 + 铃铛 | 租户配置的默认通知频道 |
| CS 客服 gate 降级 | 平台管理员 + 运营方租户管理员 | 同上 | 同上 |

### 5.2 关键设计点

1. **告警 = 公告的一种类型** — 数据结构上公告表加 `type: "announcement" | "alert"` 字段；`alert` 类型默认触发顶部横幅；UI 展示加红色图标区分
2. **去重防骚扰** — 相同告警（如"余额 < 10 元"）未恢复前不重复推送；恢复后重新触发一次"已恢复"通知
3. **频道通知是叠加** — UI 通知永远发（全局基线）；频道通知按需叠加：配置了默认频道才走，没配置就只有 UI 通知
4. **默认频道的定义**：
   - 短期：复用 CS 模块已配的 feishu chatId（每运营方租户已经有一个）
   - 长期：抽象为"租户通知频道"一等公民概念，CS / 告警 / 定时任务失败通知共享

### 5.3 告警配置管理

公告管理页（平台侧菜单）兼管告警配置：

- 告警阈值配置（余额 < X、到期 < N 天、巡检失败 N 次）
- 静默期配置（避免半夜推送，按用户时区）
- 告警历史（通过铃铛下拉 → 公告中心查看，和公告统一入口）

菜单规划 proposal §5.7 需要小幅扩展明确这一点。

## 六、和已有能力的协同

### 6.1 CS 连通性测试的自然演进

连通性测试作为合格性的唯一事实源（见 §2.4），随业务演进持续扩展覆盖范围：

- **现有检查项**：飞书凭据、飞书 API 连通、知识库已上传
- **本次 hotfix 阶段新增**：Agent 存在且绑定了 model ref（运行前提保障）
- **本方案落地后新增**：模型健康快照读取（替代即席 token ping，按 ✓/⚠/✗/? 四级展示）

### 6.2 Agent 管理页的"测试"按钮

- 同样读快照，不真 ping
- 可选增强：提供"立即巡检"按钮——强制触发一次厂商 API 查询，但**不消耗对话 token**

### 6.3 CS gate 的落地节奏

- **本次 hotfix**（最小修复范围）：gate 在 server-http.ts 内 inline 检查"配置存在"（agent+model+chatId）。该检查与 `cs.config.test` 的部分 checks 逻辑存在轻量重复，作为 hotfix 可接受——避免扩大 hotfix scope 做 refactor，保持"尽快上线修复"优先级
- **本方案 P0 落地项**：抽出 `runCSReadinessChecks` primitive，`cs.config.test` 和 gate 统一调用，消除重复，完成 §2.4 单一事实源原则的实际落地
- **本方案全量落地后**：连通性测试里的"模型可用性"项从"有 model ref"升级为"健康快照为 ✓/⚠"，gate 行为自动随之严格化，无需修改 gate 代码

### 6.4 套餐预检

- 部署就绪检查的核心一项
- 租户创建完成、配完 agent、配完模型后，通过健康快照确认可用性，再激活套餐

## 八、待讨论问题

1. **`model_health_snapshots` 的存储选型**：新建表 vs 复用某个 KV store？
2. **厂商 API 的调用策略**：多租户场景下同 provider 不同租户 key 都要查，rate limit 怎么分配？
3. **告警的「紧急度」阈值**：余额告警的多少判定为紧急（顶部横幅）vs 普通（仅铃铛）？
4. **租户通知频道一等公民化**：是本 proposal 范围内，还是单独 proposal？
5. **巡检失败的业务降级**：巡检持续失败时，是否主动 downgrade gate 判断（比如租户配置本来合格，但连续 N 次巡检失败，是否隐藏气泡）？
