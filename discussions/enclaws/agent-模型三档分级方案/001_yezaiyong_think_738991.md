> 用途:理解整体方案、做最终决策
> 配套技术稿:`2026-04-17-agent-model-tier-design.md`(供研发参考)

## 一、问题与目标

### 1.1 现在的问题

EnClaws 平台上,每个 agent 只能配置一种模型策略——不管用户问的是"明天北京天气怎么样"这种简单问题,还是让 agent"帮我重构这段几千行的代码",**走的都是同一个模型**。

这导致两类常见浪费:

- **简单问题用了高端模型** → 成本浪费
- **复杂问题用了基础模型** → 效果不达标,用户体验差

### 1.2 想达到的目标

让一个 agent **既能处理简单任务也能处理复杂任务,自动用最合适档位的模型**,同时:

- 成本可控(不会"偷偷用贵的")
- 行为可预测(不会一会儿对一会儿错)
- 用户无感(不需要用户额外学习成本)

### 1.3 解决思路:分两件事做

| 模块 | 解决什么 | 谁用 |
|---|---|---|
| **配置侧** | admin 在创建 agent 时,可以一次性配好"高/中/低"三档模型 | admin/管理员 |
| **运行时侧** | 用户实际发消息时,系统**自动**判断该用哪一档模型 | 终端用户(无感) |

下面分两块讲清楚。

## 三、配置侧设计:admin 怎么配三档

### 3.1 操作流程(用户视角)

admin 进入"创建 agent"页面,看到的界面是这样:

```
┌──────────────────────────────────────────────┐
│ 高档(适合复杂任务)         [+ 添加模型]      │
│ ─────────────────────────────────────────── │
│   Claude Opus 4.7         [设默认] [删除]   │
│   GPT-4o                  [● 默认] [删除]   │
└──────────────────────────────────────────────┘
┌──────────────────────────────────────────────┐
│ 中档(适合一般任务)         [+ 添加模型]      │
│ ─────────────────────────────────────────── │
│   Claude Sonnet 4.6       [● 默认] [删除]   │
└──────────────────────────────────────────────┘
┌──────────────────────────────────────────────┐
│ 低档(适合简单任务)         [+ 添加模型]      │
│ ─────────────────────────────────────────── │
│   (未配置 — 系统会自动用上面已配的档位)     │
└──────────────────────────────────────────────┘
```

三档独立配置,每档可加多个模型,每档独立设默认。

### 3.2 配置规则(简单清晰的几条)

| 规则 | 含义 | 通俗解释 |
|---|---|---|
| 三档**至少配 1 档** | 三个档位中,只要有一档配了模型就能保存 | 不强求一次配齐三档,可以从重点档位开始,慢慢补 |
| 每档可配**多个**模型 | 同一档可加多个备选模型 | 主用模型挂了自动用备用,业务不中断 |
| 每档**必须**有 1 个默认 | 配了模型的档,要选一个作为该档的"主用" | 系统知道这档优先用哪个 |
| 缺档时**自动借高档** | 没配中档?调用中档时系统借高档来用 | 用户不用担心"漏配"导致没法用,但要注意账单 |

### 3.3 为什么这么设计:四种"严格度"对比

我们考虑过四种约束方案,严格度从严到松:

| 方案 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| 三档**全部强制** | admin 必须配齐三档才能保存 | 行为最严谨,系统永远有所有档可用 | 负担重,只想用一档的人也得凑齐 9 个模型(每档 3 个的话) |
| **至少一档 + 缺档借高档**(我们推荐) | 配一档即可,缺档自动借更高档 | 灵活、有兜底、用户体验好 | 借更高档可能产生意外成本(账单偏贵) |
| 至少一档 + 缺档**报错** | 配一档即可,但缺档时调用直接失败 | 安全(不会偷跑高档) | 用户体验僵硬,容易遇到"找不到模型" |
| **完全自由**(全空也行) | 三档都可以不配,走平台默认 | 极灵活 | 跟"必须有默认"理念冲突,不可控 |

**我们推荐"至少一档 + 缺档借高档"**,理由:

1. **灵活性**:admin 可以从最重要的档位开始配,不强求一次到位
2. **安全兜底**:不会出现"系统找不到模型"的崩溃
3. **可控**:借的方向是**向上**(借更贵的),账单上只会"贵一点",不会"用不了"
4. **风险可缓解**:UI 上明确提示"缺档时会借哪一档",admin 心里有数

> **提示**:借高档意味着"你以为用的是低档,实际跑了高档"。这是一个真实的成本风险,UI 设计时会**显著提示**,让 admin 心里有数。

## 四、运行时设计:系统如何自动选档

### 4.1 这是整个方案的难点

**配置侧只是"摆好菜",真正难的是用户来了"自动点菜"**。

EnClaws 大部分用户**通过飞书机器人发消息**——他们没有 UI、没有按钮、不会写命令,系统只能从消息内容自己判断"这条该用高档还是低档"。

如果判错了:
- 简单问题判到高档 → **账单浪费**
- 复杂问题判到低档 → **回复质量差,用户失望**

### 4.2 主流方案对比(为什么没选别的)

我们把行业内主流的"自动选模型"方案过了一遍,关键对比:

| 方案 | 怎么决定 | 准确率 | 复杂度 | 我们的判断 |
|---|---|---|---|---|
| **A. 完全静态** | 一个 agent 锁一档,运行时不变 | 100% | 极低 | 太死板,完全失去三档配置的价值 |
| **B. UI 让用户手动选** | 对话框旁加"快/平衡/深度"按钮 | 100% | 低 | **机器人场景没 UI,这条路走不通** |
| **C. 三层智能路由(TSR)** | 缓存 + 规则 + 小模型路由,层层判断 | 80% | 高 | 太复杂,误判 5-15%,运维负担重 |
| **D. 关键词规则表** | 按"翻译""总结""重构"等关键词匹配 | 60-70% | 低 | 维护噩梦,新表达层出不穷 |
| **E. 训练专用分类器** | 用 BERT 等小模型做分类 | 85% | 高 | 需要大量标注数据,冷启动期长 |
| **F. F1+F3 组合**(推荐) | 静态多信号 + LLM 自报标签 | 首轮接近 100% / 后续 85%+ | 中低 | 最稳,详见下文 |

### 4.3 候选方案一:TSR(三层语义路由)详解

#### 核心思想

不靠静态规则、不让用户主动选——**用一个独立的"路由层"在每次请求前判断难度**,通过"缓存 → 规则 → 小模型路由"三层依次裁剪降低成本。业界 Martian、OpenRouter 等"模型路由 SaaS"基本都是这个思路。

#### 决策流程

```
用户消息进入
        ↓
第1层:会话缓存复用
       ├─ 同一对话内已经判过难度?直接复用上次结果
       └─ 没缓存 ↓
第2层:强信号规则(代码块/短消息/超长上下文)
       ├─ 含代码块 → 高档
       ├─ 短消息(<50字)+无工具 → 低档
       ├─ 上下文超 10k → 高档
       └─ 都不命中 ↓
第3层:LLM 路由调用
       专门用一个便宜的小模型(Haiku / Qwen Turbo)
       做一次"难度评估"调用,返回:tier + 决策理由
        ↓
按选定档位调用主模型并回复
        ↓
所有决策写入"路由审计表"
(每次决策都记录:走了哪一层、为什么选这档、耗时多少、成本多少)
```

#### 三个场景表现

| 场景 | TSR 流程 | 系统表现 |
|---|---|---|
| 小张问"明天天气怎么样" | 第 2 层短消息规则命中 → 低档 | OK,但仍要先过缓存层、规则层 |
| 总监改 3000 行代码 | 第 2 层代码块规则命中 → 高档 | OK |
| PM 问"分析竞品策略" | 第 2 层不命中 → 第 3 层调用小模型 → 判 high | **多 300-1000ms 延迟 + 1 次小模型 API 成本** |

#### 优点

1. 智能化程度高,理论上能覆盖大部分场景
2. 有专门的审计表,每次决策都有详细理由(满足强审计需求)
3. 用户完全无感(不需要学命令)
4. 业界成熟方案,有现成参考实现

#### 缺点(我们不推荐的关键原因)

1. **每个新对话都要走完三层** —— 缓存只在同一对话内复用,新对话都要重新判
2. **第 3 层 LLM 路由本身有 5-15% 误判率** —— 小模型理解能力有限,会出现"复杂任务被判成低档"的质量事故
3. **每次第 3 层命中都增加 300-1000ms 延迟** —— 用户感知明显,机器人场景体验劣化
4. **持续 API 成本** —— 即使是 Haiku 单价低,大流量场景下也是真金白银
5. **架构复杂** —— 4 个新组件:缓存系统 + 规则表 + 小模型对接 + 审计专表
6. **长期维护负担重** —— 规则要持续维护、阈值要持续调优、小模型要监控、缓存策略要调
7. **配置项爆炸** —— 阈值、规则集、路由模型选择、缓存 TTL,每个租户都可能想调,SaaS 化负担重

#### 工作量估算

AI 辅助开发约 **5-7 天**(纯人工开发约 2-3 周)

### 4.4 候选方案二:F1+F3 双层策略(推荐)

#### 核心思想

**永远不让系统"猜"消息的语义** —— 所有判定都基于**确定的事实**:

- 用户主动行为(命令、之前的选择)
- 客观特征(消息长度、上下文长度)
- 模型自己的判断(主模型回复时顺便给档位建议)

简单说就是:**该让用户做的让用户做、该让 LLM 做的让 LLM 做、其他都用静态规则**,不引入独立的"猜测层"。

#### 决策流程(从上到下,先命中先赢)

```
用户消息进入
        ↓
第1关:消息开头有 "/deep" "/quick" "/normal" 命令吗?
       ├─ 有 → 用对应档,记住,后续轮持续生效
       └─ 没有 ↓
第2关:这个对话之前有没有定过档?
       (用户之前发过命令,或上轮 LLM 给过建议)
       ├─ 有 → 沿用之前定的档
       └─ 没有 ↓
第3关:用户本轮输入很长 或 含代码块/附件 吗?
       (用户本轮净输入 > 2000 token,或含代码块,或含附件)
       注意:只算"用户本轮真正发的内容",
            不含 system prompt、对话历史、工具描述
       ├─ 是 → 用高档(粘贴长文本/代码大概率是复杂任务)
       └─ 否 ↓
第4关:用 admin 给这个渠道(飞书 bot/Slack bot)配的"默认档"
        ↓
按选定档位调用主模型并回复
        ↓
主模型回复的同时,顺便告诉系统"下次这类任务建议用哪档"
        ↓
系统记住这个建议,下轮同对话从第 2 关命中
```

#### 三个场景表现

##### 场景一:小张在飞书私聊问 bot "明天北京天气怎么样"

1. 第 1 关:没有命令
2. 第 2 关:对话首次,没记忆
3. 第 3 关:用户输入很短,无代码块/附件,不触发升档
4. 第 4 关:走"飞书私聊 bot"的渠道默认 = 中档
5. 主模型用中档回答,**且告诉系统"这是简单问答,下次用低档就够了"**
6. 系统记住:这个对话以后用低档
7. 小张接着问"那后天呢"——这次走第 2 关用低档回答,**省钱了**

##### 场景二:技术总监在飞书发"帮我重构这段代码 ```...3000 行...```"

1. 第 1 关:没有命令
2. 第 3 关:用户输入含代码块,且代码本身远超 2000 token
3. 直接命中"代码块 / 长输入 → 高档"规则
4. 用高档处理这段代码,**保证质量**

##### 场景三:产品经理知道这次要做深度分析,主动输入"/deep 帮我分析竞品策略..."

1. 第 1 关:命令命中,用高档
2. 系统记住这个对话用高档,后续轮都是高档
3. 用户始终掌控,**0 误判**

#### 关键机制详解:LLM 如何"自报"下次该用哪档(F3 部分)

这是整个方案中最巧妙的一环,也是为什么不需要"独立路由层"的关键。**让模型自己告诉系统"这类任务该用哪档"**,具体分四步实现:

##### 第 1 步:在 agent 的 system prompt 末尾追加一段"自评提示语"

EnClaws 在每个 agent 的 system prompt 末尾自动追加一段内部提示(用户和 admin 都看不到,只是告诉 LLM 该怎么做):

```
你的回复可以包含一个元数据建议:针对这类任务,系统下次该用哪档模型?
- 若需要多步推理 / 代码改造 / 架构判断 → 建议 high
- 若分析 / 总结 / 计划 / 中等推理 → 建议 mid
- 若直接问答 / 翻译 / 格式化 → 建议 low

请通过工具调用 set_routing_hint(suggested_tier, reason) 输出建议
(可选,不影响主回复)。
```

##### 第 2 步:系统提供一个内置工具 `set_routing_hint`

这是个**特殊工具**,不像查询天气、读文件那种"对外做事"的工具,而是"对内传信号"的工具:

```
工具名:set_routing_hint
参数:
  - suggested_tier:high / mid / low(三选一)
  - reason:简短理由(用于审计,可选)
返回:无可见输出(用户感受不到这一步)
```

##### 第 3 步:系统接收建议并写入对话记忆

LLM 调用 set_routing_hint 后,系统在背后做三件事:

1. 把建议的档位写入这个对话(session)的"粘性记忆"字段
2. 标记来源为 "llm_hint"(区别于用户主动命令 "command")
3. 主回复正常返回给用户(整个建议过程对用户完全不可见)

##### 第 4 步:下一轮请求时使用这个粘性

下一轮同对话的请求进入时,在第 2 关("对话之前定过档没")就会命中这个粘性,**直接使用 LLM 上次建议的档**——完全不需要重新判断。

##### 一个完整的对话流程示例

```
小张:你好
   │
   ▼
首轮:走 channel default = mid 档
mid 档 LLM 回复:"你好,有什么可以帮你的?"
LLM 同时调用:set_routing_hint(low, "简单问候")
系统记下:这个对话以后用 low 档
   │
   ▼
小张:介绍下你能做什么
   │
   ▼
第 2 轮:第 2 关命中粘性 = low 档
low 档 LLM 回复:"我可以帮你..."
LLM 同时调用:set_routing_hint(low, "信息查询")
系统继续:这个对话保持 low 档(省钱)
   │
   ▼
小张:帮我看下这段代码 ```...500 行...```
   │
   ▼
第 3 轮:用户输入有代码块,触发第 3 关 → 升 high 档
high 档 LLM 回复:"这段代码..."
LLM 同时调用:set_routing_hint(high, "代码分析任务")
系统更新:这个对话以后用 high 档
   │
   ▼
小张:然后呢
   │
   ▼
第 4 轮:粘性已是 high 档,继续用 high(深度对话延续)
```

##### 失败容错(LLM 不配合时怎么办)

| 异常情况 | 系统如何处理 |
|---|---|
| LLM 没调 set_routing_hint(忘了或不愿意) | 不更新粘性,下轮按原流程重新判 |
| LLM 给的档位在 agent 上没配 | 走"借更高档"规则解析,不报错 |
| 工具调用解析失败(格式错乱) | 忽略,记日志,主回复正常返回给用户 |

##### 三个关键设计点

1. **优先级保护**:用户主动发的命令(如 `/deep`)的粘性,**不会被 LLM 自报覆盖** —— 用户主动选了什么,系统就一直尊重(直到下一个命令)。
2. **零侵入**:整个机制在 LLM 回复的"幕后"运作,**用户体验上看不到任何额外延迟、额外提示、额外消息**。
3. **可关闭**:租户可在配置里关掉 F3 部分,只跑 F1 静态信号——这就是 §六 决策 4 给您选择的能力。

##### 为什么这种"自报"比独立路由层更靠谱?

|  | 独立路由层(TSR 第 3 层) | LLM 自报(F3) |
|---|---|---|
| 谁判断 | 一个**独立的小模型**(没看到任务上下文) | **主模型自己**(看完任务再判) |
| 时机 | 调用主模型**之前**(只看输入猜) | 调用主模型**完成后**(读完已知任务的全貌) |
| 准确度依据 | 仅用户输入文本 | 任务全貌 + 已生成的回复 + system prompt 全部信息 |
| 成本 | 多 1 次 API 调用 | **0**(顺手做的) |

**主模型读完任务、给完回复后,它最知道"这任务有多难"** —— 让懂行的人来评分,比让一个不懂任务的小模型猜要准得多。

#### 误判率论证

每个判定信号的可信度:

| 判定信号 | 来源 | 错的可能 |
|---|---|---|
| 用户主动打的 `/deep` 等命令 | 用户自己 | **0%** |
| 用户输入 > 2000 token / 含代码块 / 含附件 | 客观特征 | **接近 0%**(粘贴长文本/代码几乎必然复杂) |
| 渠道默认档 | admin 主动配的 | **0%**(主动配的) |
| LLM 自己说"这类任务该用 X 档" | LLM 自评 | 5-15%,**但只影响后续轮**,不影响首轮 |

**首轮判错率几乎为 0**。后续轮即使 LLM 自评失误,**最坏情况是"简单任务被升到高档"**——只是多花点钱,**不会出现"复杂任务用低档敷衍"的质量事故**(因为 LLM 倾向保守,任务难会主动升档)。

#### 优点

1. **首轮误判率接近 0**(只用确定事实判定)
2. **零额外推理成本**(不调独立的小模型)
3. **零额外延迟**(没有第 3 层 LLM 调用)
4. **架构极简**(只是一段路由代码 + 一个新工具 + 几个新字段)
5. **多轮对话自适应**(LLM 自报标签让后续轮越来越准)
6. **不堵升级路径**(未来发现不够,可在上面叠加 TSR 的第 3 层)

#### 缺点

1. **首轮没有"语义理解"**(只能按规则和渠道默认走,首轮可能不是最优档)
2. **依赖 LLM 诚实输出建议标签**(需要 prompt 工程保证,有少量失败概率)
3. **斜杠命令的覆盖率有限**(大部分普通用户不会用,主要给"懂的人")
4. **不适合"单轮就结束"的场景**(但 EnClaws 是机器人多轮主导,这点影响不大)

#### 工作量估算

AI 辅助开发约 **2-3 天**(纯人工开发约 6-7 天)

### 4.5 两方案直接对比

| 对比维度 | TSR(三层语义路由) | F1+F3(双层策略) |
|---|---|---|
| 决策依据 | 缓存 + 规则 + 小模型 LLM 路由 | 用户行为 + 客观特征 + 主模型自评 |
| 首轮误判率 | 5-15%(第 3 层小模型误判) | **接近 0** |
| 后续轮误判率 | 5-15%(每轮独立判) | 5-15%(LLM 自评,**只浪费钱不影响质量**) |
| 单次请求延迟 | +300-1000ms(第 3 层命中时) | **几乎 0** |
| 单次请求成本 | 多 1 次小模型 API 调用 | **0** |
| 新增组件 | 缓存系统 + 规则表 + 小模型对接 + 审计专表(4 个) | 路由代码 + 1 个新工具 + 几个字段 |
| 实现周期(AI 辅助) | 5-7 天 | **2-3 天** |
| 长期维护 | 复杂(多组件、多配置) | 简单(规则 + 提示词) |
| 智能化程度 | 高 | 中(但够用) |
| 用户感知 | 完全无感(纯后台) | 命令可选,粘性自适应 |
| 适合场景 | 单轮多 / 对延迟不敏感 / 有 MLOps 团队 | 多轮主导(机器人) / 延迟敏感 / 运维资源有限 |

#### 推荐结论

**EnClaws 是机器人主导场景**(飞书 / Slack / 钉钉等多轮对话),admin 资源有限,所以推荐 **F1+F3**:

> "简单的事先做好,智能的事交给 LLM 自己"——不堆架构,但用对地方。

**TSR 留作 Phase 2 候选**:如果未来发现 F1+F3 准确率确实不够,且团队有资源支撑 MLOps,可以在 F1+F3 基础上**叠加** TSR 的第 3 层(LLM 路由)做增强——而不需要推倒重做。这是 F1+F3 相对 TSR 的另一个隐形优势:**升级路径开放**。

## 七、风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|---|---|---|---|
| LLM 自报判错(把简单任务升到高档) | 中 | 成本浪费 | 用现有用量表的 tier 维度做月度审计,超阈值告警 |
| 用户不知道斜杠命令 | 高 | 部分用户体验损失 | 在 channel 介绍页加说明,bot 首次对话主动告知用法 |
| 借高档导致账单意外 | 中 | 财务问题 | UI 配置时显著提示,admin 心里有数 |
| 用户输入阈值(2000 token)定不准 | 低 | 边缘场景判错 | 阈值做成租户可配,根据实际数据调整 |
| LLM 在 system prompt 里加路由提示后,回复质量受影响 | 低 | 用户体验问题 | 提示语放最末,与主任务隔离;上线前 A/B 测试验证 |

## 八、推荐组合一览(若全部按推荐选)

如果以上 5 个决策全部按推荐选,最终方案是:

> **配置侧**:三档至少配一档,缺档借高档,旧数据迁移到高档。
> **运行时**:按"用户命令优先 → 对话粘性 → 长文本升档 → 渠道默认 → LLM 自报反馈"的顺序自动选档。

这是一个**配置灵活、运行可预测、误判率极低、实现成本可控**的折中方案,AI 辅助开发约 1 周内可完整落地。
