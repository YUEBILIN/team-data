# AI 客服 S1 上线 · dev 环境完成部署与验证

## 为什么这一版先求"老实"，不求"能说"

你最近一次被 AI 客服气到是什么时候？

大概率不是因为它答得不够华丽，而是它**答得很华丽但是错的**。你问 A，它答 B；它说的某个"配置路径"你在自己产品里根本找不到；它说的"定价方案"和官网对不上。

这种体验在 C 端是笑话，在 B 端是事故。客户看到一句错的信息，代价远大于看到"我不知道"。

所以 S1 的第一目标不是"能说会道"，而是**每一句话都能追回到知识库原文**，超出范围就直接转人工，不装懂。

---

## S1 做到了什么

**回答严格限定在 KB 范围内。** 客户问的每个具体事实（产品名、数字、步骤、方式名），KB 里没有原文就不说。新上线的 prompt 在这点上做了加固：原文引用、脑补词黑名单、回复前自检三道关。实测问"EC 有哪些安装方式"，AI 严格按 md 里的"方式一/二/三/四"列出，没擅自改成 Docker、Homebrew、apt 这类它"以为 B 端客户想听"的常见词。

**超出范围 = 飞书通知负责人跟进。** 客户问"你们 CEO 是谁"、"年费多少"这类 KB 没写的，AI 不编，直接回"根据现有资料没有相关信息，已通知负责人为您跟进"，同时飞书群实时弹出提醒。

**后台三件事可控**：
- KB 维护（上传 md，自动分块索引；当前仅支持上传 + 删除，不支持 UI 编辑）
- 通知群与频率配置
- 完整对话审阅（客户问了什么、AI 答了什么、何时转人工）

## 两种落地场景

所有 B 端 SaaS 产品的 AI 客服需求其实分两类，差别在"客户是谁"。

**内部自用型**——SaaS 厂商自己就是唯一主体，AI 代表这个主体答自己产品的问题。配置简单：一个租户 = 一个 KB = 一个客服 AI。

**多租户嵌入型**——SaaS 卖给 N 个企业客户，每个客户有自己的产品要配 AI 客服。每租户独立 KB、独立配置、独立 widget，客户把 widget 代码嵌到自己网页上，AI 以该租户的身份回答。

EC 当前走的是**内部自用型**——管理后台启动时自动把"最早注册的非系统租户"的 tenant-id 注入到页面 meta 标签，客服气泡自动出现在所有进入 EC 后台的用户眼前，零代码接入。

普通租户走的是**嵌入型**——在他们自己的 CS setup 页面会看到一段 `<cs-widget tenant-id="xxx">` 代码片段，贴到他们产品网页上就跑。

## 你可能想问的几件事

**"KB 里没写的，AI 真的一个字都不编吗？"**

严格意义上无法 100% 保证（模型的训练 prior 永远在那），但我们把这个概率按到了很低：系统 prompt 明确要求每个事实必须能在 KB 原文找到、列举题必须原文引用、回复前自检一遍有没有冒出脑补词。实测 dev 环境答"EC 有哪些安装方式"，完全按 md 原文走，没有 Docker/Homebrew 之类脑补——这在老 prompt 下会出现。

**"客户问的问题答不出，体验会不会很差？"**

不会比 AI 编一个错答案差。现在的降级话术是"根据现有资料没有相关信息，我会通知负责人为您跟进"——客户看到的是"正在处理"，不是"我被糊弄了"。负责人收到飞书通知走人工，正常 SLA 内回复。

**"不同租户的知识库会串吗？"**

当前向量库全局共享（`~/.enclaws/memory/*.sqlite`），KB md 文件按租户目录隔离但索引写进同一个 sqlite。S1 跑得通但不是终态——已在部署文档里标了未来方向：向量库路径隔离到 `~/.enclaws/tenants/{tenantId}/customer-service/vectors/`，后续配合后台 UI 统一配置 LLM/embedding 一起改。

## S2-S4 路线图

**S2：置信度打分**——AI 自评答得多稳，低于阈值主动触发转人工。不靠关键词匹配，也不靠人工审。目前代码框架已有（`confidencePreset`），尚未接入 runner。

**S3：人工接管（三角色对话）**——客户 / AI / 人工 三方接力。AI 不稳 → 人工同一对话里无感接手 → 对话结束后人工的回答可沉淀为新 KB 片段。

**S4：知识库半自动迭代**——AI 根据未覆盖问题的聚类自动草拟新 KB 条目（带引用证据），运营审核后入库。目标是让 KB "使用得越多越完整"，而不是"运营越写越累"。

这三步走完，客服 AI 从"受约束的回答器"变成"会沉淀的团队成员"。

## 今日过程补记

dev 环境折腾了大半天，根因是 S1 从合并 main 到今天才第一次被端到端真实跑通——中间大量"假设一致其实没对齐"的环境问题浮出水面。过程中顺手把 4 个埋得较深的 bug 全修了，已全部合并 main：

- **`62d91467` sys-config**：`loadAndActivateSysConfig` 在启动 race 时 snapshot 为 null，文件里配的 `agents.defaults.*` 被静默丢弃——`memorySearch` "配了但不生效"的根因。
- **`0e568712` CS 回复 strip trailer**：Qwen 等模型在 `promptMode:"none"` 下仍吐一行 `> Skills used: ...`，客户场景不该看到，regex 剥掉。
- **`2b3757f9` CS 默认 prompt 防幻觉加固**：新 CS 租户默认 prompt 新增 4 节核心约束（严格原文引用、列举题原文引、章节边界、回复前自检 + 脑补词黑名单）。
- **`c96b62a5` pi-runner tools 空数组**：`disableTools=true` 时硬塞 `tools:[]`，严 provider（Qwen/DashScope/Gemini、部分 DeepSeek）400 拒收——`undefined` 替换 `[]` 治本。顺手提了 issue #30 建议上游下沉。

这 4 个都是 CS 路径下才触发的场景 bug，线上环境不受影响。技术细节问我或查 commit log。
