## 一、合并方案总体结构

三层职责：

| 层级            | 职责                                                  | 当前状态                                | 目标状态              |
| ------------- | --------------------------------------------------- | ----------------------------------- | ----------------- |
| **EC 底层**     | 通用能力基础设施（agent/channel/session/memory/audit/health） | 部分能力复用，部分 CS 独立实现                   | 所有通用能力下沉，CS 不再自建  |
| **Agent 管理页** | 所有 agent 的标准化配置入口（含 CS agent）                       | 7 个标签页（概览/人设/频道/工具/技能/定时任务/知识库）     | 结构不变，内容由合并方案扩展    |
| **AI 客服页**    | 业务 composer：选 agent + 配通知 + 发布 widget               | cs-setup 里独立管理 prompt/知识库/飞书凭证/嵌入代码 | 瘦身到 5 框，原独立配置全归底层 |

核心思想来自 **018 ken reply**（CS 模块尽量薄）+ **020 §8.2**（CS 页只保留业务专属配置）+ **菜单规划 §1.2**（单一管理入口原则）三条脉络的交汇。

## 三、客服模块瘦身

S1 结构（独立栈）：

```
cs-setup      飞书凭证 + prompt 编辑 + 嵌入代码
cs-knowledge  知识库独立管理
cs-sessions   独立会话列表
```

合并后（业务 composer）：

```
cs-setup      5 个框：
              A. 默认 CS Agent 下拉（选已建 agent）
              B. 通知渠道选择（复用已配置的 Channels，不再输 App ID/Secret）
              C. 通知群 chat-id + 通知频率策略（CS 业务配置，不归 Channel）
              D. Widget 部署（多渠道标签 + 嵌入代码 + 外观）
              E. 部署就绪检查（Agent 绑定 / 知识库非空 / Embedding 可用 / 凭据有效）

cs-knowledge  只读查看（显示当前 CS agent 绑定的 KB 摘要 + 最近更新时间）
              编辑/上传/删除统一跳转 Agent 管理 → 知识库 tab
cs-sessions   保留独立会话列表（上层业务诉求，需单独入口）
(移除) prompt 编辑器  → 并入 Agent → 人设与规范 tab
```

保留 cs-knowledge 只读查看的理由：部署就绪检查本就包含"知识库非空"，运营者在同页看到 KB 摘要更顺手；管理入口仍收敛在 Agent → 知识库 tab，不破坏单一入口原则。保留 cs-sessions 的理由：业务侧需要独立会话列表入口（而非工作台混合视图）。

这和菜单规划 proposal §5 是自洽的，不冲突。

### Prompt 管理策略：模板库方式，放弃 runtime overlay

**结论**：废弃 S1 的 4 条 runtime restrictions overlay（`disableSkills / strictKnowledgeBase / disableMarkdown / hideInternals`），改为 **Agent → 人设与规范 tab 新增「从模板载入」**——下拉内置"客服 Agent（防幻觉优化）"等模板，载入后填充 prompt 框可继续编辑；`disableSkills` 迁到 Agent → 技能 tab 勾选。

**推演**：overlay 让用户看到的 prompt ≠ 模型实际收到的，破坏"所见即所得"、归因困难、不可演进。模板库让 content 层 100% 用户掌控；format 层（如 `> Skills used:` trailer 清洗）保留在 cs-agent-runner 输出后处理。

## 四、S2 / S3 / S4 内容在新架构下的归位

**合并后 S2/S3/S4 的内容不废弃，而是重新归到正确的层。**

### S2（流式输出 / 置信度门控 / 澄清追问 / 登录访客身份识别）

| 原 / 新增 S2 功能     | 合并后归属层                       | 理由                                                                                                                                                                                                                              |
| ---------------- | ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 流式输出             | **EC 底层 agent 运行时**          | 所有 agent 都应该支持流式，不是 CS 专属                                                                                                                                                                                                       |
| 置信度门控            | **Agent 运行时参数**              | confidence 作为 message 元数据（改造 #6）；门控阈值是 agent 的运行配置                                                                                                                                                                              |
| 澄清追问             | **Agent 行为规则**               | agent prompt 驱动；agent 收到模糊问题主动追问是通用能力                                                                                                                                                                                           |
| **登录访客身份识别**（新增） | **底层身份识别服务的 CS 消费方**（改造 #10） | 升级后的理解：这不是 CS 专属能力，而是底层"跨来源身份识别"（改造 #10）的一个消费场景。CS 作为消费方需要做的是：(a) widget connect 时把 loadAuth 的 company+email 作为 identity anchor 上报给底层；(b) session 显示、交互审计、飞书通知卡片的「客户」行都从 identity 服务读取"企业名+邮箱"而非匿名 visitor_id；(c) 未来人工接管时统一显示身份 |

本地已有的 S2 WIP 代码大部分可以沿用，但部分需要重构（流式迁出 CS、confidence 移到 message 元数据层）。新增的登录访客识别需要在 widget connect/send 时检查 loadAuth 注入租户名和邮箱。

### S3（老板通过飞书卡片回复 / HUMAN_ACTIVE / 标签匹配）

| 原 S3 功能      | 合并后归属层                              |
| ------------ | ----------------------------------- |
| 飞书卡片回复       | **飞书 channel adapter 的 inbound 扩展** |
| HUMAN_ACTIVE | **底层会话状态机**（改造 #5）                  |
| 标签匹配         | **底层会话 tags 的应用**（改造 #7）            |

### S4（反馈收集 / 日报 / Badcase 运营闭环）

| 原 S4 功能    | 合并后归属层                                           |
| ---------- | ------------------------------------------------ |
| 反馈收集       | **底层消息元数据 feedback_type**（改造 #6）+ widget UI 反馈按钮 |
| 日报         | **底层定时任务**（agent 的 cron tab）                     |
| Badcase 运营 | **审计中心 + Agent 管理的扩展**（改造 #8）                    |

### 2. 合并方案的实施责任

### 3. 迭代顺序

- **原计划**：S2+S3+S4 依次迭代
- **建议新顺序**：合并方案 + S2 功能一起迭代；S2 稳定后再做 S3+S4
- 理由：S3/S4 重度依赖底层原语（多方会话/状态机/消息元数据），没这些原语做 S3/S4 等于重复实现
- 风险：合并方案工作量大，上线时间可能拉长；需要权衡"一次到位"和"快速迭代"

### 4. 统一身份识别服务（改造 #10）范围如何定

这是本次讨论新浮现的底层刚需（跨 IM 渠道识别同一人），建议作为独立大项处理。范围有两档：

- **档 1（仅自动解析）**：底层存 anchor 表 + 按 email 等强标识自动关联；不做人工 UI
  - 适用场景：CS 登录访客识别、同邮箱跨渠道自动合并
  - 工作量：小（1-2 周）
- **档 2（自动解析 + 人工关联/纠正 UI）**：档 1 + 租户管理员 UI 看候选、确认关联、拆分误关联、operation audit
  - 适用场景：员工多 IM 渠道识别、客户经理对接外部客户
  - 工作量：中（3-4 周）

**我倾向档 1（小步快跑）**：先把底层 anchor 表 + 自动关联跑通，覆盖 CS 登录访客这一确定性场景；人工关联 UI 待员工多 IM 识别、客户经理对接这两个场景需求明确后再做档 2。请团队确认是否同意。
