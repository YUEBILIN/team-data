# OPC S2 产品规划：商业模式 / 合作模式锁定及对产品架构的影响

现状：MVP 已上线（opc-dev.enclaws.com，5-3 部署，蒋鲲方已组织过试用反馈），S2 计划进入真实用户周期。

关键判断：商业模式 / 合作模式不是商务议题，而是 S2 产品架构的前置条件。MVP 阶段 schema / UI 都是 mock，没有商业语义；S2 切真实用户后，schema 和 UI 必须承载真实商业含义——这两个判断不定，S2 会做错。

两条因果链理由：

1. S2 schema 决策依赖商业模式。走"订阅 + 抽佣"必须埋 metering（token 用量 / agent 调用数 / 项目数 log）+ introducer code（渠道分润字段）；走"项目制"全部不需要、schema 极简；走"联合运营"还要加分润快照。这些做了后期改造代价大、不做后期数据缺失。

2. S2 用户感知层依赖合作模式。走"平台 + 行业渠道伙伴"用户注册要支持渠道码 + 数科院作为渠道伙伴克制露出；走"联合运营"双 logo + 客户支持入口分流到数科院；走"项目制"无渠道露出、产品定位"数科院私有版"。这些会影响 S2 哪些菜单要做、哪些不能有。

兜底姿态：如果你认为 S2 不需要预承载商业 / 合作语义（schema 极简 + UI 中性，推迟到 S3 重做），这两个讨论可以推后，OPC 直接按 mock 状态升级——这个判断同样请拍。

## 一、术语澄清

**marketplace（双边撮合平台）**：撮合链主（发包方）和 OPC 个体（接单方）的真实交易，平台抽佣 1-3%（淘宝 / Upwork / Fiverr 同模式）。和纯 SaaS 的核心区别：marketplace 收入与平台促成的真实交易规模成正比，必须接真实支付通道。

**AI 重度**：AI 是产品核心价值（不是辅助）。OPC 4 个 AI 数字员工每用户高频触发 LLM 调用，token 成本占运营成本 30-60%。和"AI 加成"产品（OA + AI 摘要等，token 占成本 5-10%）的核心差异：AI 重度型纯订阅会被 token 成本压成负 margin，必须有按量超额或抽佣对冲。

---

## 二、平台商业模式（5 候选 × 4 维度）

| 模式 | 一句话 | 优势 | 致命问题 |
|---|---|---|---|
| ① 纯订阅 | Free + VIP + 商业版分级 | 现金流稳；市场已教育 | AI 重度产品 token 成本无法封顶在订阅里；少数高频用户单月 token 成本可能超过其订阅费（"越用越亏"），平均 margin 被拉低 |
| ② 纯按量 | 基础免费 + AI 按 token 收费 | 低门槛；公平 | 账单不可预测让企业紧张；早期没"包月安心感" |
| ③ 订阅 + 按量超额 | 订阅含额度，超出按量 | 兼顾稳定与公平；OpenAI / Notion AI / Vercel 都用 | 定价复杂 |
| ④ 平台抽佣 | 抽真实交易额 1-3% | 与平台价值正相关；客户成功才付 | 必须接真实支付（S4+）；大客户想绕 |
| ⑤ 行业打包年付 | 行业生态包年签 | 大单 | 销售周期长；早期不合适 |

**最优**：③+④ 双引擎（试运营免费+token 限量 → 正式期订阅+按量超额 → marketplace 跑起来叠加 1-3% 抽佣）

**次优**：① 纯订阅

**理由**：OPC 三重身份（SaaS+marketplace+AI 重度）只有双引擎能 cover 全部价值。

---

## 三、与数科院合作模式（5 候选 × 3 维度）

| 模式 | 一句话 | 核心代价 | 适配判断（基于 §二 推荐 ③④） |
|---|---|---|---|
| ① 项目制 | 数科院 = 客户 | 交付后无粘性 | ✗ 与抽佣模式冲突 |
| ② 联合运营 | 双方对用户负责、分润 | 决策权 50/50 | ⚠ 备选 |
| ③ 平台 + 行业渠道伙伴 | 数科院 = 首个 reseller | 数科院须接受"降级"为渠道 | ✓ 推荐 |
| ④ 白牌输出 | OPC 挂数科院品牌 | 完全失去用户触达 | ✗ 与抽佣冲突 |
| ⑤ 战略入股 | 数科院投资拿股份 | 谈判周期 6-9 月 | ⏳ 跑出 traction 后 |

**最优**：③ 平台 + 行业渠道伙伴（数科院 = 首个行业 reseller，按介绍 / 流水抽佣）

**次优**：② 联合运营

**理由**：渠道伙伴保平台主导权 + 多渠道可扩展；联合运营深度绑定换持续投入，但产品决策权让渡一半。

---

## 四、商业 / 合作模式对 S2 产品架构的具体影响（核心议题）

按"最优组合（双引擎+渠道伙伴）" vs "兜底假设（不预承载，按 mock 升级）"的 S2 差异：

| S2 必做项 | 最优组合假设 | 兜底假设 |
|---|---|---|
| schema 字段 | 埋 metering + introducer code + settlement_status 预留 | 全部不预埋 |
| 必做菜单 | "用量统计"占位页 + 注册页支持渠道码 | 无 |
| 必做 UI | 头部"试运营+限量"标识 + 注册落地页可选渠道路径 | 无 |
| 暂不做但留接口 | 订阅 / 升级 UI 隐藏但数据层留位 | 同 mock 状态 |

**取舍**：
- 最优组合：S2 增加约 5-8 项 schema / UI 工作量，避免 S3 数据回填 + UI 大改
- 兜底假设：S2 极简，但模式定下来后 S3 重做（schema migration + UI 重构）

## 五、S2 AI 能力定调（与 ken 5-6 路线对齐）

S2 不集成新 AI 能力，沿用 EC 现有：LLM 用平台通用模型 + 4 个默认 agent 自动配置 + 复用 EC 数据对象。不可用部分先降级或不做（IM 多通道 / AI 客服管理 / 文档总结 / 复杂 agent 自定义）。等公司 ai-gateway 长出 + EC 定位明朗后再迁移。

**引用**：dengke 5-6 在 pivot matter「技术架构选型讨论：Pivot / OPC / EnClaw 的关系与下一步底层 AI 基建」002 帖原话「先以 Pivot 为主战场长能力，再沉淀公司级 AI 枢纽」+ ken 早会「EC 改造暂停、重新讨论 EC 架构与商业化定位」。OPC 不预先做新 AI，避免与公司路线分叉。
