# OPC项目包装设计方案

## 一、我的意见：先做一个场景摸通

> **背景**：这个方案不是技术方案，是从"老板要做什么事"倒推回来的产品方案。"AI 公司 / 多岗位员工 / Skill 市场"这类路径的共同问题是——仍然要老板去懂模型、Agent、Skill，那是程序员思维。老板只关心一件事：我要做的事，做完了没。所以 Phase 1 只选一个场景（一个人的内容工作室），把从选题到发布回流的这条链路端到端做完——让老板打开就看见"今天该做的事都做了"，而不是看见一堆可配置的 Agent。

### 1.1 策略判断：先不做大而全

| 行业信号 | 含义 |
|---|---|
| Paperclip / HeyGen / Workforce 这类"AI 公司编排"普遍没跑出量 | 多岗位承诺重，任一员工掉链就破 |
| 跑出规模的 toC/toSMB AI 都"小而确定"（Midjourney / Notion AI） | 老板只为一眼看懂的价值付费 |
| IM bot + App 市场已验证天花板（Slack / 钉钉 10 年没跑成主流） | 不是"一个平台装一堆 bot" |

**结论**：Phase 1 不铺 8 岗位，只摸一个场景——把一件事做完。

### 1.2 场景锁定 + 画像锚点

**选「一个人的内容工作室」**：高频刚需、闭环短（选题→草稿→分发→回流几分钟）、产出可量化、EC 底座最贴合。对标叙事 =「一个人也能开一家内容公司」，不是「AI 员工公司」。

**画像 A · 小林**

| 维度 | 描述 |
|---|---|
| 身份 | 28 岁全职自媒体（前互联网运营） |
| 主营 | 公众号 + 小红书，垂类「年轻人理财科普」 |
| 体量 / 收入 | 5 万 + 3 万粉 · 月入 2–4 万（接广 60% · 知识付费 25% · 分成 15%） |
| 节奏 | 早起刷消息 → 上午写稿 → 下午剪排 → 深夜回私信 |
| 工具栈 | 剪映 + 秀米 + 创客贴 + 讯飞 + ChatGPT + 若干小众 AI——切换本身就是负担 |

**行业结构性痛点（小林只是典型）：**

| # | 痛点 | 本质 |
|---|---|---|
| 1 | 选题枯竭 | 单人无法持续扫描热点/竞品/算法偏好 |
| 2 | 创作重负 | 从观点到初稿全靠自己 |
| 3 | 多平台搬运 | 公众号 ≠ 小红书 ≠ 视频号，一稿多改 |
| 4 | 私信雪崩 | 粉丝咨询/合作询盘/普通互动/spam 混在一起 |
| 5 | 接广失血 | 没经纪人，报价跟进全靠脑子 |
| 6 | 数据盲区 | 不知道哪条内容真赚钱 |
| 7 | 合规红线 | 金融/医疗等垂类踩线，偶尔被限流不知原因 |

**不做**：CRM / 法务 / 客户项目管理 / 团队协同——她就一个人。

**关键洞察**：小林不缺 AI 工具，缺的是**一支覆盖「选题→创作→分发→互动→变现→度量」完整闭环、按内容公司真实节奏每天自己跑起来的固定班底**。

### 1.3 产品机制：选公司模板 → 自动生成班底 → 每日排班

跟 001 的"挑岗位 → 入职 → 配工位"路径有本质差别：**用户不挑岗位，只选公司类型；岗位由公司模板决定，每日工作由排班决定。**

```
① 登录
    ↓
② 选「你开的是哪种公司」
   · 内容工作室   ← 本期唯一选项
   · 电商工作室   （灰）
   · 法律咨询     （灰）
   · 教培工作室   （灰）
    ↓
③ 系统直接生成 6 个固定员工 + 每日排班表
   （不问诊、不让用户挑、不让用户排，第一版即开即用）
    ↓
④ 绑主要分发平台 + 主要 IM（小红书/公众号 + 微信）
    ↓
⑤ 进主场「今日排班表」—— 看见今天每个员工几点干什么
    ↓
⑥ 老板的日常动作只有 4 个：
   · 早上看排班 · 给灵感 · 审稿 · 看晚报
```

**为什么"选公司模板"比"问诊推荐"好：**

| 维度 | 问诊推荐 | 选公司模板 |
|---|---|---|
| 决策成本 | 用户回答 N 题、看推荐结果、再决定要不要 | 一次点击 |
| 一致性 | 同一画像问出 N 个不同结果 | 同一公司类型固定输出 |
| 可解释 | 「为什么推荐这俩？」难答 | 「内容公司就这 6 个岗位」天然合理 |
| 升级路径 | 问诊逻辑越来越复杂 | 加一个公司模板就行 |
| 对小林的契合度 | 她要的是"开门就能用" | 完全匹配 |

### 1.4 内容工作室固定班底（6 员工）

按内容公司真实业务链条拆，每个员工锚定一个痛点——不拼凑、不重叠。

| 员工 | 一句话 | 解决痛点 | 主要 workspace 读 → 写 |
|---|---|---|---|
| **选题策划官** | 每日扫行业热点 + 竞品爆款 + 平台算法偏好，产出今日选题清单 | ① 选题枯竭 · ⑦ 合规红线（提前规避） | 外部热点 / 竞品 / `published/` 历史复盘 → `topics/` |
| **内容创作官** | 基于选题 + 老板灵感产出可发布初稿（含文案 + 配图建议） | ② 创作重负 | `topics/` `inspiration/` `assets/` → `drafts/` |
| **分发编辑** | 把审过的初稿按各平台调性适配 + 排期 + 到点发布 | ③ 多平台搬运 | `drafts/`(status=approved) → `published/` |
| **社区管家** | 把 IM 私信 + 评论按「粉丝咨询 / 合作询盘 / 普通互动 / spam」分类归档并起草回复 | ④ 私信雪崩 | IM 原始流 → `messages/inbox/` `messages/replies/` `messages/leads/` |
| **商务经理** | 从社区管家分出来的"合作询盘"起步，维护询盘 → 谈判 → 合同 → 结款全流程 | ⑤ 接广失血 | `messages/leads/` → `deals/{inquiries,negotiating,signed,completed}/` |
| **财务助理** | 归档每日收入（接广/知识付费/分成）+ 对应到具体内容，做 ROI 归因、生成日报/月报 | ⑥ 数据盲区 | `published/` `analytics/` `deals/completed/` `income/raw/` → `income/{daily,monthly}/` |

**设计说明**：

- **6 个覆盖"内容公司最小闭环"**：选题 → 创作 → 分发（生产链）· 社区 → 商务（变现链）· 财务（度量链）。砍掉任何一个都会有明显断点
- **为什么选题和商务独立出来**：
  - 选题枯竭是所有自媒体的通病，不能当作"创作官顺手的事"。独立一个员工，每天固定时间产出选题清单，才有仪式感和复盘基础
  - 接广是小林收入的 60%，却是她当前最没有系统管理的一块。询盘如果没有独立员工负责"接 → 记 → 跟"，就一定会漏。商务经理的价值高于创作/分发那几个（因为这是真金白银）
- **员工之间的接力靠文件状态**（`status: draft` → `approved` → `published` / 询盘 `new` → `negotiating` → `signed` → `completed`），不需要事件总线
- 每个员工的 system prompt 出厂内置读写路径——这是「工作契约」的最小闭环实现
- **固定 6 个，Phase 1 不允许用户增删**——降低选择负担、避免老板乱配出非闭环团队

### 1.5 岗位 × Skill 矩阵 + EC 资产盘点

6 个员工在产品层面对用户是"人"，内部实现是 **Agent（LLM + prompt + workspace 契约）+ 一组 Skill**。每个岗位能不能每天自主跑起来，完全取决于 Skill 是否就绪。下面盘一遍。

**每岗位必需 Skill + EC 现状盘点**（❓= 需要刘昱/Captain 确认现有 60+ Skill 资产；❌= 本期必须新建）：

| 员工 | 必需 Skill | EC 现状 | 本期必须新建 |
|---|---|---|---|
| 选题策划官 | 各平台热榜抓取 / 竞品账号扫描 / 合规敏感词检查 / `workspace.*` | 热榜❓ · 竞品❓ · 合规❌ · workspace❓ | 合规词库 + 竞品扫描（至少支持小红书/公众号账号拉新内容） |
| 内容创作官 | LLM 长文创作 / 配图 prompt 生成 / `workspace.*` | LLM✅ · 配图 prompt ❌（LLM 即席可做）· workspace❓ | 无强新建项（都是 LLM prompt） |
| 分发编辑 | 公众号发布（全自动）/ 小红书等平台发布包生成（半自动）/ 多平台格式适配 / Cron 排期 | 公众号❓（有官方 API，可全自动）· 小红书❌（**无官方 API 且逆向有封号风险，不走自动发布**）· 适配器❌ · Cron✅ | **公众号 publish（自动）+ 发布包生成器（半自动）+ 多平台格式适配器**（见下文"发布策略分档"） |
| 社区管家 | IM 消息拉取 / 平台评论拉取 / 4 档消息分类 / IM 回复发送 | IM✅（41+ 渠道是 EC 现成优势）· 平台评论❓ · 分类❌ | 4 档分类器（粉丝咨询/合作询盘/普通互动/spam） |
| 商务经理 | 询盘识别（从社区管家分类中）/ 报价建议生成 / 合作历史查询 / `deals/*` 状态机 | 全部❌ | 询盘识别 + 报价建议 + deals 状态机（状态机本质是 workspace 文件约定，不算重 Skill） |
| 财务助理 | 平台分析数据拉取 / 银行/支付流水导入 / ROI 归因 / 报表生成 | 平台分析❓ · 流水导入❌ · 归因❌ · 报表（LLM 即可） | 各平台分析数据拉取（至少小红书+公众号）+ ROI 归因逻辑 |

**结论：**

1. **Phase 1 无论 EC 现状如何都必须亲手做的 4 个 Skill**（产品差异化锚点）：
   - `platform.format.adapt` + `platform.publish.pack` — 一稿转多平台"翻译官" +（非公众号平台的）发布包生成器，内容工作室最核心的价值
   - `message.classify` — 社区管家的价值全在这 4 档分流上
   - `deal.inquiry.detect` + `deal.quote` — 商务经理的价值锚点，接广 60% 收入的护城河
   - `roi.attribute` — 财务助理把"每篇内容赚了多少"归因到具体 `published/` 条目

2. **等盘点才能定的关键未知**：
   - `workspace.*`（读写/列/查询租户沙箱文件）— 所有员工的基础，没有就整条路跑不通
   - `platform.wechat.publish` / `platform.xiaohongshu.publish` — 分发编辑的命脉；EC 60+ Skill 里**具体有哪些平台的发布/拉评/拉数据能力**必须盘一次清单
   - `platform.hotlist.*` — 选题策划官靠这个输出

3. **EC 现有资产能直接复用的两块**：
   - **41+ IM 渠道**：社区管家拉消息 + 发回复的底座，这是 001 对比 Paperclip 的核心优势之一
   - **Cron 系统**：排班表的执行底座，现成可用

4. **产品层 vs 实现层分层**：产品层对小林**不暴露** Agent/Skill 概念，她只看见 6 个员工；内部实现上"员工" = Agent + 一组 Skill——**这个分层在 Phase 1 就必须存在**，只是不是用户概念。

5. **对 Captain "LLM 解析层位置错配" 风险的回应**：6 员工 × 9 时刻的排班是**确定性触发**（Cron 到点 → 唯一员工 + 有限 Skill 组），不需要让 LLM 判断"现在该谁干"。这天然避开了多 Agent 自由协作中的调度错配问题。

**发布策略分档：全自动 vs 半自动**

"分发编辑全自动把稿子发到各平台"这条路一半走不通——公众号有官方 API 可以真自动，但**小红书/视频号/抖音/知乎没有开放 API 给个人账号，第三方逆向方案封号风险极高**。对小林来说账号就是她的命脉，产品不能替她拿这个风险。

所以分发 Skill 必须**在产品层分档**，不追求"全平台全自动"：

| 档位 | 平台 | 分发编辑做什么 | 老板做什么 | 风险 |
|---|---|---|---|---|
| **全自动** | 公众号（官方 API） | 排期 → Cron 到点真发 | 无 | 低（官方限频守规则即可） |
| **半自动** | 小红书 / 视频号 / 抖音 / 知乎 | 生成"发布包"（适配后标题+正文+标签+封面图打包）→ 推到主场"待你点发"区 → 一键复制全文 + 一键打开 App | 打开 App 粘贴 + 贴图 + 点发送（≈ 30 秒） | 零（账号安全由老板掌控） |
| 完全手动 | 超出以上 | 不做 | 不涉及 | — |

**为什么这样分档是对的：**

- **对小林**：账号安全 > 不用动手。她月入 2-4 万全靠这几个账号
- **对产品**：从 6-8 小时手工 → 30 秒点一下，价值 99% 已经交付。真正耗时大头（选题/写稿/适配/回私信/对账）依然是全自动
- **对 EC**：不用去填"N 个平台 publish"的无底洞，也不承诺做不到的事
- **对合规叙事**：半自动 = 用户本人发布，平台合规与 EC 天然解耦

**对排班的影响**：11:00 分发编辑正常做适配 + 排期；14:00 的 Cron 触发**按平台类型分叉**——公众号真发，其他平台只把"发布包"推到主场"待你点发"区等老板点。

### 1.6 每日排班表（Cron 编排）

**机制**：每个公司模板自带一份默认排班，落到 EC 的 Cron 系统。用户可以禁用某条、改时间，但**不能自由编排任务——编排是产品出厂能力，不是用户能力**。

**内容工作室默认排班：**

| 时刻 | 员工 | 动作 | 触发后产出 |
|---|---|---|---|
| 06:30 | 选题策划官 | 扫昨晚至今晨行业热点 + 竞品爆款 + 平台限流预警 | `topics/{date}.md`（3-5 个候选选题，含合规注记） |
| 07:30 | 社区管家 | 拉昨夜 IM 私信 + 评论，按 4 类分档 | `messages/inbox/{date}.md` 摘要推送给老板 |
| 08:00 | 商务经理 | 扫社区管家分出来的"合作询盘"，生成询盘卡 + 初步报价建议 | `deals/inquiries/{date}-{shortId}.md` 推送给老板 |
| 08:30 | 内容创作官 | 读 `topics/{date}.md` + 老板灵感 → 生成今日初稿（含配图建议） | `drafts/{date}-{shortId}.md`（待老板审） |
| 11:00 | 分发编辑 | 扫 `drafts/` 中 `status=approved` 的稿，做双平台适配，生成发布包，排 14:00 分发 | 更新 draft + 写 `publish_packs/` + 写 Cron job |
| 14:00 | 分发编辑 | Cron 触发：**按平台分叉** — 公众号真发 / 小红书等推"发布包"到主场"待你点发"区 | 公众号：`published/{date}-{shortId}.md`；其他：`publish_packs/{date}-{shortId}-{platform}.md`（待老板点发） |
| 18:00 | 社区管家 | 处理白天累积的私信/评论，起草回复给老板审 | `messages/replies/{date}.md` |
| 20:30 | 商务经理 | 跟进当日询盘进度，生成"今日商务"：待决策 / 待对方回 / 可签 | `deals/daily/{date}.md` 推送给老板 |
| 21:00 | 财务助理 | 拉今日各平台数据 + 合作结款 + 知识付费流水，生成小结 + 标记最赚钱内容 | `income/daily/{date}.md` 推送给老板 |

**老板的一天因此被压缩成 5 个动作**：
① 早上刷「昨夜摘要 + 今日选题 + 新询盘」三连推送 → ② 选一个选题（或丢自己的灵感） → ③ 11:00 前审稿 → ④ 18:30 审社区回复 → ⑤ 晚上看「商务今日 + 财务晚报」。

### 1.7 主场页：今日排班表

001 落地的主页是「人才市场」，002 改良的主页是「我的团队 + 成长档案」。本方案的主页是**今日排班表**——这是小林每天打开的第一眼。

**主场布局（左中右）**：

```
┌──────────────────────────┬──────────────────────────────┬────────────────────────┐

│  11:00 ⏳ 分发编辑        │  · 【14:00 后】2 条发布包     │  · 待跟进询盘: 2          │
│  14:00 ⏳ 分发编辑        │    待粘贴发送（一键复制）     │  · 谈判中合作: 3          │
```

「我的团队」「人才市场」「驾驶舱」全部降级为侧栏次级入口——**老板的认知主场是"今天怎么过"，不是"我有谁"**。

### 1.8 数据与协作（workspace 布局）

数据走**文件系统**，不绑任何 SaaS。文件住在 EC Gateway 租户沙箱目录（`data/tenants/{tenant_id}/workspace/`）。

```
workspace/
  inspiration/        灵感笔记              老板写，内容创作官读
  topics/             每日选题清单          选题策划官写，内容创作官 + 老板读
  drafts/             内容草稿              内容创作官写,分发编辑读
  published/          已发作品（公众号自动+小红书等老板确认后回写）   分发编辑 + 老板点发动作写
  publish_packs/      待老板点发的发布包（小红书/视频号/抖音/知乎）   分发编辑写，主场"待你点发"区读
  analytics/          平台数据回流          分发编辑 Cron 拉，财务助理读
  assets/             素材（图/视频）       老板 + 内容创作官共用
  messages/
    inbox/            私信原始归档 + 分类    社区管家写
    replies/          回复草稿(待老板审)    社区管家写，老板审
    leads/            高价值线索 / 合作线索 社区管家写，商务经理读
  deals/
    inquiries/        新合作询盘卡          商务经理写
    negotiating/      谈判中合作            商务经理写
    signed/           已签合作              商务经理写
    completed/        已执行已结款合作      商务经理写，财务助理读
    daily/            今日商务日报          商务经理写
  income/
    raw/              各平台/合作流水原始   财务助理 Cron + 老板手贴
    daily/            每日小结              财务助理写
    monthly/          月报                  财务助理写
  schedule/           今日排班执行日志      Cron 执行器写，主场页读
  _index/             聚合索引
```

**格式**：MD + frontmatter，append-only，命名 `{collection}/{YYYY-MM-DD}-{shortId}.md`。

**示例** `drafts/2026-04-21-a3f2.md`：

```markdown
scheduled_publish_at: 2026-04-21T14:00:00+08:00
---

...
## 小红书版（已适配）
...
## 公众号版（已适配）
...
```

**协作范式（状态共享，无事件总线）**：

| 链路 | 文件流转 |
|---|---|
| 选题链 | 选题策划官 06:30 扫外部 + 历史 `published/` → 写 `topics/{date}.md` → 老板/内容创作官 08:30 读 |
| 内容主链 | 老板写 `inspiration/` + `topics/` 被选中 → 内容创作官读 → 写 `drafts/(draft)` → 老板审改为 `approved` → 分发编辑读 → 排期 → Cron 触发：**公众号真发写 `published/`；小红书等平台写 `publish_packs/` 等老板点发后回写 `published/`** |
| 私信链 | 社区管家拉 IM → 写 `messages/inbox/` 分类 + `messages/replies/(待审)` → 老板审 → 系统按审过的草稿真实发出回复；其中合作类线索同时写 `messages/leads/` 交给商务经理 |

## 二、迁移原稿 · 001 · 刘昱 · EnClaws OPC 产品方向：「数字员工」业务包装层设计

> **⚠ 迁移原稿提示**：以下是 001 原帖完整内容。其中部分设计（如"8 预设岗位一字排开"、"纯前端包装层"、"挑岗位 → 入职 → 配工位"主线）在后续讨论中已被收敛或推翻，请以上方第一节「我的意见」为最新立场。保留原稿是为了保留上下文，不代表当前方案。

### 一、问题与判断

基于对 EnClaws 源码（`D:\web3\EnClaws`）的深度分析，当前产品现状：

- **底座能力充足**：多租户隔离、Agent/Channel/Model 三层配置、60+ 内置工具、完整 RBAC、用量追踪，架构上完全能支撑 OPC 场景
- **呈现层是瓶颈**：现有管理端（tenant-agents / tenant-channels / tenant-models）面向的是懂技术的配置者，非技术用户面对 System Prompt、Tool Deny List、API Protocol 等概念会直接懵掉
- **结论**：不需要重做产品，需要在现有能力上加一层**业务语言的翻译层**

### 二、核心策略：概念映射

将 EnClaws 现有技术实体**一对一映射**为 OPC 业务概念，让老板用经营语言操作技术系统：

| EnClaws 技术概念 | OPC 业务概念 | 用户看到的 |
|---|---|---|
| Agent + System Prompt + Tools | **数字员工** | "我的法务专员" |
| Agent 模板/预设 | **人才市场** | "去招一个客服" |
| Channel (Telegram/飞书/微信...) | **工位/工作台** | "安排到微信群上班" |
| Model Config (GPT-4/Claude...) | **能力等级** | "高级版/标准版" |
| Tools deny list | **岗位权限** | "能看文件但不能发消息" |
| Skills | **专业技能证书** | "持有法律文书处理资格" |
| Tenant | **我的公司** | "XX工作室" |
| Cron Job | **排班/日程** | "每天早9点发日报" |
| Usage/Tokens | **薪资/工时** | "本月人力成本 ¥320" |

**关键原则**：不改后端，不改数据模型，纯前端映射。每个"数字员工"本质就是一条 `tenant_agent` 记录。

#### 要点

- **框架选型**：可与 EnClaws 一致用 Lit（复用组件），也可用 Vue/React（纯前端，自由度高）
- **通信**：直连 EnClaws Gateway 的 WebSocket RPC，不需要额外后端
- **部署**：静态资源 + CDN，通过 URL/子域名区分租户
- **模板数据**：前端 JSON 打包，后期可迁移到后端支持用户自建模板

#### Adapter 层核心逻辑

```typescript
// 入职 = 创建 Agent
async function hire(talent: DigitalTalent, name: string) {
  return rpc('tenant.agents.create', {
    agentId: `${talent.id}-${Date.now()}`,
    name,
    config: { systemPrompt: talent._agentConfig.systemPrompt },
    modelConfig: [{
      providerId: getDefaultProvider(),
      modelId: talent.tier === 'senior' ? getTierModel('senior') : getTierModel('standard'),
      isDefault: true
    }],
    tools: { deny: talent._agentConfig.toolsDeny },
    skills: talent._agentConfig.skills,
  });
}

// 安排工位 = 绑定 Channel
async function assignWorkplace(employeeId: string, channelId: string) {
  return rpc('tenant.channels.update', { channelId, config: { agentId: employeeId } });
}

// 薪资报表 = Usage 聚合
async function getSalaryReport(period: string) {
  const usage = await rpc('tenant.usage.metrics', { period });
  return usage.map(u => ({
    employeeName: getEmployeeName(u.agentId),
    workHours: tokensToHours(u.inputTokens + u.outputTokens),
    salary: tokensToCost(u),
  }));
}
```

### 七、几个关键判断

**关于抓手**：人才市场本身就是获客抓手。OPC 老板的第一反应是"我需要人手"而不是"我需要配置一个 AI Agent"。获客话术从"AI 平台"变成"按需招聘数字员工，月薪几十块"——这比任何技术功能描述都好传播。

**关于"先不细究能力"**：完全正确。第一步只需把 system prompt 写好、模型选对，就能达到 70 分的效果。用户先用起来，我们再根据实际使用反馈打磨每个岗位的 prompt 和工具组合。

**关于"仅前端包装"**：架构上完全可行。EnClaws 的 `tenant.*` RPC 接口已经非常完整，前端直连 Gateway 即可。独立仓库也方便针对不同客群出不同皮肤（OPC 版、企业版、行业版）。

**待讨论**：
1. 首批 8 个岗位的优先级？是否需要调整？
2. 人才模板后期是否开放用户自建和交易（呼应之前帖子提到的知识包市场模式）？
3. 前端框架选型偏好——跟 EnClaws 一致用 Lit，还是用团队更熟悉的？

## 三、迁移原稿 · 002 · Daisy · 数字员工的「行为」映射到了，「成长」和「管控」还没映射

> **⚠ 迁移原稿提示**：以下是 002 原帖完整内容。其中补充的"记忆回写 / 预算上限 / Agent 审计"三件事、"成长档案"界面、"累计沉淀"指标在第一节「我的意见」中明确推迟到 Phase 2 不做（Phase 1 前 PMF 不是瓶颈）；Paperclip 对比的价值判断仍然成立。请以上方第一节为最新立场。

001 的翻译框架方向正确，人才市场 + 一键入职的转化逻辑清晰。这里补充一个更底层的视角：**当前映射描述的是员工「在哪上班、干什么活」，但 EC 真正的差异化在「员工会成长、组织有管控」，这两件事在当前设计里完全没有呈现。**

---

### EC vs Paperclip 横向对比

| 维度 | EC | Paperclip |
|---|---|---|
| **组织结构** | 运行时动态生成 subagent | 配置文件静态定义 org chart |
| **知识积累** | 分层记忆（企业→部门→用户） | 无，任务结束 Agent 失忆 |
| **渠道覆盖** | 41+ IM 平台 | 无任何渠道集成 |
| **预算管控** | ❌ 无上限机制 | ✅ 每 Agent 有 cost hard stop |
| **行为审计** | ⚠️ 仅记录人类操作 | ✅ 完整 Agent 行为回溯 |
| **多租户** | ✅ 原生支持 | ❌ 不支持 |
| **自托管** | ✅ | ✅ |

**Paperclip**（paperclipai/paperclip）是一个开源的 AI 公司编排工具，定位是「零人工干预运转一家公司」。核心设计是在配置文件里写好 org chart——CEO Agent 负责拆解目标，法务、财务、运营等各职能 Agent 认领执行，每个 Agent 有预算上限和完整的行为审计链路。它解决的是「让 AI 员工按流程干活」，但没有 IM 渠道集成，也没有跨任务的知识积累机制。

---

### EC 目前比 Paperclip 缺的三件事

**① 预算硬上限。** OPC 老板不敢让 Agent 自主运行，根本原因是「不知道这个月要花多少钱」。驾驶舱展示成本是被动的，让用户能设上限才是主动的。

**② Agent 行为审计。** EC 现有的 audit\_log 记录的是「人类登录和操作」，不是「Agent 自主行为」。当一个 subagent 发出一封邮件、修改一份文档，老板能查到是谁授权的吗？企业客户的合规要求必然问这个问题。

**③ 记忆回写闭环。** 当前 Experience Extraction 是「只写不读」——经验被提取写入记忆层，但下次任务时没有注入回 Agent context。分层记忆变成了只存不取的数据库，「越用越聪明」就只是话术。

---

### 业务包装层要把「成长」做进界面

001 的映射里，Usage/Tokens → 薪资/工时，是合理的成本视角。但只展示「这个月花了多少钱」，用户感知到的是「AI 很贵」。如果同时展示「沉淀了多少条业务经验」，用户感知到的是「这个员工越来越值钱」。

**两个具体设计点：**

「我的团队」员工卡片加「成长档案」入口：*已积累经验 86 条 · 覆盖：合同审查 / 客户服务 / 产品问答*。让用户看到「TA 已经学了这么多，换掉可惜」。

「经营驾驶舱」并列三列：本月投入（¥320）/ 本月产出（1,240 次任务）/ 累计沉淀（86 条）。「累计沉淀」单调递增，是用户感知「越用越值钱」最直观的数字。

---

### 一句话定位

> Paperclip = AI 员工任务调度器；EC = AI 员工组织知识基础设施。
>
> 前者让员工知道「要做什么」，后者让员工知道「这家公司是怎么做事的」。后者的价值随时间增长，前者不会。

业务包装层的下一步，不只是翻译技术概念，而是把「记忆成长」这件 EC 独有的事，变成用户每天打开就能看见、感受到的东西。

## 四、迁移原稿 · 003 · Ken · 控制台应当是生态，而非功能——EC 的平台化演化方向

> **⚠ 迁移原稿提示**：以下是 003 原帖完整内容。其中"EC 从产品演化为平台 / Template 商城 / 行业控制台生态"的长线愿景在第一节「我的意见」中明确推迟到 Phase 3 不做（Phase 1 只保留最小 `CompanyTemplate` 数据模型）；"preset 必须住后端"的判断被采纳。请以上方第一节为最新立场。

### 001 中被低估的一个架构洞察

001 的核心论点是「概念映射」，但它里面有一个更底层的架构观点被顺带提及，没有被展开：

> 「独立仓库也方便针对不同客群出不同皮肤（OPC 版、企业版、行业版）」

这句话的意思是：**EC 后端对多个前端控制台是中立的**。标准技术控制台和 OPC 控制台是并行关系，部署 EC 时可以选择部署其中任意一个，或者同时部署两个。控制台是可插拔的，后端不感知。

这个观点值得被显式提炼出来，因为它指向的不只是「做一个 OPC 版本」，而是 EC 更长远的演化方向。

---

### 行业控制台带出了一个新的后端需求

001 的 OPC 控制台设计里，有一个关键部分：**8 个预设数字员工模板**。这些模板数据（systemPrompt、modelConfig、toolsDeny、skills）目前的处理方式是打包进前端 JSON 文件。

001 自己也意识到这不够，留了一句：「后期可迁移到后端支持用户自建模板」。

但这句话掩盖了一个更根本的问题——**一旦 preset 数据需要持久化，「纯前端包装层」这个前提就已经不成立了**。

静态 JSON 无法解决的需求，随着产品演化必然出现：

- **租户差异**：同一个「法务专员」模板，A 公司和 B 公司想用不同的 systemPrompt
- **用户自建**：OPC 老板把自己调好的 Agent 保存为模板，下次复用
- **版本管理**：模板升级时，已入职的员工是否自动跟进？
- **跨部署的 ID 引用**：模板里引用的 modelId、skillId 在不同 EC 部署里可能不同

结论：**行业控制台本质上带出了自己的数据层需求**，这部分数据应该住在 EC 后端，而不是分散在各个控制台的 JSON 文件里。

---

### EC 的演化方向：从产品到平台

把上面两点合在一起，可以看到一条清晰的演化路径：

**现在**：EC 提供 AI Agent 基础设施 + 一个官方控制台

**演化后**：EC 提供 AI Agent 基础设施 + 一个开放的 Frontend API，任何人可以基于它构建自己的行业应用

这个模式有成熟的先例：Salesforce 是 CRM 内核 + AppExchange 行业前端生态；Shopify 是电商引擎 + 主题/应用生态。EC 有机会走同一条路。

要支撑这个演化，后端需要做三件事：

| 需要做的 | 说明 |
|---|---|
| **Template/Preset 数据模型** | 让 preset 数据住在后端，各控制台共享同一套模板库，只是呈现方式不同 |
| **稳定的公开 API 契约** | 现有 `tenant.*` RPC 接口需要被当作平台 API 来维护，不能随意破坏性变更 |
| **前端身份与权限** | 不同前端应用有自己的 client identity，访问同一租户数据时有明确授权边界 |

对外的价值主张也随之升级：

> 「用 EC 部署 AI Agent 基础设施，然后从社区选择适合你行业的控制台——OPC 老板版、律所版、连锁餐饮版……」

---

### 模板即资产

这个方向也回应了 001 留下的一个悬而未决的问题：「人才模板后期是否开放用户自建和交易？」

如果后端有了 template 数据模型，答案就自然浮现：**模板本身可以成为生态里流通的资产**。这与之前讨论过的知识包市场方向完全吻合——不同的是，知识包市场卖的是 prompt 和配置，而这里卖的是可以一键部署到自己 EC 实例的完整行业控制台。
