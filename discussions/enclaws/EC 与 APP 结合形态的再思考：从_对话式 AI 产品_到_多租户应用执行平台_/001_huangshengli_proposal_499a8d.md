# EC 与 APP 结合形态的再思考：从"对话式 AI 产品"到"多租户应用执行平台"

## 一、起因

最近在推进 `enclaws/ec对app运行时的支持需求` 的两条轴——APP 子进程反向调 Gateway LLM 接口（轴 A），以及 CLI 正向调 APP pipeline（轴 B）。每推一步我越发觉得：我们在具体技术细节上吵得很细（UUID vs slug、一次性 token、方案 A vs B），但在**"EC 和 APP 结合到底是什么形态"这个上位问题上，团队其实是默认叙事在驱动，没有真正讨论过**。

我想把这层退回去，重新审视。本文不给最终答案，但会带着已经想清楚的几层推论。希望团队在继续轴 A/B 之前，先对齐这个形态问题——因为它决定后面所有细节的形状。

---

## 二、三个绕不开的观察

### 观察 1：EC APP 本质是三件事的捆绑

1. **LLM 做意图解析**（自然语言 → 结构化调用）
2. **Runner 执行确定性 pipeline**
3. **IM 作为入口和输出通道**

看 team-pivot 就很明显——它的 pipeline（git pull → 写 MD → commit/push → 发飞书）是**极致确定**的。当 pipeline 越确定，第 1 件事（LLM 解析）的价值越低：用户其实知道自己要干什么，只是懒得学 UI。此时 LLM 不是在理解用户，是在把明确意图翻译两遍——用户反而要"学咒语"。

### 观察 2：AppExchange + 对话式入口 = 已被验证的天花板

Slack App Directory（带 slash command 解析）、钉钉微应用（带智能推荐）、飞书 botbuilder（带自然语言触发）——这三家十年内都做过，都没跑成"企业主流形态"。不是因为技术不行，是因为企业付费买的是**深度集成**（SAP/用友/金蝶），不是轻量工具集合；"一个超级 bot 懂所有 app"都退回到 slash command 或 app-specific bot。

EC 如果沿着"LLM 壳 + AppExchange"走，是在重走这条被验证过天花板的路。

### 观察 3："靠 LLM 理解力去执行"是逆向价值主张

企业采购决策者（CIO/CTO/部门负责人）关注**可控性、确定性、集成度、单位经济**。LLM 对话式入口在这四项上全弱：
- 对话式操作的审计比 click stream 复杂
- LLM 输出本质随机
- 到核心系统的集成是另一个项目
- token 消耗随措辞变化、成本难预算

对"已知自己要什么"的企业用户，中间这层 LLM 是纯负债。对"不知道自己要什么"的用户才有价值——但那不是企业采购的对象。

### 三个观察合起来，指向一个"位置错配"

把上面三点叠在一起看，它们其实指向同一件事：

> **我们把 LLM 放到了不该放的位置——让它在每个请求面前充当"必经的意图解析层"。**

- 真正让企业用户付钱的 APP 价值（确定性执行、凭证管理、审计合规、深度集成），**不需要 LLM，甚至会被 LLM 削弱**
- 真正需要 LLM 的场景（自由文本解析、结构化写作、跨工具编排），**本该发生在客户端或 APP 内部，而不是 EC 平台中间层**
- 当前架构恰好把 LLM 塞进了最不需要它的位置（作为所有请求的必经入口），又在真正需要它的位置（客户端/APP 内部）把它当成附属

**这是一次"位置错配"，不是某个细节上的 bug**。我们在轴 A/B 里反复纠结的不少细节（UUID vs slug、一次性 token、方案 A vs B），有一部分其实可以追溯到这处错配——**是症状，不是根因**。

如果这个判断成立，接下来必须顺着三个问题推下去，才能给 EC 找到正确形态：

1. **EC 作为"共享后端"，真正共享的是什么？** 哪些事只有平台能做，哪些 APP 自己就能搞定？
2. **APP 在 EC 里应该是什么角色？** 它靠什么机制注入 EC、被 EC 识别？
3. **一个请求从进入 EC 到执行，究竟走过哪些层？** 哪些是确定性路由，哪些才真需要 LLM？

下面三章沿着这三个问题逐层展开。

## 三、重新拆"共享后端"与"APP 角色"

### "共享"不是一个词，是一个维度

| 共享维度 | 在共享什么 | 能被替代吗 | EC 不可替代？ |
|---|---|---|---|
| 跨租户 | 一套代码服务 N 家公司 | 传统 SaaS | ✅ |
| 跨 APP | 多 APP 复用 IM 通道/凭证/审计 | 企业服务总线 | ✅ |
| 跨客户端 | Feishu/Claude Code/Web 看到同一数据 | APP 自己的后端 | ❌ |
| 跨用户 | 租户内多人协作 | APP 自己的 DB | ❌ |
| 跨租户 APP 市场 | 一家装别家的 APP | App Store | ❓ 未来 |

**真正只有 EC 能提供、APP 自己搞不定的，只有前两条**。其他是便利，不是必要。

### APP 角色应该窄到什么程度

APP 存在的意义是让 EC 能：路由事件、管理生命周期、做跨 APP 治理、提供统一审计。满足这些的最小定义：

```yaml
# app.yaml（比 v0.5 精简）
name: team-pivot
version: 1.0.0

handlers:                     # EC 可以把事件路由到这些
  - event: rpc.discuss_list
    entry: list_handler
  - event: feishu.message
    entry: im_handler
  - event: cron.daily
    entry: digest_handler

bindings:                     # 我依赖 EC 的哪些能力
  - type: llm_gateway         # 可选
  - type: credential_vault
    needs: [git_push, feishu_bot]
  - type: storage
    scope: tenant
  - type: identity
```

**当前 APP Protocol v0.5 的 `WebView + triggers + config_webview + pivot.yaml collection` 都应降级为可选能力层**。team-pivot 其实不需要 WebView、不需要 collection、不需要 config_webview——把这些做成强制项，就是把可选能力硬塞进每个 APP。

## 四、请求是怎么走到 handler 的？——L1-L5 路由模型

这是"EC 执行平台"的运行核心。把"事件进入 EC"到"handler 执行完成"拆 5 层：

```
┌──────────────────────────────────────────────────────────────┐
│                     事件源 (Event Sources)                    │
│  HTTP · WebSocket · IM Webhook · Cron · Inter-APP Event       │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ L1  传输识别                                                  │
│     URL/协议/签名 → 标准化 event envelope                     │
│     【确定性】                                                │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ L2  租户 & 身份解析                                           │
│     JWT / openId / app_id → tenant_id + identity              │
│     【确定性】                                                │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ L3  APP 绑定解析                                              │
│     频道/URL/cron 配置 → 该租户下的 app_instance_id           │
│     【确定性，查路由表】                                      │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ L4  Handler 匹配                                              │
│     event_type → APP 内的 handler 入口                        │
│     【确定性，查 handlers 声明】                              │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ L5  参数推导                                   ← 分岔点       │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 场景 A: RPC/cron/webhook  → 结构化透传               │   │
│  │ 场景 B: IM 自由文本       → APP handler 内部解析     │   │
│  │ 场景 C: Client 已预解析   → 透传                     │   │
│  │ 【禁止】: EC 统一用 LLM 猜消息该给哪个 APP           │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│          Handler 执行（子进程隔离，注入 context）             │
│   ctx = { event, identity, tenant, bindings, trace_id }       │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│     响应与副作用：return / 出站消息 / 发事件 / 审计落盘       │
└──────────────────────────────────────────────────────────────┘
```

### 各层责任

| 层 | 内容 | 确定性 | 谁决定 |
|---|---|---|---|
| L1 | 是哪种传输 | ✅ | URL / 协议 / 签名 |
| L2 | 哪个租户、谁 | ✅ | JWT / openId 映射 |
| L3 | 绑定哪个 APP | ✅ | 路由表（安装时写入） |
| L4 | 调用哪个 handler | ✅ | 事件类型 + handlers 声明 |
| L5 | params 长什么样 | 🟡 看场景 | 结构化透传 / APP 内部解析 / client 预解析 |

**关键观察**：**L1-L4 完全不需要 LLM**。走到 L5，才出现"要不要 LLM"的分岔。而且在 L5，LLM 要么在 client 侧（Claude Code）、要么在 APP 内部（通过 llm_gateway binding 自调用），**EC 平台层不承担 LLM 解析责任**。

### L5 的三个选项对照

| 选项 | 场景 | params 怎么来 | LLM 在哪 | 推荐度 |
|---|---|---|---|---|
| **透传** | CLI / Web UI / Webhook / Cron | 调用方/配置已给 | 客户端侧（可选） | ✅ 主路径 |
| **APP 内解析** | IM 自由文本 | handler 自己处理 raw_text | APP 内部（可选） | ✅ IM 场景 |
| **EC 统一 LLM 路由** | "一个超级 agent 懂所有 APP" | EC 解析后填 | EC 平台层 | ❌ 禁止 |

**为什么禁第三种**：Slack/钉钉/飞书都试过，都没跑通。会让 EC 必须理解所有 APP 的语义，幻觉责任由平台兜底，multi-APP 冲突随 APP 数量超线性增长，审计困难。

## 五、几个由此推出的具体结论

### 1. EC 的产品定位需要修正

| 旧叙事 | 新叙事 |
|---|---|
| 企业级 ChatGPT + APP 市场 | 企业级 LLM-aware 应用的多租户执行平台 |
| EC 有 AI 大脑，APP 是它的工具 | APP 是主角，EC 是它们的共享底座 |
| 用户通过对话与 AI 交互 | 用户通过客户端/IM/Web 与各 APP 交互 |
| IM 是主入口 | IM 是入口之一，常常只作通知 |

**EC 在这个定位下更像 Cloudflare Workers 或 AWS Lambda 的企业 IM 变种**——一个执行平台，不是一个对话产品。

### 2. RPC 端点设计

推荐 `POST /v1/app/rpc`，body 结构：

```json
{
  "app": "team-pivot",
  "method": "discuss_list",
  "params": { "project": "enclaws" }
}
```

比 `/v1/app/<name>/rpc` 更灵活——鉴权/路由/审计逻辑统一在一个 handler 里，APP 名字作为数据而非 URL 段，便于批量扩展。

### 3. Bot 与 APP 的绑定原则

**事件源到 APP 的绑定必须是确定性的，不依赖 LLM 判断**。

- 1:1 bot↔APP 是最简实现
- 多 APP 共享 bot 也可以（靠 slash command 或频道绑定做确定性路由）
- **唯一禁止**："靠 LLM 猜消息该给哪个 APP"

### 4. Skill 体系的重新定位

Skill 不是 APP，是 APP 或客户端可引用的**可复用资产**：

| 当前 skill 类型 | 新模型下 |
|---|---|
| 服务型（GitHub / Slack / Notion） | **升级为 APP**（有 handler 和 bindings） |
| 流程型（/review / /init / /security-review） | **部分转 APP，部分留 prompt 模块** |
| 纯 prompt（编码风格 / 习惯） | **留 prompt 模块**，供 APP 或客户端加载 |

Skill 不再是"平台级 agent 的能力货架"——因为我们不再有内置 agent。

### 5. Web UI 的 agent 聊天页

在新模型下需要重新定位。三个选项：
- **管理后台路线**：不做对话，只做 APP 安装/卸载、审计查看、租户管理
- **客户端路线**：像 Claude Code 一样，前端自带 LLM 能力，调 APP RPC
- **chat APP 路线**：Web UI 承载一个"chat APP"的 WebView，这个 APP 自己管对话

最干净的是第一种。

## 六、这对已在做的工作意味着什么

**好消息**：团队已经在做的工作大部分**已经在这条线上**，只是没有显式归类到"handler 路由层"这个抽象：

| 已做 | 新模型中的角色 | 保留度 |
|---|---|---|
| 一次性 `GATEWAY_TOKEN` + sessionKey 反查 | L3-L4 的 context 注入机制 | ✅ 保留 |
| `/v1/agent/chat/completions` 免 token 直连 | LLM binding（建议改名 llm_gateway） | ✅ 保留 |
| Option 2 纯版 Runner（子进程隔离） | 执行基质（安全红线） | ✅ 保留 |
| CLI 正向通道方案 A（invokeSkillPipeline） | L1-L4 的 RPC 入口 | ✅ 契合 |
| `parseSessionKey` / 多租户身份映射 | L2 租户解析 | ✅ 保留 |

**需要重审的工作**：

| 当前 | 新模型下 |
|---|---|
| Gateway 里"飞书消息 → EC agent → skill 判断" | 改为"IM 消息 → 路由表 → APP handler" |
| 60+ skill 作为 agent 货架 | 重定位为 APP/客户端可引用资产，分层处理 |
| agent 三档分级方案（`enclaws/agent-模型三档分级方案`） | 相关性下降——除非作为 dispatcher APP 内部实现 |
| APP Protocol v0.5 的 WebView/triggers/pivot.yaml 作为强制项 | 降级为可选能力层 |
| Web UI 的 agent 聊天页 | 重新定位（推荐纯管理后台） |
| system prompt optimization / token optimization | 仅在 dispatcher APP 内部相关 |

**这不是"推翻"，是"归位"**——让分散的技术选择都有一个清晰的抽象坐标。

## 七、我还没想清楚的地方

下面几点我自己也在摇摆，希望团队一起打磨：

### 1. 要不要允许"dispatcher APP"作为逃生阀？

如果某个租户真的想要"一个入口对接所有 APP"，可以写一个 `dispatcher-agent` APP：`needs: [llm_gateway, app_registry]`。它作为 APP 存在，**不是平台层**。

但疑问是：即使作为 APP 写出来，"一个 LLM 懂所有 APP"的幻觉问题并没有消失，只是换了归属。也许答案是 dispatcher APP 也应该自我约束——只做明确的 slash command 路由 + 可选的 per-APP LLM 帮助，不做跨 APP 语义理解。

### 2. Skill 迁移路径

现有 60+ skill 要切成三类，一步到位重构代价太大。哪些先转 APP（服务型里哪些优先）？哪些先搁置？这个迁移路线需要具体讨论。

### 3. OpenClaw 与 EC 的代码复用边界

OpenClaw 的 skill + agent 模型在个人场景是成立的（LLM 本来就是主角，个人用户没有多租户/治理/审计需求）。EC 很多代码是从 OpenClaw 继承来的，但在多租户 + 企业场景下走不通。

**OpenClaw 和 EC 的代码复用要不要重新划边界**？EC 可能需要一条独立演进路径，不再等同于"OpenClaw + 多租户"。

### 4. Web UI 和 gateway 聊天通道的处置

当前 gateway WebSocket 和 Web UI 聊天页是团队投入较重的部分。在新模型下，它们是：
- 删掉？（激进）
- 改为管理后台？（主流推荐）
- 保留为"本租户可选装的 chat APP"？（兼容老用户但新租户不默认开）

### 5. 这次修正的落地节奏

如果这个形态成立，意味着：
- `ec的app支持规范构想` 需要补一版 v0.6，明确"handler + bindings 最小契约 + 可选能力层"
- `ec对app运行时的支持需求` 轴 A/B 的技术选型可以维持，但定位要重贴标签
- `agent-模型三档分级方案` / `system-prompt-optimization-inline-skill` / `token-optimization-plan` 这类线程可能需要暂缓或重新 scope

**怎么安排节奏，团队精力不被翻来覆去消耗**，这个本身是个组织问题，值得一起讨论。

---

## 八、结语

这篇不是定论，是我在轴 A/B 实作过程中自己推导出来的一层退后视角。我把它写出来，不是说我有答案，而是**这几个根问题——"共享后端到底共享什么"、"APP 角色怎么定义"、"请求如何走到 handler"、"LLM 解析在哪层"——我觉得比我们在轴 A/B 上吵的任何细节都更值钱，前者决定后者的形状**。

希望团队在继续推技术细节之前，先花一点时间把这层形态问题对齐。如果大家觉得方向有偏，欢迎直接反驳；如果方向没偏，那我们的 APP Protocol、skill 体系、Web UI、gateway 对话通道，都值得在这个新定位下重新审视一遍。
